## 0. 如何使用本文档

本文档是“可点菜菜单”，不是 AI 自己排期的路线图。

- 用户以后手动点名后，再为被选功能建立工作包 / 实现；AI 不从本文档自行推进功能。
- 每个条目都以 `FEA-xx` 编号。以后可以直接说“推进 FEA-03”或“先做 FEA-20 的第一片”。
- “第一片”只代表最小可验证切片，不代表最终形态。
- “可能改哪些文件”是预估范围，真正开工前仍要读目标模块文档和源码。
- 若功能需要新 id、字段、事件、状态、音频 id、pool id、InputMap action 或 save kind，必须先查 `docs/词表与契约.md`，必要时补契约并跑同步。
- 若功能改变玩法规则、系统边界或长期承诺，必须写入 GDD / ADR / 代码模块文档；如果只是内容数据扩展，通常只同步数据手册、locale 和当前工作包。
- 如果一个功能明显依赖另一个功能，建议先做前置功能，不要硬绕。

本文档里所有候选默认都遵守现有原则：

- 数据驱动优先，内容走 `client/data/` 和 `client/locale/`。
- 运行时逻辑走已有 autoload / 模块边界：`DataLoader`、`RNG`、`GameClock`、`GameState`、`UIManager`、`PoolManager`、`SaveManager`、`AudioManager`、`PlatformServices`。
- 高频实体走对象池。
- 玩家可见文本走双语 locale。
- 随机和时间分别走 `RNG` 与 `GameClock`。
- 不为了某个 id 写一次性 `if id == ...` 逻辑。
- 未来 mod / 多模式 / 多人边界继续保持友好，不把资源写死到标准模式里。

## 1. 当前项目基线

当前 Demo 已经有一个比较完整的最小体验：

- 标题菜单、局内玩法、暂停、失败结算、局外成长、设置、本地化、保存 / 续局都已有首片。
- 玩家、基础武器、子弹、敌人、刷怪、经验球、升级三选一、HUD、死亡结算可正常跑通。
- 局外成长已经能购买永久升级，并影响下一局。
- F8 回放 / 测试 / 性能基线已经成为 F9 之后扩内容的护栏：四条 checked-in golden replay、`runtime-smoke`、`f9-demo-smoke`、`perf-probe` 等都可用。
- F9 第一轮已补了一个小内容包、基础表现可读性、命中火花和伤害数字，并由用户人工确认 Demo 没明显问题。
- 目前没有临时音频资源，因此音频接入可以先规划，等资源到位再落地。

这意味着后续加内容已经不用先大拆底层，可以比较自由地选方向。但越靠近“系统首片”的功能，越要有清晰工作包和回归命令，避免小改动慢慢把边界弄乱。

## 2. 选择维度与复杂度标尺

选择功能时可以看六个维度：

| 维度 | 说明 |
|------|------|
| 玩家立刻感知 | 玩家开一局后是否马上感觉游戏变丰富 / 更好玩 |
| Demo 展示价值 | 是否适合给别人看，是否提升完成度感 |
| 架构收益 | 是否把已有数据和 autoload 真正用起来，降低后续扩展成本 |
| 内容生产效率 | 是否让以后加敌人、遗物、武器、关卡更快 |
| 风险 | 是否容易影响存档、回放、性能、输入、UI 或平衡 |
| 资源依赖 | 是否依赖音频、美术、Steam 后端、商店页面或外部资源 |

复杂度标尺：

| 复杂度 | 含义 |
|--------|------|
| S | 1 个小切片，主要是数据 / UI / 单模块逻辑；验证范围清晰 |
| M | 需要 2~4 个模块协作，有新 UI 或新 runtime 行为；适合一个短工作包 |
| L | 新系统首片，影响存档 / 回放 / 数据 schema / 多 UI；需要分阶段做 |
| XL | 长线方向或发行 / 平台级工程，不建议一口气做完整，只能拆成多个工作包 |

## 3. 总览优先级矩阵

如果目标是“让下一版 Demo 更像一个游戏”，优先级大致如下：

| 候选 | 推荐度 | 复杂度 | 为什么值得 |
|------|--------|--------|------------|
| FEA-01 武器选择 / 第二武器获取路径 | 很高 | M | 直接解决构筑感不足，让玩家局内有更明确目标 |
| FEA-03 遗物 runtime 首片 | 很高 | L | 把已有遗物数据真正变成玩法核心，长期收益最大 |
| FEA-18 角色选择与角色差异 | 高 | M | Demo 开局选择感强，复玩价值高 |
| FEA-12 Hazard 机关 runtime | 高 | M/L | 让地图和走位压力变丰富，能复用已有 hazard 数据 |
| FEA-20 武器 / 构筑状态面板 | 高 | S/M | 低风险提升可读性，帮助玩家理解自己变强了 |
| FEA-21 升级选择 UX 扩展 | 高 | M | reroll / skip / banish 很 Roguelike，能强化升级策略 |
| FEA-25 内容模板 / 数据生产流水线 | 高 | S/M | 后续加内容会明显更快、更少出错 |
| FEA-30 AudioManager 资源接入 | 中高 | M | 一旦有资源，完成度提升很明显 |
| FEA-35 Steam 成就 / 统计 / 富状态 | 中 | M/L | 发布感强，但真实平台依赖和测试成本更高 |
| FEA-37 Demo 发行清单 | 中高 | S/M | 如果准备发给别人玩，这是最务实的一组收尾 |

我个人建议的第一批路线：

- 想增强“构筑感”：先做 FEA-01，再做 FEA-03 或 FEA-21。
- 想增强“内容变化”：先做 FEA-18 或 FEA-03。
- 想增强“地图和走位”：先做 FEA-12，再做 FEA-13。
- 想增强“发行完成度”：先做 FEA-30、FEA-37，再看 FEA-35。
- 想让以后开发更快：先做 FEA-25，再做 FEA-28。

## 4. 核心构筑与战斗系统

### FEA-01 武器选择 / 第二武器可见获取路径

**玩家价值**

现在玩家默认只有起始武器，升级选择主要是属性强化。增加“可见的第二武器获取路径”后，玩家会明显感觉局内构筑变丰富：这局拿什么武器、两个武器怎么互补、是否偏单体 / 清场 / 安全，都会成为自然目标。

**当前前置条件 / 可复用基础**

- `client/data/weapons.json` 已有武器数据 schema。
- `characters.json` 已有 `starting_loadout` 概念。
- `GrowthSystem` / `LevelUpPanel` 已经支持升级候选和选择流程。
- `WeaponSystem` 已有基础自动射击 runtime。
- `SaveManager` run snapshot 已保存局内关键实体，后续可扩展武器列表。

**第一片怎么做**

第一片不要直接做复杂掉落系统，可以先做“升级选项里出现第二武器”：

- 在成长候选类型中支持 `grant_weapon` 或等价的奖励类型。
- 新增 1 把第二武器，例如环绕弹、散射弹、慢速高伤穿透弹三选一中的一种。
- 玩家升级到指定等级后，候选池有概率出现“获得新武器”。
- `WeaponSystem` 支持维护多个武器定义，每个武器独立冷却并开火。
- HUD 或构筑面板至少能显示当前武器列表。
- run save / restore 能恢复已获得武器。

**可能改哪些文件**

- `client/data/weapons.json`
- `client/data/growth_pools.json`
- `client/data/game_modes.json`
- `client/locale/strings.csv`
- `client/scripts/gameplay/weapon_system.gd`
- `client/scripts/gameplay/gameplay_run_loop.gd`
- `client/scripts/gameplay/level_up_panel.gd`
- `client/scripts/gameplay/gameplay_hud.gd`
- `docs/代码/gameplay_runtime.md`
- `docs/词表与契约.md`
- `docs/测试策略.md`
- 新工作包，例如 `docs/AI协作/工作包/F10-WeaponChoice.md`

**验证建议**

- `sync_contracts --check`
- `validate_data`
- `test_data_loader_schema`
- `lint_gdscript_rules`
- `runtime-smoke`
- `save-smoke`，确认第二武器保存 / 续局不丢
- 四条 checked-in replay runner，确认未意外扰动旧 golden
- 新增一个短 replay 或 smoke，验证获得第二武器后两把武器都能开火

**复杂度**

M。

**风险**

- 多武器冷却、伤害来源、pool 使用和 save restore 容易出现漏字段。
- 如果第二武器直接进升级池，升级候选平衡会被改变，短 replay 可能需要重新确认。
- UI 不显示当前武器时，玩家可能不知道自己拿到了什么。

**什么时候适合选**

当你觉得 Demo “能玩但构筑感还不够”时，优先选它。它是局内玩法最直观的一步。

### FEA-02 新武器族与武器升级 / 进化

**玩家价值**

武器不只是“多一把”，而是能形成武器族：直线弹、散射、环绕、爆炸、穿透、链式、地雷 / 区域。后续再加“进化”后，玩家会有明确追求：把某个武器练到高等级，搭配某个遗物后进化。

**当前前置条件 / 可复用基础**

- FEA-01 如果先完成，会有多武器 runtime。
- `weapons.json` 已经能承载基础武器定义。
- `growth_pools.json` 可以承载武器强化候选。
- `ModifierEngine` / stat modifier 概念可复用一部分。

**第一片怎么做**

建议分两步：

- 第一步只做“武器等级”：同一武器可以被升级，提高伤害 / 射速 / 子弹数量 / 穿透。
- 第二步再做“武器进化”：满足武器等级 + 遗物 / 条件后替换成进化武器 id。

第一片可以只给起始武器做 3 级升级：

- `weapon_level` 进入 run state。
- 升级候选支持 `upgrade_weapon`。
- `WeaponSystem` 根据等级叠加参数。
- UI 显示武器等级。

**可能改哪些文件**

- `client/data/weapons.json`
- `client/data/growth_pools.json`
- `client/scripts/gameplay/weapon_system.gd`
- `client/scripts/gameplay/gameplay_run_loop.gd`
- `client/scripts/gameplay/gameplay_hud.gd`
- `client/locale/strings.csv`
- `docs/代码/gameplay_runtime.md`
- `docs/游戏设计文档.md`

**验证建议**

- 新增 weapon-level smoke。
- 保存 / 续局后武器等级仍生效。
- `perf-probe` 观察多弹幕时预算状态。
- 手动试玩 3~5 分钟，看伤害成长是否明显但不失控。

**复杂度**

M/L，取决于是否同时做进化。

**风险**

- 如果武器升级直接改 runtime 参数，容易和通用属性 modifier 重叠。
- 进化条件若过早做复杂，会牵动遗物 runtime、图鉴、UI、存档。
- 多 projectile pattern 对性能和可读性都有压力。

**什么时候适合选**

当 FEA-01 已完成，玩家能拿第二武器后，再做它最顺。如果现在就做，也建议只做“起始武器升级第一片”。

### FEA-03 遗物 runtime 首片

**玩家价值**

遗物是这个类型最能产生“这局不一样”的系统。当前已有 `relics.json` 数据边界，但还没有 runtime。做首片后，Demo 会从“属性升级”进入真正的 Roguelike 构筑阶段。

**当前前置条件 / 可复用基础**

- `client/data/relics.json` 已有数据 schema。
- `game_modes.json` 已有 relic pool 引用。
- `ModifierEngine` 的设计边界已有，但需要确认 runtime 现状。
- `LevelUpPanel` 可作为遗物获取入口。
- `SaveManager` 可扩 run payload。

**第一片怎么做**

第一片建议只做“被动 stat modifier 遗物”，不要同时做触发型 behavior：

- 新增 `RelicInventory` 或在 `GameplayRunLoop` 中维护已获得遗物列表。
- 从 `relics.json` 读取 `modifiers`。
- 获得遗物时把 modifier 注入玩家 / 武器相关 stats。
- 升级候选池中加入 2~3 个遗物候选。
- HUD 或构筑面板显示已获得遗物。
- run save / restore 保存遗物 id 列表。

候选遗物示例：

- 玻璃弹头：伤害提高，最大生命降低。
- 磁石：拾取范围提高。
- 余温核心：射速提高，移动速度降低。

**可能改哪些文件**

- `client/data/relics.json`
- `client/data/growth_pools.json`
- `client/data/game_modes.json`
- `client/locale/strings.csv`
- `client/scripts/gameplay/gameplay_run_loop.gd`
- `client/scripts/gameplay/player.gd`
- `client/scripts/gameplay/weapon_system.gd`
- 可能新增 `client/scripts/gameplay/relic_inventory.gd`
- `docs/代码/gameplay_runtime.md`
- `docs/代码/data_loader.md`
- `docs/测试策略.md`
- `docs/游戏设计文档.md`

**验证建议**

- `validate_data` 和 schema 回归。
- `runtime-smoke` 验证获得遗物后 stat 生效。
- `save-smoke` 或新增 run smoke 验证遗物续局。
- 新增 `golden_relic_synergy` 的前置可以先作为 smoke，不必立刻 checked-in replay。
- 手动试玩看遗物文案、获得反馈和 HUD 可读性。

**复杂度**

L。

**风险**

- modifier 生命周期和叠加顺序如果没定好，后续会难维护。
- 遗物既可能影响玩家，也可能影响武器、掉落、敌人、经济，边界要收紧。
- 触发型 behavior 诱惑很大，但第一片如果做太多会扩大风险。

**什么时候适合选**

当你想从“Demo 完整”跨到“游戏核心循环更有灵魂”时，选它。它是长期收益最高的方向之一。

### FEA-04 升级选项池扩展

**玩家价值**

升级三选一目前主要是基础属性。扩展后可以加入回血、护盾、武器升级、遗物、主动道具强化、局内货币等，让升级界面从“数值按钮”变成真正决策点。

**当前前置条件 / 可复用基础**

- `growth.csv` 和 `growth_pools.json` 已存在。
- `LevelUpPanel` 已经有候选展示和选择。
- `RNG.ui_choice` 已用于候选抽取。
- 待决策 E 正好和本项相关。

**第一片怎么做**

建议先关闭待决策 E 的最小范围：

- 保留基础属性奖励。
- 加入 `heal_small` 或 `max_hp_restore` 这类即时奖励。
- 如果 FEA-03 已做，则加入少量遗物候选。
- 如果 FEA-01 已做，则加入武器获取 / 武器升级候选。
- 为候选增加 rarity / weight / min_level / max_stack 等字段时，同步数据手册和 schema。

**可能改哪些文件**

- `client/data/growth_pools.json`
- `client/data/growth.csv`
- `client/locale/strings.csv`
- `client/scripts/gameplay/gameplay_run_loop.gd`
- `client/scripts/gameplay/level_up_panel.gd`
- `client/scripts/gameplay/player.gd`
- `docs/修改建议.md`
- `docs/决策记录.md`
- `docs/游戏设计文档.md`

**验证建议**

- 固定 seed smoke 验证候选数量、权重和等级门槛。
- `golden_level_up_choice` 可能需要评估是否仍稳定。
- 人工试玩看升级候选是否有“无脑选某个”的问题。

**复杂度**

M。

**风险**

- 选项池变复杂后，候选过滤、去重、权重、稀有度容易出隐性 bug。
- 如果回血太强，会降低走位压力。
- 如果过早加入太多类型，UI 文案和说明会变乱。

**什么时候适合选**

当你希望不做大系统也能让局内每次升级更有决策感时，选它。

### FEA-05 主动道具 runtime

**玩家价值**

主动道具给玩家一个“关键时刻按键”的表达空间。它能补足当前自动战斗偏被动的问题，让玩家在被包围、残血、清场时有主动选择。

**当前前置条件 / 可复用基础**

- `client/data/active_items.json` 已有数据 schema。
- InputMap 已有主动道具 action 预留。
- `Settings` 已有基础输入绑定。
- `GameClock` 可提供冷却时间。
- `LevelUpPanel` / 角色 starting loadout 可作为获取来源。

**第一片怎么做**

先做单槽主动道具：

- 默认角色可以带一个基础主动道具，或升级候选中获得。
- UI 显示图标 / 名称 / 冷却进度 / 可用状态。
- 按主动道具键触发效果，例如小范围击退、瞬间回血、短时加速、清除附近子弹。
- 冷却走 `GameClock`，暂停时冻结。
- save / restore 保存主动道具 id、冷却剩余和充能。

**可能改哪些文件**

- `client/data/active_items.json`
- `client/data/characters.json`
- `client/data/game_modes.json`
- `client/locale/strings.csv`
- `client/scripts/gameplay/gameplay_run_loop.gd`
- `client/scripts/gameplay/gameplay_hud.gd`
- `client/scripts/gameplay/player.gd`
- `client/scripts/autoload/settings.gd`
- `docs/代码/gameplay_runtime.md`
- `docs/代码/settings.md`

**验证建议**

- `settings-smoke` 确认按键绑定没有破。
- `runtime-smoke` 验证主动道具触发和冷却。
- `save-smoke` 验证冷却保存 / 续局。
- 手动试玩键盘和手柄兜底。

**复杂度**

M/L。

**风险**

- 使用效果会诱发很多新 effect primitive，容易范围失控。
- UI 和输入要清楚，否则玩家不知道能按。
- 主动效果如果能清屏，会影响敌人 / 波次平衡。

**什么时候适合选**

当你想让玩家操作层更有存在感，而不只是移动 + 自动射击时，选它。

### FEA-06 消耗品 / 临时增益

**玩家价值**

消耗品可以让地图拾取和短期决策更丰富，例如临时护盾、临时射速、一次性回血、吸附全屏经验。它比主动道具轻一些，也适合做局内奖励点。

**当前前置条件 / 可复用基础**

- `client/data/consumables.json` 已有数据 schema。
- `PickupOrb` / 对象池可作为拾取物实现参考。
- `GameClock` 可承载临时效果持续时间。

**验证建议**

- `perf-probe` 确认拾取物池不造成额外抖动。

**复杂度**

M。

**风险**

- 掉落概率会影响难度和节奏。
- 如果马上做背包 / 堆叠 / 使用输入，会膨胀成主动道具系统。

**什么时候适合选**

当你想给局内地图和敌人掉落增加一点惊喜，但暂时不想做完整主动道具时，选它。

### FEA-07 状态效果 / 元素伤害

**玩家价值**

燃烧、中毒、冰冻、减速、易伤、流血等状态能让武器、遗物和敌人机制更丰富。它是后续内容扩展的“语法层”。

### FEA-09 精英 / miniboss

**玩家价值**

精英敌人能制造短期目标和压力峰值。玩家看到更大、更醒目的敌人，会自然改变走位和火力集中。

- 精英有体型倍率、血量倍率、伤害倍率、掉落倍率。
- 视觉上有外圈或独特色。
- 死亡时掉落大量经验 / 宝箱 / 额外升级机会。

**可能改哪些文件**

**风险**

- 视觉必须够醒目，否则玩家不知道为什么压力变了。

**什么时候适合选**

当你想在不做 Boss 的情况下增加节奏峰值时，选它。

### FEA-10 Boss 首片

- 生成大体型 Boss，有独立血条。
- Boss 有 1 个移动模式和 1 个攻击模式，例如周期性冲刺或圆形弹幕。
- 击败后进入胜利结算或发放高额奖励。

**可能改哪些文件**

**风险**

### FEA-11 刷怪导演 / 强度曲线

**玩家价值**

当前刷怪主要是按时间窗配置。刷怪导演可以根据玩家强度、实体数量、时间、击杀速度来调节压力，让局内节奏更稳定。

**风险**

**什么时候适合选**

### FEA-12 Hazard 机关 runtime

**玩家价值**

机关能让地图不只是空旷平面。地刺、毒雾、火圈、落雷、旋转伤害区都能制造走位压力，让玩家不能只绕圈。

**当前前置条件 / 可复用基础**

**第一片怎么做**

- `f9-demo-smoke` 验证指定时间段出现 hazard。
- `save-smoke` 决定是否保存 active hazards。
- 手动试玩看预警是否足够。

**复杂度**

M/L。

**风险**

- 预警时间短会显得不公平。

- active hazards 是否进入 run snapshot 要提前定。

**什么时候适合选**

当你想增强地图走位和压力变化时，选它。它比 Boss 更早产生“场地感”。

### FEA-13 地图兴趣点 / 软边界

**玩家价值**

大地图如果只有无限背景，玩家会缺少方向感。兴趣点和软边界可以让玩家有“去那边看看”的目标，例如宝箱、祭坛、商人、挑战区域、危险高收益区。

软边界可以后做：

- 离出生点太远后刷怪变强、背景变暗或出现危险提示。

**可能改哪些文件**

- replay runner 不因非确定生成漂移。
- 手动试玩看指示是否清楚、距离是否合理。

**复杂度**

**风险**

**什么时候适合选**

当你想让游戏从“原地生存”变成“有探索味道的生存”时，选它。

优点：玩家感知最明显，长期扩内容价值高。

主要风险：会触碰 run save / replay / UI，必须拆工作包做。

### 路线 B：最快提升局内变化和压力

适合目标：让玩家 3~5 分钟内遇到更多不同局面。

1. FEA-12 Hazard 机关 runtime。
2. FEA-09 精英 / miniboss。
3. FEA-13 地图兴趣点 / 软边界。
4. FEA-14 宝箱 / 事件 / 奖励点。

优点：走位和地图更有变化，截图 / 试玩反馈会更丰富。

主要风险：难度曲线容易突然变尖，需要手动试玩和 perf-probe。

### 路线 C：最快提升复玩和开局选择

适合目标：让玩家愿意换角色 / 换规则再开一局。

1. FEA-18 角色选择与角色差异。
2. FEA-19 游戏模式 / 难度 / Challenge。
3. FEA-16 解锁条件与收藏图鉴。
4. FEA-17 成就 / 挑战任务。

优点：标题菜单和局外目标更完整，Demo 复玩感提升快。

主要风险：数值差异如果太浅会像换皮，太深又会影响平衡。

### 路线 D：最快提升发行完成度

适合目标：准备把 Demo 给外部玩家或录制展示素材。

1. FEA-23 教学 / 新手引导。
2. FEA-24 结算与局外引导强化。
3. FEA-30 AudioManager 资源接入。
4. FEA-37 Demo 发行清单。
5. FEA-34 UI 主题 / 品牌包装。

优点：少碰核心战斗风险，外部试玩体验更稳。

主要风险：音频和美术资源依赖较强。

### 路线 E：最快提升后续开发效率

适合目标：后面要连续加很多内容，希望先把工具和调试铺好。

1. FEA-25 内容模板 / 数据生产流水线。
2. FEA-29 Replay / DebugTools 扩展。
3. FEA-28 AIPlayer / 平衡 sim。
4. FEA-26 本地 mod 管理 UI。

优点：后续每个内容包都会更安全。

主要风险：玩家短期感知弱，容易“工程很好但游戏没变”。

## 13. 点名推进时的推荐说法

- “推进 FEA-01，只做第一片。”
