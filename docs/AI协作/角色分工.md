# AI 角色分工

> 长任务/复杂改动时，单一 AI agent 一把梭容易"既设计又实现还自评"——质量难保。
> 本约定把工作分成四个角色，可由同一 agent 串行扮演（在对话里显式切换），也可团队模式并行。

---

## 四个角色

### 1. 设计 (Designer)
- **职责**：理解需求 → 出 ADR 草稿 + 数据契约草稿（涉及新 stat/effect/event 时先在词表登记）
- **产出**：`决策记录.md` 新增条目（标"草稿"）+ `词表与契约.md` 新增 id（标"草稿"）+ 设计描述
- **不做**：实现代码；具体数据填入
- **完成标志**：草稿可被人评审通过 → 转给"实现"

### 2. 实现 (Implementer)
- **职责**：按设计草稿写代码 / 填数据 / 改本地化
- **输入**：设计角色的草稿 + 编码规则
- **产出**：代码 + 数据条目 + locale 条目；通过 `实时验证回路.md` 的本地校验
- **不做**：自评（交给 Reviewer）；自创设计（交给 Designer）
- **完成标志**：本地 hook 全过 → 转给"评审"

### 3. 评审 (Reviewer)
- **职责**：按 PR 模板的「自检清单」逐条核对
- **输入**：实现角色的产出 + `.codebuddy/rules/game-coding-rules.md` 自检清单
- **产出**：通过 / 发回（带具体规则号 + 文件 + 行号）
- **不做**：自己改实现（最多给小修建议，大问题打回）
- **完成标志**：清单全过 + 文档同步 → 转给"平衡"或入库

### 4. 平衡 (Balancer)
- **职责**：用回放/Headless 模拟跑回归 + 关键平衡点采样
- **输入**：评审通过的产物 + 黄金回放 + 平衡测试入口
- **产出**：回归报告（任意敌人/遗物/掉落数值的 diff），关键 build 强度抽样
- **不做**：改代码（发现问题转给 Designer 重新立项）
- **完成标志**：报告无回归 / 回归被显式接受 → 入库

## 切换机制

### 串行（单 agent）
agent 在每个角色开始/结束时显式声明：
```
【切换至设计角色】
…设计内容…
【设计完成，切换至实现角色】
…
```
便于人快速理解阶段。

### 并行（团队模式）
用 codebuddy 的 team 工具开多个 agent：
- `designer` 出草稿
- `implementer` 等草稿出来后开干
- `reviewer` 拿到 implementer 产物后审
- `balancer` 在 PR 合入前跑回放

通信走 `send_message`，状态走 `项目记忆.md` 的"下一步候选"。

---

## 何时不分角色（避免过度工程）

简单任务（如"调一个数值"、"修一处 typo"）**不需要切角色**——直接做即可。本约定主要面向：

- 新增系统模块
- 新增效果原语
- 跨多文件的设计变更
- 涉及契约/规则的改动
