# AI 角色分工

> 长任务/复杂改动时，单一 AI agent 一把梭容易"既设计又实现还自评"——质量难保。
> 本约定把工作分成执行守门角色与创意设计角色，可由同一 agent 串行扮演，也可用 subagents 并行。
>
> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。本文档是协作角色权威；新增角色或改职责边界时，必须同步 `.codebuddy/agents/` / `.codex/agents/` / `.opencode/agents/`（如适用）、`AGENTS.md`、`CODEX.md`、`OPENCODE.md`、`docs/AI协作/README.md`、`docs/AI记忆/项目记忆.md`。

## 执行守门角色

### 1. 设计 (Designer)
- **职责**：理解需求 → 出 ADR 草稿 + 数据契约草稿（涉及新 stat/effect/event 时先在词表登记）
- **产出**：`决策记录.md` 新增条目（标"草稿"）+ `词表与契约.md` 新增 id（标"草稿"）+ 设计描述
- **不做**：实现代码；具体数据填入
- **完成标志**：草稿可被人评审通过 → 转给"实现"

### 2. 实现 (Implementer)
- **职责**：按设计草稿写代码 / 填数据 / 改本地化
- **输入**：设计角色的草稿 + 编码规则
- **产出**：代码 + 数据条目 + locale 条目；通过 `实时验证回路.md` 的本地校验
- **不做**：自评（交给 Reviewer）；自创设计（交给 Designer）
- **完成标志**：本地 hook 全过 → 转给"评审"

### 3. 评审 (Reviewer)
- **职责**：按 PR 模板的「自检清单」逐条核对
- **输入**：实现角色的产出 + 当前平台编码规则入口的自检清单
- **产出**：通过 / 发回（带具体规则号 + 文件 + 行号）
- **不做**：自己改实现（最多给小修建议，大问题打回）
- **完成标志**：清单全过 + 文档同步 → 转给"平衡"或入库

### 4. 平衡 (Balancer)
- **职责**：用回放/Headless 模拟跑回归 + 关键平衡点采样
- **输入**：评审通过的产物 + 黄金回放 + 平衡测试入口
- **产出**：回归报告（任意敌人/遗物/掉落数值的 diff），关键 build 强度抽样
- **不做**：改代码（发现问题转给 Designer 重新立项）
- **完成标志**：报告无回归 / 回归被显式接受 → 入库

## 创意设计 subagents

这些角色默认是 advisory，不直接写业务代码；主要输出方案、评估、参考对象、风险和落地建议。

| subagent | 何时调用 | 主要产出 |
|----------|----------|----------|
| `game-designer` | 用户提出玩法 / 系统 / 机制 / 循环 / 关卡 / 存档设计 | 优缺点、参考对象、风险、替代方案、是否需要 ADR |
| `numeric-designer` | 需要生命、伤害、成长曲线、掉落、局外成长成本、难度波次 | 数值目标、曲线、范围、表格、验证指标 |
| `ip-designer` | 需要世界观、主题、角色、阵营、怪物生态、长期 IP 框架 | IP 核心句、世界规则、命名体系、内容扩展框架 |
| `copywriter-packager` | 需要 UI、道具、遗物、商店页、宣传语等文案包装 | zh_CN / en 草案、locale key 建议、占位符检查 |
| `ui-art-designer` | 需要 HUD、菜单、升级选择、局外成长界面、图标、信息层级 | UI 目标、布局、wireframe、色板、动效和可读性建议 |
| `game-art-designer` | 需要角色、敌人、场景、子弹、特效、图标、资产规格 | 形状语言、色彩、动画、资产 brief |
| `marketing-strategist` | 需要定位、卖点、Steam 页面、预告片、截图、Demo、节日投放 | 宣发定位、核心卖点、资产策略、反向设计建议 |

创意设计角色给出的内容要回到项目权威文档落地：新设计进 GDD / ADR，新 id 进词表，新数值进 `client/data/README.md` 与 CSV / JSON，新文案进 `client/locale/README.md` 与 CSV。

## 切换机制

### 串行（单 agent）
agent 在每个角色开始/结束时显式声明：
```
【切换至设计角色】
…设计内容…
【设计完成，切换至实现角色】
…
```
便于人快速理解阶段。

### 并行（团队模式）
支持团队 / 子 agent 的平台可开多个 agent（不支持时由同一 agent 串行扮演）：
- `designer` 出草稿
- `implementer` 等草稿出来后开干
- `reviewer` 拿到 implementer 产物后审
- `balancer` 在 PR 合入前跑回放
- 创意方向可按任务并行开 `game-designer` / `numeric-designer` / `ip-designer` / `copywriter-packager` / `ui-art-designer` / `game-art-designer` / `marketing-strategist`

通信走平台原生消息机制，长期事实走 `项目记忆.md`，当前状态和下一步走 `docs/AI记忆/current_state.json`。

## 何时不分角色（避免过度工程）

简单任务（如"调一个数值"、"修一处 typo"）**不需要切角色**——直接做即可。本约定主要面向：

- 新增系统模块
- 新增效果原语
- 跨多文件的设计变更
- 涉及契约/规则的改动
