# 代码审核流程

> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。
> 本文档是 AI / 人执行代码审核时的流程权威；改审核顺序、lint 分层、warning 处理或 Reviewer 职责时，必须同步三平台 `code-review-factual` skill、`docs/AI协作/角色分工.md`、`docs/AI协作/README.md`、`docs/AI导航.md`、`docs/决策记录.md` 与 AI 记忆。

## 目标

代码审核采用“工具先行，LLM 聚焦差异”的流程：能由 lint / test / docs check 判断的规则先交给脚本；LLM 不重复全仓巡逻这些规则，只解释工具输出、审查当前 diff 的行为风险，并补足脚本覆盖不到的设计 / 架构判断。

## 标准流程

1. **确认范围**
   - 先看 `git status --short` 与本次任务目标。
   - 只审核本次任务相关 diff；用户已有脏改动、`draft/` / `DRAFT/` 默认不读、不 stage、不提交。

2. **先跑工具**
   - 本地已安装 pre-commit 时，优先跑 `pre-commit run --all-files` 或 `pre-commit run --files <changed>`。
   - 没装 pre-commit 时，按 `docs/AI协作/实时验证回路.md` 的等价命令运行。
   - 阻塞型失败必须先修；不要在红灯状态下进入“已通过 review”的结论。

3. **处理第三档 advisory**
   - `tools/lint_semantic_rules.py` 默认非阻塞；warning 必须人工判断。
   - 每条 warning 归为三类之一：`fix`（确实违反项目规则，修）、`accept`（合理例外或误报，说明理由）、`defer`（风险存在但不属本轮，记录后续）。
   - 不允许无视 warning；也不允许为了清零 warning 做无意义重构。

4. **LLM 只重点读 diff**
   - 已由工具覆盖的规则不再靠 LLM 全仓重复扫描。
   - LLM 审查重点是：行为 bug、回归风险、数据 / 存档 / 回放 / autoload 语义是否正确、测试义务是否匹配、文档同步是否足够。
   - 发现问题时给出文件和行号；没发现实际问题就明确说“未发现问题”。

5. **输出结论**
   - 有问题：按严重度列 findings，随后列测试缺口和建议。
   - 无问题：说明工具状态、advisory 处理结果和剩余风险。

## 等价命令

没有 pre-commit 时，至少运行：

```powershell
python tools/sync_contracts.py --check
python tools/validate_data.py
python tools/test_data_loader_schema.py
python tools/lint_gdscript_rules.py
python tools/lint_project_rules.py
python tools/test_project_rules_lint.py
python tools/lint_semantic_rules.py
python tools/test_semantic_rules_lint.py
python tools/docs_health_check.py
git diff --check -- . ":(exclude)draft/**" ":(exclude)DRAFT/**"
```

涉及 Godot 启动、autoload、场景或运行时脚本时追加：

```powershell
python tools/godot_bridge.py headless-boot
```

## Reviewer 输出模板

```markdown
Findings:
- 未发现问题。

工具状态:
- pre-commit / lint / tests: passed
- semantic advisory: none / accepted with reason

剩余风险:
- 未跑 Godot headless / 未覆盖 GUT / 本轮无需额外测试，因为 ...
```
