# 工具适配指南 —— 通用 AI agent 接入

> 本项目核心资产与具体 AI 工具无关：`AGENTS.md`、`docs/`、任务模板、agent prompt、command 步骤都是 markdown。
> `.codebuddy/`、`.codex/`、`.opencode/`、`CLAUDE.md`、`CODEX.md`、`OPENCODE.md` 是平台适配层，只负责让不同工具更容易加载同一套项目约束。
>
> 本文件告诉你：不同 AI 工具该读哪个入口、如何复用 subagent / slash command 资产。
>
> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。本文档是跨工具适配权威；新增或调整 AI 平台配置时，常见联动为 `AGENTS.md`、`CLAUDE.md`、`CODEX.md`、`OPENCODE.md`、`.codebuddy/` / `.codex/` / `.opencode/` 对应配置、`docs/AI协作/README.md`、`docs/AI导航.md`、`docs/AI记忆/项目记忆.md`。

## 🎯 先记住一条万能兜底

不论换什么工具，**每次新对话开头说一句**：

> 先读 `AGENTS.md`。

`AGENTS.md` 是本项目所有 AI agent 的通用入口。Claude Code 可先读 `CLAUDE.md`，Codex 可先读 `CODEX.md`，OpenCode 可先读 `OPENCODE.md`；不自动读入口文件的工具（ChatGPT / Copilot Chat / Claude Web 等），你手动提一句也立即生效。

`AGENTS.md` 已写明快速开工 5 步：日常先走 `docs/AI协作/快速开工.md` + `docs/AI记忆/current_state.json` + 平台规则 + `docs/AI导航.md` 相关段，完整长期记忆、词表、GDD、ADR 和测试策略按任务触发读取。

默认沟通语言、需求评估、需求不明先问、上下文恢复、AI Git 提交策略和 `draft/` 禁区均以 `AGENTS.md` 与当前平台规则入口为权威；本指南只保留接入路径，不复制完整规则正文。

项目维护者已授权支持 subagent 的平台在复杂、专业或可并行任务中主动启用对应项目 subagent，无需每次再次确认；只读小任务或直接实现更高效时不必强行拆分。若工具没有原生 subagent 或运行时策略临时限制调度，则读取同名 agent `.md` 作为 prompt 模板执行。

## 🪟 Windows PowerShell 稳定执行

本节是当前 shell 为 PowerShell 时的可复制模板；强制语义见各平台编码规则第 29 节。其他 shell 使用自身原生语法，不要在 PowerShell 中嵌套 `cmd` / Bash。

PowerShell 自动变量不能拿来当普通可写变量：尤其不要赋值 `$Host`、`$PID`、`$HOME`、`$PWD`、`$Error`、`$Args` 或 `$Input`。需要表达服务器进程、主机日志等含义时使用 `$hostProcess`、`$hostLog` 等明确名称。

### 固定字符串搜索（默认）

`rg` 的全部选项必须放在 `--` 前；`--` 后只放 pattern 和路径。`rg` 返回 `1` 只是“无匹配”，处理后要把 `$LASTEXITCODE` 归一化，避免后续 fail-fast 调度误报。

```powershell
$paths = @('AGENTS.md', 'docs')
& rg -n -F -g '!draft/**' -g '!DRAFT/**' -- 'PowerShell' $paths
$code = $LASTEXITCODE
if ($code -gt 1) {
    throw "rg failed with exit code $code"
}
if ($code -eq 1) {
    Write-Output 'No matches.'
}
$LASTEXITCODE = 0
```

### 正则搜索

确需正则时把 pattern 放进单引号字符串；PowerShell 不使用 Bash 的 `\"` 规则转义引号。

```powershell
$pattern = 'standalone|has_feature.*template'
& rg -n -g '!draft/**' -g '!DRAFT/**' -- $pattern 'output/steamworks_lab'
$code = $LASTEXITCODE
if ($code -gt 1) {
    throw "rg failed with exit code $code"
}
$LASTEXITCODE = 0
```

### PowerShell cmdlet 与中文路径

cmdlet 的路径参数使用 `-LiteralPath`，失败使用 `-ErrorAction Stop` / `try-catch`；`$LASTEXITCODE` 只用于外部原生程序。

```powershell
$ErrorActionPreference = 'Stop'
$state = Get-Content -Raw -LiteralPath 'docs/AI记忆/current_state.json' -ErrorAction Stop
$state | ConvertFrom-Json | Out-Null
```

### 原生程序与动态参数

路径和参数用变量 / 数组传入，禁止 `Invoke-Expression` 或拼接完整命令字符串。

```powershell
$godot = 'C:\path\to\godot.exe'
$project = 'output\steamworks_lab'
if (-not (Test-Path -LiteralPath $godot -PathType Leaf)) {
    throw "Godot executable not found: $godot"
}
& $godot --headless --path $project --quit-after 1
if ($LASTEXITCODE -ne 0) {
    throw "Godot failed with exit code $LASTEXITCODE"
}
```

```powershell
$changedFiles = @('AGENTS.md', 'docs/AI导航.md')
& pre-commit run --files @changedFiles
if ($LASTEXITCODE -ne 0) {
    throw "pre-commit failed with exit code $LASTEXITCODE"
}
```

### 接受 `git diff --no-index` 的差异退出码

Git 对缺失输入也可能返回 `1`，所以必须先用 `Get-Item -LiteralPath ... -ErrorAction Stop` 校验两个输入都是文件；只有校验通过后，`1` 才表示“存在差异”，`>=2` 表示命令错误。

```powershell
$left = Get-Item -LiteralPath 'AGENTS.md' -ErrorAction Stop
$right = Get-Item -LiteralPath 'CODEX.md' -ErrorAction Stop
if ($left.PSIsContainer -or $right.PSIsContainer) {
    throw 'git diff --no-index inputs must be files.'
}
& git diff --no-index -- $left.FullName $right.FullName
$code = $LASTEXITCODE
if ($code -gt 1) {
    throw "git diff --no-index failed with exit code $code"
}
$different = $code -eq 1
$LASTEXITCODE = 0
Write-Output "Files differ: $different"
```

允许非零退出码的命令必须在各自调用内完成上述判断和归一化，再进入并行或批量调度。一次 shell 调用只承载一个语义操作；仓库文件修改优先使用当前平台的结构化 patch / edit 工具。

### Steamworks Lab 自动回归

Steamworks Slime Lab 不使用本节的原始 Godot 示例拼接多进程 smoke。统一在仓库根运行 `py -3 tools\steamworks_lab_toolchain.py smoke --suite <目标>`；先跑目标 suite，交付前跑 `--suite all`。runner 负责 Godot 4.7 / Windows console 解析、隔离 `user://`、动态 ENet 端口、host `READY`、client 轻量副本、超时终止、精确 `ALL PASS` 与错误日志判定，禁止另写 `Start-Process` 双进程脚本。会读写配置或存档的 fixture 必须在节点 `_ready()` 前注入，不能备份玩家真实文件。

## 🛠️ 各工具具体配法

### A. Claude Code（Anthropic 官方 CLI）

**本项目已内置 `CLAUDE.md` 入口 + 项目自有的活跃 `.claude/` 原生配置**（ADR #87），与 `.codebuddy/` / `.codex/` / `.opencode/` 同源、核心语义一致：

- `CLAUDE.md` — Claude Code 入口速查；读完后回到 `AGENTS.md`
- `.claude/rules/game-coding-rules.md` — Claude 侧编码规则入口（与三平台规则同源）
- `.claude/agents/` — 10 个项目级 subagents（执行类 `balancer` / `contract-validator` / `data-author`；创意类 `game-designer` / `numeric-designer` / `ip-designer` / `copywriter-packager` / `ui-art-designer` / `game-art-designer` / `marketing-strategist`），frontmatter 工具名按 Claude 命名（Read / Grep / Glob / Edit / Write / Bash）
- `.claude/commands/` — 5 个 slash commands（`/sync-contracts` / `/new-relic` / `/run-replay-regression` / `/health-check` / `/update-memory`）
- `.claude/skills/` — 9 个项目级 skills（四平台同步）；也可读取 `.codebuddy/skills/`、`.codex/skills/` 或 `.opencode/skills/` 下同名 skill
- `.claude/settings.json` — 当前只配 `draft/` / `DRAFT/` 的 deny（草稿禁区 harness 级强制）；权限 allow 白名单按需由用户单独授权后再加
- Claude Code 需在**仓库根**目录启动才会自动加载 `.claude/`；从上级目录启动则不会自动加载

> 此 `.claude/` 是项目自建配置，**不是** ADR #56~#59 / #72 拒绝的外部 `.claude/` 整包（CCGS / ECC 的 hooks、外部 agents、settings、vendor）；不接外部 vendor reference、不引入活跃 hooks。ECC 全工具面筛选记录见 `docs/AI协作/ECC工具吸收清单.md`。

外部 AI 库若与本项目 `AGENTS.md`、平台规则、GDD 或 ADR 冲突，以本项目规则为准。

### A2. OpenAI Codex CLI（零成本，已内置适配）

**本项目已内置 `CODEX.md` 与 `.codex/` 配置**，用于 OpenAI Codex CLI。它与 `.codebuddy/` 共享同一套项目核心约束，但允许按 Codex 的工具名、frontmatter、提示风格与命令入口做专属优化：

- `CODEX.md` — Codex CLI 入口速查，读完后回到 `AGENTS.md`
- `.codex/agents/` — 项目级 subagents（执行类：`balancer` / `contract-validator` / `data-author`；创意设计类：`game-designer` / `numeric-designer` / `ip-designer` / `copywriter-packager` / `ui-art-designer` / `game-art-designer` / `marketing-strategist`）
- `.codex/commands/` — 5 个 slash commands（`health-check` / `new-relic` / `run-replay-regression` / `sync-contracts` / `update-memory`）
- `.codex/rules/game-coding-rules.md` — Codex 侧编码规则入口（核心语义与 CodeBuddy 规则一致，可按 Codex 表达优化）

Codex CLI 会自动读取 `.codex/` 下的规则与配置。你无需做任何额外设置。

> **多平台适配规则（规则 21）**：修改 `.codebuddy/` 或 `.codex/` 时，不要求另一侧字节级同步；但如果改动涉及项目红线、数据契约、测试义务、文档维护等核心规则，两侧必须表达同一语义。平台专属 agent / command 优化可以只改对应平台，并在必要时更新本指南。

### A3. OpenCode（零成本，通用入口）

**本项目已内置 `OPENCODE.md` 与 `.opencode/` 配置**，用于 OpenCode 快速接入。`.opencode/opencode.json` 按 OpenCode schema 注册命令并加载入口说明：

OpenCode 使用方式：

- 先读 `OPENCODE.md`，再按 `AGENTS.md` 的快速开工 5 步继续。
- 强制编码规则入口为 `.opencode/rules/game-coding-rules.md`；完整规则正文也由 `.opencode/opencode.json` 加载 `.codebuddy/rules/game-coding-rules.md`。
- 需要 subagent 时，优先用 `.opencode/agents/*.md`；角色语义与 `.codebuddy/agents/`、`.codex/agents/` 同步。
- 需要 slash command 时，优先用 `.opencode/opencode.json` 注册的命令；步骤正文在 `.opencode/commands/*.md`。
- 需要可复用流程时，OpenCode 优先加载 `.opencode/skills/*/SKILL.md`，Codex / CodeBuddy 读取各自同名 skill；当前 skills 覆盖 Godot/GDScript、场景验证、Godot 测试诊断、试玩复盘、文档同步、安全提交、事实 review、AI 资源筛选与协作面审计、MCP 评估。
- 外部 AI 资源不再保留整包 vendor、plugin、hooks、整目录 skills 或大批量 subagents；只把有用流程吸收进项目级 skill。

改 `.opencode/` 后必须重启 OpenCode；运行中的 session 不会热重载配置。

### B. Cursor

`.cursor/rules/*.mdc` 启用，但 frontmatter 字段不一样。**最省事做法**：建一个 stub 转发到原文件。

```bash
mkdir -p .cursor/rules
```

新建 `.cursor/rules/main.mdc`：

```markdown
---
description: 主项目规则
globs: ["**/*"]
alwaysApply: true
---

完整规则见 `AGENTS.md` 第 3 步指定的当前平台编码规则入口；无专属入口时读 [.codebuddy/rules/game-coding-rules.md](../../.codebuddy/rules/game-coding-rules.md)。

开工前必读：
- AGENTS.md
- docs/AI协作/快速开工.md
- docs/AI记忆/current_state.json
- docs/AI导航.md
```

Cursor 会把这份 stub 一直挂在上下文，里面的相对链接 Cursor 也会顺着读。

> Subagents/commands 在 Cursor 里没原生入口，但你可以把 `.codebuddy/agents/balancer.md` 当 prompt 片段——`@balancer.md` 引用即可。

## 🤖 关于 subagents 与 slash commands

它们是 CodeBuddy / Codex / OpenCode / Claude Code 等工具的"调度糖"。本项目默认允许支持的平台主动调度 subagent；**不支持也没关系**：

- `.codebuddy/agents/*.md` / `.codex/agents/*.md` / `.opencode/agents/*.md` 同时是**精确的角色 prompt 模板**
- `.codebuddy/commands/*.md` / `.codex/commands/*.md` / `.opencode/commands/*.md` 同时是**精确的操作步骤手册**
- `.codebuddy/skills/*/SKILL.md`、`.codex/skills/*/SKILL.md` 与 `.opencode/skills/*/SKILL.md` 是三平台同步的项目级可复用流程；不支持原生 skills 的平台可把对应 `SKILL.md` 当普通 prompt 模板读取

当前 subagents 分两类：

| 类型 | agents |
|------|--------|
| 执行 / 守门 | `data-author` / `contract-validator` / `balancer` |
| 创意 / 策略 | `game-designer` / `numeric-designer` / `ip-designer` / `copywriter-packager` / `ui-art-designer` / `game-art-designer` / `marketing-strategist` |

当前项目级 skills：三平台 `skills/` 下均有 `godot-gdscript` / `godot-scene-validation` / `godot-test-diagnostics` / `playtest-review` / `project-doc-sync` / `safe-git-commit` / `code-review-factual` / `ai-resource-curator` / `mcp-tool-evaluation`；其中 `ai-resource-curator` 同时负责外部资源筛选和 AI surface audit。

需要按需使用项目级 skill 时直接说：

> 按 `.codex/skills/playtest-review/SKILL.md`（或同名 `.codebuddy/skills/` / `.opencode/skills/`） 复盘这次 MVP 试玩，并按项目规则给出优先级建议。

所以哪怕你的工具不识别它们，要"跑平衡分析"时直接说：

> 按 `.opencode/agents/balancer.md`、`.codebuddy/agents/balancer.md` 或 `.codex/agents/balancer.md` 的角色定位帮我跑回放回归。

要"同步词表常量"时直接说：

> 按 `.opencode/commands/sync-contracts.md`、`.codebuddy/commands/sync-contracts.md` 或 `.codex/commands/sync-contracts.md` 的步骤跑同步。

大型代码改动完成后，AI 应在提交前自动使用事实型 code review 口径；不支持原生 skill 的工具可直接说：

> 按 `.codex/skills/code-review-factual/SKILL.md`（或同名 `.codebuddy/skills/` / `.opencode/skills/`）和 `docs/AI协作/代码审核流程.md` 审核当前 diff：先看 pre-commit / lint / test / docs 输出，再只列实际 bug、回归风险和缺测试；没发现问题就明确说未发现问题。

效果一样——只是少了 `/` 这个快捷键。

## 📋 一次性适配 checklist

不论换什么工具，做这些检查：

- [ ] 仓库根有 `AGENTS.md`（已有 ✅）
- [ ] 如果使用 Claude / Codex / OpenCode，仓库根有 `CLAUDE.md` / `CODEX.md` / `OPENCODE.md`，并有 `.claude/` / `.codex/` / `.opencode/` 配置（已有 ✅）

- [ ] （可选）按上面 A~G 节做工具专属的规则加载配置
- [ ] 跑一次冷启动测试：新开一个对话，让 AI 用中文描述本项目当前状态——它应当能答出"v1.5、最新 ADR 见 `current_state.json`、正式项目 F3 DataLoader 已收口、当前按 F4 MinPlayableLoop 工作包推进"

如果冷启动测试答得出，说明上下文加载成功；答不出就是规则文件没被读到，回去检查工具配法。
