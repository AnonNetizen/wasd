# 文档维护指南（AI 修改说明）

> 本文件告诉 AI agent：修改项目文档时，哪些文件必须一起看、一起改、一起验证。
> 目标是防止“改了 A，漏了 B/C”的文档漂移。
>
> **AI 修改说明**：本文档是文档维护规则的权威入口。修改本文档时，必须同步检查规则 22/24（`.codebuddy/rules/game-coding-rules.md` / `.codex/rules/game-coding-rules.md` / `.opencode/rules/game-coding-rules.md`）、`AGENTS.md`、`CLAUDE.md`、`CODEX.md`、`OPENCODE.md`、`docs/代码文档规范.md`、`docs/AI协作/README.md`、`docs/AI导航.md`、`docs/决策记录.md` 与 `docs/AI记忆/项目记忆.md`。

---

## 1. 总原则

- 改任何长期文档前，先读该文档开头的 **AI 修改说明**。
- 如果该文档没有 AI 修改说明，先补说明，再继续修改正文。
- 改动必须遵守“单一权威来源”：
  - 编码规则权威：当前平台编码规则入口（`.codebuddy/rules/game-coding-rules.md` / `.codex/rules/game-coding-rules.md` / `.opencode/rules/game-coding-rules.md` 等）
  - 设计权威：`docs/游戏设计文档.md`
  - 约定字符串权威：`docs/词表与契约.md`
  - 测试权威：`docs/测试策略.md`
  - 代码对应文档权威：`docs/代码文档规范.md` + `docs/代码/`
  - 决策权威：`docs/决策记录.md`
  - 跨会话长期记忆权威：`docs/AI记忆/项目记忆.md`
  - 机器当前状态权威：`docs/AI记忆/current_state.json`
- 不要只改引用方；权威文档变了，索引、记忆、模板、CI 规划要按联动清单检查。

---

## 2. 新文档必须包含的 AI 修改说明

长期维护文档的开头必须放一段短说明，位置在标题和正文之间：

```markdown
> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。
> 本文档的权威范围：<一句话说明本文件管什么>。
> 常见联动：<列出 2~5 个最常见的同步文件>。
```

短文件可以只写一行，但必须明确：

- 本文档管什么
- 改它时最容易漏改什么
- 是否需要更新 ADR / 项目记忆 / AI导航

## 4. 修改流程

1. 读目标文档开头的 AI 修改说明。
2. 按上表列出本次可能联动的文件。
3. 代码变更先按 `docs/代码文档规范.md` 判断对应文档，再改权威来源、索引 / 指南 / 记忆。
4. 如果产生新规则或新决策，必须追加 ADR，并更新项目记忆。
5. 修改后用 `rg` 检索旧词、旧编号、旧路径，确认没有矛盾口径残留。
6. 知识库结构变化后运行 `python tools/docs_health_check.py`。
7. 在最终回复里说明改了哪些权威文件，以及是否有未验证项。

## 5. AI 友好文档写法

- 标题稳定、层级稳定，方便被链接和检索。
- 表格用于“映射关系”和“必须同步清单”；长段落用于解释原因。
- 每个约定写清“权威来源”和“引用方”，不要让规则散在正文里无人知道。
- 新增条目优先追加，不随意重排历史编号。
- 涉及路径时写真实相对路径，避免“见上文”“同前”这类弱引用。
- 重要变更用一句话结论 + 一句话理由，和 ADR 风格保持一致。
