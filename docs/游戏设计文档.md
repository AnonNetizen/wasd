## 1. 游戏概述

### 1.1 一句话定位
《破巢者》是一款多英雄固定斜俯视 2.5D 射击刷宝生存游戏：玩家选择一名英雄主动杀进敌方“巢”的腹地，用 WASD / 手柄左摇杆在 2D 平面移动，默认用鼠标控制射击朝向（手柄右摇杆 / D-pad 与方向键作为兜底输入），按住左键 / 右扳机持续开火、松开停火；显示上参考《哈迪斯》的斜俯视舞台表达，`Camera2D` 保持屏幕水平和等比缩放，斜俯视感由量化菱形地图格、地面美术、资产落地点 / 阴影、障碍物、遮挡层级、房间 / 舞台构图和低模角色表现承担，通过反复刷取装备、词条、射击强化技能、消耗品、被动遗物与局外成长不断变强，在机关与敌群中推进破巢行动并尝试打穿巢核。游戏可以保留局内随机奖励、单局构筑和生存压力，但核心体验是玩家主动射击突破敌巢。

### 1.2 IP 核心设定
**核心句**：通道已经打开，防守只会慢慢死；真正能伤到敌人的办法，是派一名英雄杀进敌人的老家。

本项目的 IP 方向定为 **《破巢者》**（英文暂定 `Nestbreakers`）。未知原因导致其他宇宙与本宇宙的通道突然打开，原本拥有银河系星际文明的人类被跨宇宙敌群打得七零八落，只剩首都星域和少数核心据点受到影响较轻。这里的“巢”不是单指虫巢，而是敌方在本宇宙建立或拖拽进来的核心据点、生产源头、通道锚点或意志中枢；每局是一场“破巢行动”：一名英雄从外围切入敌方腹地，在无尽守军、机关和精英围攻中夺取巢内遗物，把敌人的力量反过来变成自己的构筑，并尝试破坏巢核、切断通道或让敌方战区失去供给。完整世界规则、英雄 / 敌方势力方向、遗物包装、命名体系和商标注意见 `docs/IP设定.md`。

### 1.3 设计支柱
- **射击核心、构筑服务枪线**：移动、瞄准与按住开火是主操作，技能、遗物和词条优先强化弹道、射速、穿透、分裂、过载、弱点和命中触发，而不是抢走主武器的表达。
- **掉落与构筑驱动（Loot-driven / Build-driven）**：装备、词条、技能、遗物和局外成长共同形成可长期追求的 build；局内随机奖励服务刷宝循环，而不是旧类型定位本身。
- **美术极简**：几何/像素风格，便于 AI 批量生成素材，也便于程序化生成。
- **数据与逻辑分离**：所有可调数值集中在独立配置文件，策划 / 玩家可零代码调整；每类数据必须有可人工上手的字段说明文档。
- **扩展优先（Extension-first）**：当前操作、属性、栏位、摄像机等玩法规则是默认配置，不是写死上限；未来角色 / 道具若要“破限”，必须通过数据声明的 capability、effect 原语或可复用策略扩展，而不是散落特殊分支。
- **模式友好资源复用**：角色、遗物、道具、敌人、成长奖励等资源本体默认不绑定单一游戏模式；模式只通过数据化的可用池、权重、条件和轻量覆盖来组合资源，避免为某个模式复制或硬编码一套内容。
- **未来多人友好边界**：当前只做单人，但输入、伤害、阵营、模式、回放和存档避免写死“唯一玩家 / 玩家对敌人”假设；为未来多人 PvE / PvP 保留 actor、participant、team 与 intent 边界，不提前实现联网。
- **框架级基础设施**：从设计阶段就内建**多语言本地化**与**可扩展的设置系统**，所有玩家可见文本走本地化键、所有玩家偏好走统一设置管理器；文案表必须便于人工逐语言校对，UI 尺寸以英文文案长度作为本地化验收基准。

### 1.4 目标平台与引擎
- 平台：PC（键盘 + 手柄）。
- 引擎：**Godot 4.7（GDScript）**，详见第 9 节。

## 2. 核心玩法循环（Core Loop）

```
选择英雄 → 突入敌巢 → 移动探索大地图 → 按住开火并主动瞄准清怪 → 拾取掉落（经验/金币/装备/词条/道具）
   → 变强（升级/遗物/射击强化技能/装备词条/局外成长） → 遭遇更强敌群与机关 → 打穿巢核/撤离/死亡 → 结算战利品 → 整理构筑 → 再次开局
```

- **短循环（秒级）**：移动走位 → 按住开火 / 调整准星 → 躲机关/弹幕 → 拾取。
- **中循环（分钟级）**：升级选择 → 获取遗物 / 装备词条 / 技能资源 → 充能并释放主动道具或射击强化技能。
- **长循环（一局/跨局）**：难度随时间递增 → 结算战利品、局外货币和账号经验 → 整理装备 / 词条 / 永久升级 / 解锁新内容 → 再次开局刷更高危险等级。

## 3. 玩家系统

### 3.1 移动
- 键盘 WASD 与手柄左摇杆控制 8 方向移动，移动速度由 `move_speed` 决定。
- 建议加入轻微加速度/惯性（可配置），手感更顺滑；初版可先做即时速度。

### 3.2 射击（按住开火）
- **朝向**：键鼠默认用鼠标相对玩家 / 视口中心方向控制射击方向，子弹可朝任意角度发射；手柄右摇杆、D-pad 或键盘方向键保留为无鼠标动作时的兜底瞄准输入。角色表现只区分向左 / 向右两种朝向，不做向上 / 向下朝向。
- **开火**：默认按住 `fire` action 持续射击，松开停火；键鼠默认鼠标左键，手柄默认右扳机。开火间隔由 `fire_rate`（射速）决定，冷却在未开火时正常恢复，重新按下后可立即响应已就绪的下一发。
- **移动与瞄准解耦**：键鼠用 WASD + 鼠标瞄准，手柄用左摇杆移动 + 右摇杆 / D-pad 瞄准；可以朝右移动同时向左或斜向射击。

### 3.3 核心属性（全部可在数值文件中调整）
| 属性 | 字段名 | 说明 |
|------|--------|------|
| 生命上限 | `max_hp` | 浮点血量上限；默认角色初始为 600.0，便于按 Dota 式血量尺度微调 |
| 自动生命恢复 | `health_regen` | 每秒恢复生命；默认角色初始有少量恢复 |
| 移动速度 | `move_speed` | px/s |
| 玩家中心排斥半径 | `player_separation_radius` | 玩家中心不可重叠区域；敌人靠得过近时推开敌人，避免敌人中心贴到玩家中心 |
| 射速 | `fire_rate` | 每秒射击次数（或射击间隔） |
| 伤害 | `damage` | 单发子弹伤害 |
| 子弹速度 | `bullet_speed` | px/s |
| 子弹射程 | `bullet_range` | px 或按武器换算为存活时间 |
| 子弹数量 | `bullet_count` | 一次开火发射的子弹数 |
| 暴击率 / 暴击伤害 | `crit_chance` / `crit_mult` | 可选扩展属性 |
| 拾取范围 | `pickup_range` | 自动吸附经验 / 金币的范围 |
| 幸运值 | `luck` | 影响掉落、稀有度、升级候选是否从 3 个提升为 4 个 |

> 属性采用「基础值 + 加法修正 + 乘法修正」的计算模型，便于遗物叠加（见 6.3）。
> 运行时应提供按住显示的详细数值面板，集中展示当前生命、等级、经验、击杀、计时、武器、拾取、幸运与主动技能资源 / 冷却等只读状态；面板属于 HUD 叠层，不进入暂停或模态 UI。

### 3.4 角色配置与破限能力
- 默认角色只是一条 `characters.json` 数据：基础属性、初始 tags、可用 capability、默认控制配置和起始携带都应来自数据；起始携带引用必须指向对应数据注册表。
- “键盘 WASD 或手柄左摇杆移动 / 鼠标相对玩家方向瞄准（手柄右摇杆、D-pad 与键盘方向键兜底）/ 按住开火 / 玩家居中 / 角色视觉只区分左右朝向”等是默认角色配置；未来角色可以通过已登记 capability 改写瞄准、开火、移动、栏位、摄像机策略，但不得在业务代码里写 `if character_id == ...` 的特殊分支。
- 角色能力优先复用 `modifiers`、`behaviors`、`StatusEffect`、`Combat`、`GameState`、`UIManager` 等既有管线；表达不了时新增可复用 primitive / strategy，并登记到 `docs/词表与契约.md`。

## 4. 子弹系统

- 子弹是独立实体，拥有位置、速度向量、伤害、存活时间/射程。
- **有飞行速度**，非瞬间命中：敌人移动时需要预判，玩家也可用走位躲避敌方子弹。
- 子弹与敌人/墙体/机关碰撞后结算（命中、穿透、反弹等行为可由遗物改写）。
- 性能要点：使用**对象池**复用子弹实体，避免频繁创建销毁导致卡顿（大地图弹幕量可能很大）。

## 5. 大地图与世界

### 5.1 地图设计
- 取消《以撒》的房间制，改为《吸血鬼幸存者》式的**开放有限大地图**：玩家可自由走位与探索，但一局地图有明确边界。
- 地图首片由 `client/data/map_layouts.json` 声明：模式 id、地图宽高、菱形格尺寸、玩家出生点、安全半径、敌人刷怪边距、PCG 机关规则和人工摆放点。
- 菱形格是当前地图的量化度量，不再只是背景参照：`grid.cell_width` / `grid.cell_height` 定义单格水平 / 垂直对角线；`bounds.width` / `bounds.height` 必须是格宽 / 格高的奇数倍，并满足 `bounds.height == bounds.width * grid.cell_height / grid.cell_width`，让边界斜率和边线落点都贴住菱形格线；玩家出生点必须落在格中心；人工机关与 PCG 机关按 `radius_tiles` 奇偶吸附到合法锚点，奇数尺寸在格心，偶数尺寸在网格顶点。
- 可见地图边界和逻辑地图边界必须使用同一套菱形：当前 `MapManager` 用 `bounds` 外接框的上 / 右 / 下 / 左四点绘制菱形边框，不再显示旧的轴对齐矩形边框；玩家、敌人、刷怪和机关摆放都 clamp 到这个菱形内。
- 玩家出生安全区属于贴地范围提示：`safe_radius` 仍作为 PCG 机关避开出生点的最小距离下限，但可见标记必须按当前菱形格斜率画成贴格菱形，并吸附到菱形格线，不再使用正圆安全圈。
- PCG 只负责生成可复现的初始地图内容；人工调整通过 `manual_hazards` 等显式摆点覆盖关键节奏和测试场景，避免全靠随机导致 Demo 难以打磨。
- 视角：当前正式口径为**固定斜俯视 2.5D**；玩法判定仍在 2D 平面内，`Camera2D` 只负责水平居中和等比缩放，不通过旋转或非等比缩放模拟透视；斜俯视观感由地面美术、障碍物、遮挡层级、房间 / 舞台构图和 2.5D 角色 / 敌人视觉层承担，但不得因此把移动、碰撞、刷怪或地图边界改成真 3D 依赖。

### 5.2 摄像机（玩家始终居中）
- **核心规则**：与《吸血鬼幸存者》一致，**玩家角色始终固定显示在屏幕正中央**，移动时是世界（地图/敌人/子弹）相对玩家滚动，而非角色在屏幕内移动。
- Godot 实现：在 `Player` 节点下挂一个 `Camera2D` 并设为 `enabled`/当前相机，相机随玩家移动即可让玩家恒居屏幕中心；当前相机必须保持屏幕水平和等比缩放，不滚转整个 2D 画面，也不使用 `zoom.x != zoom.y` 的非等比缩放。哈迪斯式斜俯视感由场景绘制和层级表达，而不是由相机压扁世界。
  - **不要**开启相机的 `limit`（边界限制），否则地图边缘会破坏居中效果；有限地图边界由 `MapManager` / `Player` 的世界坐标 clamp 和可视边界提示负责。
  - **不要**使用 `drag margin`（拖拽边距），将其关闭以保证严格居中（拖拽边距会让玩家在中心附近偏移）。
  - 可选：开启 `position_smoothing` 做轻微平滑跟随；若要绝对锁定中心则关闭平滑。
  - 背景（地块/网格）随相机移动产生滚动感，建议用 `TileMap` 或可平铺纹理表现移动参照。
  - 鼠标瞄准必须按当前 canvas transform 把屏幕偏移换算回世界方向，保证等比相机下仍然是“屏幕上指哪打哪”。

### 5.3 敌人与刷怪（Spawner）
- 敌人在玩家视野外围**按波次/时间曲线**刷新，随时间提高数量、血量与种类。
- 敌人出生、普通移动、逃离、冲锋、中心分离和续局恢复都应保持在有限地图菱形逻辑边界内；地图边界约束的是实体中心点，避免 AI 行为把敌人带出可玩区域。
- 敌群允许视觉上互相重叠以保留尸潮密度，但敌人中心应有一个小的不可重叠区域，避免多个敌人完全压成同一点；该中心间距应数据化调参。
- 敌人 AI 采用数据驱动生态 profile：同一敌人运行时可根据玩家、其他敌人 content tag、出生点 / 领地和周围环境选择接近、逃离、环绕、冲锋或守家。目标是逐步接近《雨世界》式“怪物之间也有关系”的生态感，而不是所有敌人只朝玩家冲。
- 怪物之间允许经 `Combat` 互相造成接触伤害；非玩家造成的击杀不计入玩家击杀数，也不掉经验，避免生态内斗变成刷资源漏洞。
- 敌人类型示例：
  - **追击型**：直线冲向玩家。
  - **猎物型**：数量多、单体弱，会靠近玩家但会逃离掠食者或领地怪。
  - **掠食型**：优先狩猎带猎物 tag 的敌人，近距离可蓄力冲锋。
  - **领地型**：围绕出生点活动，越界后回守，并会驱赶特定敌人。
  - **远程型**：保持距离并发射子弹（需要躲弹幕）。
  - **群集型**：数量多、单体弱，制造压迫感。
  - **精英/Boss**：定时出现，掉落更好的道具。

### 5.4 机关（Hazards）
沿用《以撒》的环境伤害机关，置于大地图中：
- 尖刺地块（踩上扣血）、火焰/喷火口（周期性）、毒池/减速带、可推动的石块、激光等。
- 机关基础数值放在 `client/data/hazards.csv`（伤害、伤害类型、触发周期、格数尺寸、持续时间），具体地图摆放放在 `client/data/map_layouts.json`。
- 当前通用机关表现为贴合哈迪斯式地面的菱形危险地块；`radius_tiles` 表示从中心到顶点占用多少个地图菱形格，必须为正整数。视觉菱形和触发判定都由 `map_layouts.json.grid` 推导出的半宽 / 半高生成；奇数尺寸中心吸附格心，偶数尺寸中心吸附网格顶点，避免出现“看着贴格、判定是另一套数学”或偶数尺寸边缘穿过格心的错位。
- 机关可由 PCG 按规则撒布，也可由人工摆点固定；两者都会进入 `run` 快照，保证暂停续局和回放诊断可复现。
- 首个 PCG 测试机关为 `hazard_fea_12_pulse`，用于验证有限地图、PCG 摆放、手工摆点、对象池和 `Combat.apply_damage()` 伤害链路。

## 6. 道具系统

### 6.1 主动道具（可充能，Active Items）
- 占用一个主动栏位，通过**充能**后手动释放（如击杀/拾取/时间充能）。
- 示例：全屏清屏、短时无敌冲刺、范围爆炸、临时召唤物。
- 关键字段：`charge_type`（充能方式）、`charge_max`（所需充能）、`effect`（效果 id）。

### 6.1-A 主动技能（Skills）
- 主动技能系统吸收 Unreal GAS 的核心思想，但实现为适合本项目的 Godot 轻量 ability system：ability 定义、activation tag gating、cost / cooldown、targeting primitive 和 effect primitive 分离；当前不吸收网络预测 / replication / Blueprint 类型系统。
- 技能是可复用运行时能力定义，不绑定英雄本体；角色、主动道具、局外奖励或未来模式规则都应通过 `skill_id` 引用技能，而不是复制逻辑。
- 技能数据集中在 `client/data/skills.json`，核心字段为 `ability_tags`、`activation`、`costs`、`targeting`、`effects` 和 `cooldown`；玩家可见名称 / 描述走 `skill_*` 本地化 key。
- 释放成本使用可扩展资源池：当前默认角色使用 `mana`，后续英雄可声明怒气、能量、弹药等其他 `skill_resources`，技能只引用资源 id，不在代码写死“魔法”。
- 释放条件使用 ability tag：`required_tags` 全满足才可释放，`blocked_tags` 命中任一 owned tag 时阻断释放，`granted_tags` 在激活 / commit 期间临时授予。当前首片用 `ability_tag_silenced` 表达沉默阻断，用 `ability_tag_activating` 表达即时激活期。
- 目标与效果必须 primitive 化：AOE、指向敌人、指向队友等目标类型登记到词表；伤害、治疗、状态、召唤、武器临时修正等效果登记为 effect primitive。当前默认内容不再使用旋风斩、点燃斩或燃烧法术；首片 `skill_overdrive_rounds` 使用 `target_ally` + `skill_effect_weapon_modifiers`，向玩家主武器临时施加射速与弹速修正。状态原语仍可用于系统能力（例如沉默阻断），但首批玩家技能应优先包装为射击强化 / 武器模组 / 战术开火能力。
- 主动道具未来可以“释放一个技能”作为效果之一；主动道具负责充能 / 栏位 / 触发来源，技能系统负责目标筛选、资源消耗、冷却和效果解释。

### 6.2 一次性消耗品（Consumables）
- 拾取即用或可携带后使用，用完消失。
- 示例：药水（回血）、临时增益（限时加攻速）、炸弹（手动放置炸开机关/敌群）。

### 6.3 被动遗物（Passive Relics）
- 拾取后**永久生效**（本局内），可叠加，是 build 构筑的核心。
- 效果类型：
  - **数值修正**：+伤害、+射速、+移速等（加法/乘法）。
  - **行为改写**：子弹穿透、追踪、分裂、反弹、变更弹道（环形/扇形）。
  - **触发型**：命中时几率点燃/冰冻/中毒；受伤时反弹等。
- **协同（Synergy）**：遗物、技能、装备词条和状态效果组合产生质变效果（如「分裂」+「子弹数量」=弹幕海），是刷宝构筑乐趣核心。

### 6.4 修正叠加模型（建议）
最终属性 = （基础值 + Σ加法修正） × Π乘法修正。
所有遗物通过往一个「修正器列表」注册修正项来生效，便于动态增删与 AI 生成新遗物。

### 6.5 破限道具 / 遗物扩展规则
- 允许未来内容突破默认限制，例如额外主动栏、改变开火模式、反转伤害规则、临时改变摄像机、重写掉落逻辑等。
- 破限内容必须走声明式数据：`capabilities` / `modifiers` / `behaviors` / `tags` / `params`。若现有字段表达不了，先抽象成可复用 effect、status、capability 或 strategy，再让数据引用。
- 禁止为单个遗物 / 道具 / 角色写一次性硬编码分支；确实唯一的行为也要包成命名 primitive，并在词表、测试、文档中登记。
- 工程红线不可被内容突破：随机仍走 `RNG`，时间仍走 `GameClock`，伤害仍走 `Combat`，UI 仍走 `UIManager`，存档仍走 `SaveManager`。

### 6.6 多模式资源复用规则
- 后续可能存在标准生存、挑战、剧情、每日种子、Boss Rush 等多种模式；角色、遗物、主动道具、消耗品、敌人、机关、成长奖励等资源定义默认应保持**模式无关**，只描述自身能力、标签、稀有度、解锁条件和兼容约束。
- 模式配置（`client/data/game_modes.json`）负责声明本模式使用哪些资源池、权重、禁用列表、解锁 / 难度条件、初始规则和少量覆盖参数；禁止在资源本体中写死“只属于某模式”的逻辑，除非该资源确实是模式专属且仍通过 tags / availability 数据声明。
- 代码层不得出现 `if mode_id == ...` 决定某个角色 / 遗物 / 敌人特殊行为；模式差异应落在可复用 strategy、capability、effect primitive、spawn / growth pool 配置或 `GameState` 的模式上下文中。
- 存档、回放、埋点和结算需要记录 `mode_id`，但资源系统应按同一套 `DataLoader` / 词表 / 校验管线读取，避免每个模式维护一套平行数据。

### 6.7 未来多人 PvE / PvP 预留边界
- 当前正式项目仍以单人离线为目标，不实现联机、房间、服务器权威、同步协议、延迟补偿或反作弊；多人只作为架构边界预留，避免早期代码写死单玩家假设。
- 玩法实体默认应能归属到 `actor` / `participant` / `team` / `faction` 这类中立概念；本地玩家只是一个 participant，敌人、召唤物、队友和未来远端玩家都应能走同一套阵营 / 目标筛选 / 伤害入口。
- 输入层产出可记录、可回放的 intent / action 序列；本地键鼠 / 手柄、AIPlayer、未来远端玩家输入都应适配到同一输入语义，业务实体不直接读取物理输入。
- `Combat.apply_damage` 未来必须能依据 source / target / team / friendly_fire / mode rule 判断合法伤害；现在也禁止写死“玩家只能打敌人、敌人只能打玩家”的二元关系。
- `game_modes.json` 已预留 `participants`、`teams` 与 `friendly_fire` 字段；后续可扩展 `player_count`、`pve_enabled`、`pvp_enabled`、`participant_rules` 等字段。这些字段只用于模式规则和校验，不代表当前要实现联网。
- 回放、存档、埋点和结算格式预留 `participant_id` / `team_id` / `mode_id` 概念；单人模式可以固定为 `local_player` / `team_player`，未来扩展多人时不推翻文件结构。

## 7. 成长与难度

### 7.1 经验与升级选择
- **经验来源**：击杀敌人掉落经验球，经验球走对象池（词表 §8 `pickup_orb`），进入 `pickup_range` 后自动吸附；经验数值、等级阈值和掉落规则都外置到 `client/data/`。
- **升级触发**：经验达到阈值后进入 `GameState.LEVEL_UP`，玩法时间冻结，`UIManager` 推入升级选择面板；选择完成后即时应用奖励并回到 `PLAYING`。
- **默认候选数**：参考《吸血鬼幸存者》，每次升级默认提供 **3 选 1**。
- **幸运扩展候选数**：生成候选前使用 `RNG.ui_choice` 做一次由 `luck` 影响的判定；判定成功时本次升级提供 **4 选 1**。具体概率曲线（如每点 `luck` 增加多少 4 选 1 概率、是否有上限）必须数据化，暂不在代码中硬编码。
- **确定性要求**：候选数量判定、候选池抽取、稀有度权重都走 `RNG.ui_choice`；录制回放记录等级、候选数量、候选 id 列表、玩家选择与当时 `luck` 快照。
- **选项内容待定**：升级选项最终给什么（遗物、属性升级、主动道具强化、回血、刷新 / 跳过等）尚未定案，先作为待决策项保留；实现前不得把某类奖励写死进升级系统。F4 当前仅落地数据驱动的 `stat_modifier` 最小切片，用于验证“击杀 → 掉经验 → 升级选择 → 即时变强”闭环，不代表最终升级内容上限。

### 7.2 局外成长（Meta Progression）
完整项目需要较深的局外成长，但它必须服务刷宝动作游戏的长期追求，而不是替代本局操作和构筑。第一局应可玩；局外成长提供长期目标、内容解锁、装备 / 词条追求和轻度永久强化。

- **数据入口**：局外成长配置集中在 `client/data/meta_progression.json`，包含 `currencies`、`run_rewards`、`account_level`、`upgrade_tracks` 与 `unlocks`。所有玩家可见名称 / 描述只存 `name_key` / `desc_key`，译文在 `client/locale/strings.csv`。
- **局外货币**：默认货币为 `meta_essence`，通过结算获得。奖励公式按存活时间、击杀数、Boss / 精英首杀等数据字段计算，并设置单局上限，避免刷低难度无限膨胀。
- **账号等级**：结算同时给账号经验；阈值数据化，等级奖励只授予解锁项或开启新升级轨道，不直接写死在代码里。
- **永久升级轨道**：基础轨道包括生命、伤害、移速、拾取范围、幸运和开局选择。每条轨道声明 `max_level`、逐级 `costs`、解锁条件和修正器；运行时通过 `ModifierEngine` 注入最终属性，禁止直接改 `player.json` 基础值。
- **内容解锁**：解锁类型包括 `character`、`relic_pool`、`mode`、`difficulty_tier`、`run_start_option`。解锁条件可来自账号等级、挑战、成就或升级轨道等级；新类型必须先登记词表，再让数据引用。
- **深度结构**：后续可把 `upgrade_tracks` 扩展成多层成长树：节点前置、分支互斥、重置 / 退款规则、赛季或挑战页签都应是数据字段，而不是 UI 或业务代码分支。
- **存档边界**：购买等级、货币余额、账号经验、已解锁 id、累计统计写入 `SaveManager` 的 `meta` kind；玩家偏好仍走 `Settings`，二者不混用。
- **确定性边界**：一局内随机不受局外随机影响；需要跨局随机奖励时使用 `RNG.meta`。回放记录开局时的 meta 快照 / 数据指纹，确保同一局可重现。
- **平衡原则**：永久升级数值要小而可感知，优先提供横向内容解锁与开局选择深度；避免必须长时间刷数值才能进入核心乐趣。

### 7.3 其他成长与难度
- **金币/商店**：可选，掉落金币用于在地图刷新点购买道具；金币是局内资源，除非另有 ADR，否则不等同于局外货币。
- **难度曲线（数据化）**：随存活时间推进，刷怪强度、密度、机关复杂度递增。**难度曲线、刷怪规则、掉落概率全部外置为数据**（`spawn_waves.json` 等），留出快速调参空间，平衡通过实际体验迭代调整，避免硬编码。

## 8. 美术与音效（极简、AI 友好）

### 8.1 美术风格
- 推荐**几何形状 / 简约像素 / 2.5D 低模混合**风格：玩家首片可用低模 3D 胶囊角色表现，敌人、子弹、拾取物和机关仍可先用几何占位；当前 IP 美术基准见 `docs/IP美术风格.md`。
- 优点：AI 易生成、文件小、辨识度高、性能好；后期可逐步替换为更精细素材。
- 统一规范：固定调色板、固定贴图尺寸（如 32×32），便于批量生成与替换。
- 当前首个敌巢视觉方向是“被异宇宙规则钙化出来的活体建筑”：敌巢 / 虫族使用骨白、蜡黄、干肉粉、深红、黑紫与少量毒蓝；青、红、白归属虫族 / 敌巢，玩家和玩家子弹默认避开青、红、白，敌方远程攻击可使用红色，宝箱与地图兴趣点按功能色区分。

### 8.2 美术规范先行（工作流）
- **流程**：先制定**统一调色板 + 固定贴图尺寸 + 资源命名规范**，再让 AI 批量生成素材，保证风格统一、可随时替换。
- **占位图先行**：开发初期优先用几何占位图或低模 3D 占位，把玩法跑通后再替换为正式美术，避免美术返工阻塞玩法验证。
- **2.5D 边界**：玩家可以显示为 3D 模型 / 胶囊 / 低模角色，但玩法实体仍按 2D 平面移动和碰撞；后续扩展敌人或场景 3D 表现时也应先保留 2D 玩法判定。

### 8.2-A 固定斜俯视资产落地规则
固定斜俯视不等于“所有资产都做成菱形”。菱形只是一种**地面平面语言**，用于让玩家读懂地面范围和危险区域；其他资产靠落地点、底座、阴影、遮挡和排序统一到同一个斜俯视舞台。

- **地面范围类资产**：地砖、机关危险区、AOE 预警、房间边界、裂缝、地面符号等贴在地面上的范围，应优先使用菱形或与地图菱形格对齐的外轮廓；机关、地面危险区和规则型 tile footprint 的尺寸应写成格子的整数倍。如果玩法判定不是菱形，视觉必须明确展示真实判定边界，避免“看着安全但实际受伤”。
- **实体 / 障碍类资产**：角色、敌人、宝箱、石柱、墙、树、火盆、机器等轮廓可以自由，不强制菱形；但必须定义一个清晰的脚底 / 落地点，并用底座、接触阴影或投影把它钉在地面上。立体物上半部分通常向屏幕上方延展，并通过 Y-sort / 遮挡层级决定谁挡住谁。
- **拾取物 / 子弹 / 特效**：经验球、金币、掉落物、子弹、命中特效可以是圆形、三角形、线条或粒子；需要贴地读位置时加小阴影或落点标记，需要表达地面影响范围时另加地面菱形 / 椭圆 / 清晰边界。
- **AI 生成资产 brief**：每个非 UI 资产至少说明 `asset_type`（ground_range / character / enemy / obstacle / pickup / projectile / vfx）、`footprint_shape`（diamond / ellipse / circle / free / point）、`anchor_point`（脚底或中心落点）、`shadow`（是否有地面阴影 / 底座）、`sort_layer`（地面 / 实体 / 空中 / UI）和 `collision_or_trigger_shape`（真实玩法判定形状）。缺这些字段时先补 brief，不直接生成最终资源。
- **Hades 式参考边界**：可用 3D 建模 / 动画辅助生成 2D 帧，也可在 Godot 中用低模 3D 视觉层，但最终进入玩法时仍以 2D 平面坐标、落地点、阴影和遮挡层级统一，不把地图 / 碰撞 / 回放改成真 3D。

### 8.3 音效
- 简短电子音效（射击、命中、拾取、升级、受伤），可由 AI/合成器批量生成。

## 9. 技术架构（AI 友好 + 数据驱动）

### 9.1 引擎/语言
- **当前引擎基线为 Godot 4.7 + GDScript**（ADR #85）：开源、轻量、场景/脚本均为文本化资源，AI 易读写与生成，自带 2D 节点（`Node2D` / `Camera2D` / `TileMap` / `Area2D` 等）非常契合本项目需求。
- 项目目录约定：`res://scenes/`（场景）、`res://scripts/`（脚本）、`res://data/`（数值配置）、`res://locale/`（翻译表）、`res://assets/`（美术/音效）；玩家设置存于 `user://settings.cfg`。

### 9.2 架构原则（让代码易于 AI 分析/生成）
- **数据与逻辑分离**：数值放配置文件，逻辑读取配置（见 9.3）。
- **模块清晰、单一职责**：Player / Bullet / Enemy / Item / Hazard / Spawner / Map 各自独立场景与脚本。
- **组件化或实体系统**：用「修正器/效果」组合实现遗物，而非为每个遗物写硬编码分支。
- **未来多人友好但单机优先**：当前不实现网络层；业务模块通过 actor / participant / team / intent 等边界避免强依赖唯一 `Player`，为未来多人 PvE / PvP 保留重构空间。
- **善用 Godot 机制**：信号（`signal`）解耦事件、`Resource`（`.tres`）做强类型配置、对象池复用 `Bullet` 节点。
- **命名规范统一**、函数短小、注释明确，便于 AI 上下文理解。

### 9.3 数值配置文件（核心需求 #10）
所有需人工调整的数值集中放在独立文件，放于 `res://data/`，由 `DataLoader` 统一读取与校验。格式原则为：**平表数值优先 CSV，复杂配置优先 JSON**。

- **CSV 优先场景**：敌人基础数值、经验曲线、刷怪波次、掉落权重、价格、简单等级表等“一行一个条目、列结构稳定、人工需要频繁扫表 / 排序 / 批量改”的配置。
- **JSON 优先场景**：遗物 `modifiers` / `behaviors`、角色能力、局外成长树、嵌套参数、可变长度数组、多层 unlock / condition 等结构化配置。
- **Godot 读取方式**：CSV 使用 `FileAccess.get_csv_line()`，JSON 使用 `FileAccess.open()` + `JSON.parse_string()`；二者都必须经过 `DataLoader` 转成统一的字典 / 数组结构后再交给业务系统。

- **配置热重载**：支持运行时重新读取 `/data` 配置即时生效（如监听文件变化或提供调试热键重载），调参无需重启，效率极高。
- **字段说明文档**：`client/data/README.md` 是完整项目数值配置手册；每个数据文件、字段、单位、取值范围、默认值、常见调参入口都必须写清，方便人和 AI 正确填写。
- **文案分离**：数据文件不得存玩家可见硬文本；名称、描述、提示统一存 `name_key` / `desc_key` / `hint_key`，译文在 `client/locale/strings.csv`。
- **命名统一**：数据字段与词表 id 使用 `snake_case`，避免 `maxHp` / `fireRate` 这类代码风格字段混入配置。

建议结构：
```
/data
  README.md         # 数值配置手册，字段含义、单位、范围、调参流程
  player.json       # 玩家基础属性（现有入口；若重构为纯平表可迁移为 CSV）
  game_modes.json   # 游戏模式配置：可用资源池、权重、模式规则与轻量覆盖（复杂配置优先 JSON）
```

### 9.4 多语言本地化（框架级需求）
本地化必须从框架设计阶段就纳入，而非后期补丁，否则大量硬编码文本难以返工。

**核心原则：代码与配置中不出现面向玩家的硬文本，一律用「文本键（key）」引用。**

- **统一文本键**：所有 UI、道具名、描述、提示等使用 key（如 `relic_sharp_rounds_name`），运行时按当前语言查表得到译文。
- **Godot 实现**：优先使用 Godot 内建本地化系统。
  - 翻译表用 `CSV`（`res://locale/strings.csv`）导入为 `.translation` 资源，调用 `tr("key")` 取译文。
  - 在 `Project Settings → Localization` 注册各语言；运行时用 `TranslationServer.set_locale("en")` 切换。
  - UI 控件文本绑定 key，切换语言时通过 `notification(NOTIFICATION_TRANSLATION_CHANGED)` 或重载界面刷新。
- **首批语言范围**：正式项目先维护简体中文 `zh_CN` 与英文 `en` 两种语言；`general.locale` 默认 `zh_CN`，可切换到 `en`。
- **AI 自动翻译**：新增玩家可见文案时，AI 必须自动补齐另一语言的首版译文；人工校对是最终权威，发现语气或含义不准时只改译文列，不改 key。
- **英文长度基准**：UI 布局、按钮宽度、面板宽度、换行和 HUD 信息密度以英文 `en` 文案长度作为最小设计与验收基准；中文 `zh_CN` 信息密度高，不能作为唯一尺寸依据。新增 / 修改 UI 文案或布局时必须切到英文检查文本不截断、不溢出、不遮挡。
- **数据文件与本地化解耦**：`relics.json` 等只存 `name_key` / `desc_key`（文本键），具体译文放翻译表，新增语言不必改数据文件。
- **致谢数据**：游戏内 Credits UI 的数据源为 `client/data/credits.json`；分组标题与角色 / 用途标签走 `ui_credits_*` 本地化 key，人名、项目名、许可证名、URL 与版权声明保持原文；根目录 `CREDITS.md` 是代码库级同步清单。
- **格式化与占位符**：动态数值用占位符（如 `"伤害 +{value}"` / `"Damage +{value}"`），避免拼接句子；注意数字、单位顺序在不同语言的差异。
- **文案配置手册**：`client/locale/README.md` 是完整项目多语言文案配置手册；新增 key、语言列、占位符约定或翻译流程变化时必须同步该文档。
- **字体**：选用覆盖目标语言字符集的字体（至少中、英；如需 CJK 用支持中日韩的字体），避免缺字方块。
- **扩展语言**：繁中、日、韩等语言后续另行决策；新增前必须同步 `client/locale/README.md`、设置项和 Godot Localization 注册。

翻译表 `strings.csv` 示例：
```csv
keys,zh_CN,en
relic_sharp_rounds_name,锋利弹头,Sharp Rounds
relic_sharp_rounds_desc,伤害 +{value},Damage +{value}
ui_settings,设置,Settings
ui_resume,继续,Resume
```

数据文件改为引用文本键（与 9.3 配合）：
```json
{
  "id": "relic_sharp_rounds",
  "name_key": "relic_sharp_rounds_name",
  "desc_key": "relic_sharp_rounds_desc",
  "modifiers": [ { "stat": "damage", "type": "add", "value": 1.5 } ]
}
```

### 9.5 设置系统（框架级需求）
设置系统同样需在框架阶段搭好，提供统一的「读取 / 修改 / 持久化 / 即时生效」通路，方便后续不断扩充设置项。

- **统一设置管理器（autoload 单例 `Settings`）**：集中管理所有设置项的默认值、当前值、读写与广播。
- **持久化**：存档到 `user://settings.cfg`（Godot `ConfigFile`）或 JSON；启动时加载，修改时保存。
- **即时生效**：设置变更通过信号（如 `setting_changed(key, value)`）广播，相关系统订阅后实时响应，无需重启。
- **可扩展**：每个设置项以「键 + 类型 + 默认值 + 取值范围」描述，新增设置=加一条配置 + 一个 UI 控件，逻辑统一走管理器。
- **设置界面本地化**：设置项标签全部使用文本键（与 9.4 联动），其中包含「语言选择」项。

建议覆盖的设置分类：
| 分类 | 示例设置项 |
|------|-----------|
| 语言 | 界面语言切换（联动 9.4） |
| 画面 | 全屏/窗口、分辨率、垂直同步、画面缩放、特效/伤害数字开关 |
| 音频 | 主音量、音乐音量、音效音量（分总线 Bus 控制） |
| 操作 | 键盘 / 手柄重绑定（移动/瞄准兜底/主动道具/详细数值面板/暂停）、是否「松开瞄准输入停火」、瞄准模式（鼠标/4 方向兜底/自动瞄准） |
| 辅助功能 | 屏幕震动开关、高对比/色盲友好配色、字体大小 |
| 游戏 | 难度倾向（若开放）、伤害数字显示、教程提示开关 |

`settings.cfg` 示例：

### 9.6 数据埋点 / 玩家数据收集（框架级需求）
项目需收集玩家数据用于后续分析与平衡迭代，**必须从框架阶段就预留统一的数据收集接口**，避免后期在业务代码里临时硬塞导致难以维护。

- **统一埋点管理器（autoload 单例 `Analytics`）**：对外暴露简洁接口，如 `track_event(event_name, params)`，由它统一缓冲、批量上报或落盘。
- **埋点与业务解耦**：各系统通过 `signal` 或调用 `Analytics.track_event()` 上报，**不把上报细节散落进业务逻辑**；事件名与字段集中定义为常量/配置，避免裸字符串。
- **上报实现可插拔**：通过接口隔离具体上报方式（本地落盘 JSON / HTTP 上报后端等），切换后端不改业务代码；即使后端暂未就绪，也先把接口与事件留好（先本地落盘）。
- **隐私合规**：在 `Settings` 中提供「数据收集开关」，不收集敏感个人信息；遵守目标平台与地区的隐私要求。

建议预留的关键埋点事件：
| 事件 | 触发时机 | 关键字段（示例） |
|------|---------|-----------------|
| `run_start` | 开局 | 模式、participant、team、角色、起始遗物、版本、locale |
| `run_end` | 一局结束 | 存活时长、击杀数、原因（死亡/退出）、最终等级 |
| `player_death` | 玩家死亡 | 位置、时间、击杀数、当前波次、死于何种敌人/机关 |
| `level_up` | 升级 | 等级、候选数量（3/4）、提供选项、玩家选择、`luck` 快照 |
| `relic_pick` | 获得遗物 | 遗物 id、当前已有遗物组合 |
| `item_use` | 使用主动道具/消耗品 | 道具 id、剩余充能 |
| `wave_milestone` | 难度/波次节点 | 波次、敌人密度、当前 DPS/承伤 |
| `meta_currency_earned` | 结算获得局外货币 | currency_id、amount、来源明细 |
| `meta_upgrade_purchased` | 购买局外升级节点 | upgrade_id、level、cost、余额 |
| `meta_unlock_granted` | 授予局外解锁 | unlock_id、kind、来源 |
| `account_level_up` | 账号等级提升 | old_level、new_level、奖励 unlock_ids |
| `save_written` | 存档写入成功 | slot、kind、version、路径 |
| `save_loaded` | 存档读取成功 | slot、kind、version、迁移链 |
| `save_deleted` | 存档删除 | slot、kind、原因 |
| `save_migrated` | 存档自动迁移 | slot、kind、from_version、to_version、备份路径 |
| `save_corrupted` | 存档损坏 | slot、kind、错误信息 |
| `run_suspended` | 保存当前局并退出 | slot、run_time、seed、玩家等级 |
| `run_resumed` | 从 run 存档续局 | slot、run_time、seed、恢复状态 |

`track_event` 调用示例：
```gdscript

    "pos": player.global_position,
    "time": run_time,
```

> 设计要点：埋点是框架第三条横向基础设施（与本地化、设置并列）——**所有数据上报走 `Analytics` 统一接口，关键节点从第一行代码起就留好钩子**。

### 9.7 AI 友好工程实践（让 agent 易读、易写、易扩展）
本项目目标之一是让 AI agent 能高质量续写/生成代码，为此除"数据驱动 + 单一职责 + 信号解耦"外，还约定以下工程实践（强制规则见当前平台编码规则入口第 14~18 条）：

- **项目索引（`AI导航.md`）**：项目地图，列出"加某类内容该改哪些文件、扩展点在哪"。agent 开工前先读它定位，避免盲目全仓搜索。
- **词表与契约（`词表与契约.md`）**：集中维护所有"约定字符串"白名单（`stat` / `effect` / `behavior.event` / 埋点 `event_name` / 设置 `key` / 本地化 key 前缀）。代码以**常量/枚举**引用，**禁止裸字符串与编造 id**；新增 id 先登记再实现再使用。这是数据驱动可靠性的关键——防止 AI 写错字段名。
- **数据校验 + fail-fast**：`DataLoader` 加载时校验字段/类型/取值范围，出错打印**具体文件名+字段+期望值**，让 AI 能据报错自我纠错。
- **黄金样例**：每类数据文件保留一条带注释的范例条目，AI 照结构填写最可靠。
- **类型化 GDScript**：变量/参数/返回值都标类型，配合静态检查，帮助 AI 推断用法、减少错误。
- **脚手架模板（`res://templates/`）**：新增同类内容复制模板改名，结构统一、便于模仿。
- **决策记录（`决策记录.md`）**：轻量 ADR，记"决策+理由"，避免 agent 误改既定约束。
- **日志规范 / README**：日志带模块前缀；`README.md` 写清运行调试方式，便于 agent 自行验证。
- **小文件**：脚本保持小而聚焦，便于 AI 整体读入上下文。

> 配套文档：`AI导航.md`（索引）、`词表与契约.md`（白名单）、`决策记录.md`（ADR）。

### 9.8 暂停与游戏状态管理
游戏需提供**暂停功能**：玩家可随时暂停以打开暂停菜单（继续 / 设置 / 退出），暂停期间整局游戏世界冻结。

- **统一暂停机制**：用 Godot 内建 `get_tree().paused = true/false` 暂停整棵场景树。暂停时移动、按住开火、子弹飞行、刷怪、机关、计时等业务节点全部停止（默认 `process_mode = PROCESS_MODE_PAUSABLE`）。
- **暂停菜单仍可交互**：暂停菜单及其下需要响应的 UI 节点设 `process_mode = PROCESS_MODE_ALWAYS`，保证暂停时仍能接收输入。
- **暂停键可重绑定**：通过 `Settings` 重绑定的 InputMap action `pause`（默认 `Esc` / 手柄 Start 或 Menu）触发，**不硬编码按键或手柄按钮**；同一 action 在暂停态再次触发=继续。
- **非暂停 HUD 叠层**：详细数值面板通过 `show_stats_panel` action（默认 Tab，可重绑定）按住显示、松开隐藏；只刷新 HUD，不调用 `UIManager.push()`、不切换 `GameState`、不设置 `get_tree().paused`。
- **菜单内容**：继续（`ui_resume`）、设置（`ui_settings`，复用 9.5 设置界面）、退出到主菜单（`ui_quit`）。所有文本走本地化键（9.4），**不硬编码**。
- **联动设置**：打开设置菜单时自动暂停；可提供「窗口失焦自动暂停」设置项（`gameplay.pause_on_focus_loss`）。
- **解耦实现**：由独立的 `PauseMenu`/状态管理节点统一控制暂停，通过 `signal`（如 `game_paused(bool)`）广播，其它系统按需订阅，不在各业务脚本里散落暂停判断。
- **埋点（可选）**：可在暂停/继续时上报 `game_pause` / `game_resume`（见 9.6 与词表），用于分析玩家中断行为。

> 设计要点：暂停是**游戏状态层**的统一能力——靠 `get_tree().paused` + `process_mode` 实现，按键走设置、文本走本地化，避免散落硬编码。

### 9.9 录制回放系统（框架级，第四条横向基础设施）
射击刷宝生存的平衡迭代极依赖**回归测试**——任何一次改数值、掉落权重、射击强化或效果原语都可能破坏既有 build；而单元测试只能验"片段"，回放系统验"整局"。本系统与 9.4~9.6（本地化 / 设置 / 埋点）并列，从框架阶段就建。

- **统一录制器（autoload 单例 `Replay`）**：开局即开始记录 `seed + tick 序列 + 输入事件`；一局结束后存到 `user://replays/<timestamp>.replay`。
- **确定性前提**：所有随机走 `RNG`（见修改建议 J）；时间走 `GameClock`（见修改建议 P）；不读 `Time.get_ticks_msec()` 等非确定源。**这是回放可重现的硬约束**。
- **录制内容**：`seed` / `version`（设计文档版本 + 数据指纹哈希）/ 玩家输入（按 InputMap action 帧时间戳）/ 关键决策（升级候选数量、候选 id 列表与玩家选择）。
- **不录制**：玩家位置/敌人状态等可重算量（节省体积，且确定性已保证可重算）。
- **重放接口**：`Replay.play(file)` 启动回放；可选倍速、可选"对照模式"（同一 seed 跑当前代码 vs 录制时的产物，自动 diff 帧状态）。
- **黄金回放**：维护一组**长期回放样本**作为回归基准（如 `tests/replays/golden_*.replay`），CI 触发时跑一遍——任意敌人血量/伤害/掉落概率改了，差异立即暴露。
- **可视化调试**：回放 UI 含时间轴、关键帧、伤害事件列表，让 AI/策划能直观读取一局。
- **埋点联动**：回放可作为 `run_end` 埋点的附件压缩上传，便于线上分析。

> 设计要点：录制回放是刷宝构筑和平衡迭代的**回归安全网**，与本地化/设置/埋点同等重要——必须在框架阶段把"确定性约束"立住，否则后期补无解。

### 9.10 平衡测试与 Headless 模拟（长期方向，预留接口）
独立游戏迭代到中后期，必须有**自动化平衡测试**才能稳定发版。本节定义未来落地形态，框架阶段先**预留接口**，避免后期硬塞。

- **AI Player 接口**：`AIPlayer` 实现与 `Player` 同形的输入接口（输出 InputMap action），可由简单策略（贪心走位 / 朝最近敌人）或 RL agent 驱动。
- **Headless 模拟入口**：`godot --headless -- script tools/sim.gd --runs 1000 --seed-base 0` 跑一千局，输出 CSV：存活时长、击杀数、死因、最终 build。
- **统计产物**：胜率分布、build 强度排名、"无人选择"遗物清单、过强组合 top-N。
- **接入点**（框架阶段就留）：
  - `Player` 节点的输入解耦——所有动作走 InputMap action，便于 `AIPlayer` 替换；
  - `Spawner` / `MapManager` / `RNG` 都接受外部 seed；
  - `Analytics` 在 headless 模式下可改用同步落盘。
- **不做的事（正式可玩闭环前）**：不要现在就实现 AI Player；只确保接口形态允许后续接入。
- **多人输入边界**：该接口也作为未来远端玩家输入的适配边界；业务系统只消费归一化 intent / InputMap action，不区分本地玩家、AIPlayer 或未来网络 participant。

> 设计要点：正式可玩闭环前**只锁接口形态**，避免做空中楼阁。一旦核心循环跑通，立即启动平衡测试是刷宝动作游戏商业化的关键。

### 9.11 AI 协作工程（框架级，第五条横向基础设施）
本项目从设计阶段就强调 AI 协作。除了现有的"项目索引 / 词表 / ADR / 记忆"，还需把**AI 协作本身**作为一条工程基础设施沉淀下来。

- **任务模板（`docs/AI协作/任务模板/`）**：把高频任务（"加一个遗物" / "加一种效果原语" / "调一类敌人数值" / "新增设置项" / "增加埋点" 等）标准化为 prompt 模板，明确「读哪些文件 → 改哪些文件 → 自检哪几条」，AI agent 直接套模板执行。
- **上下文预算（`docs/AI协作/上下文预算.md`）**：约定不同任务类型应当读取的文件序列与上限（例如"加遗物"只需读 `relics.json` + `词表与契约.md` + `enemy_template.gd`，不必全仓扫），降低 token 浪费、提升一致性。
- **AI 角色分工（`docs/AI协作/角色分工.md`）**：约定 `设计 / 实现 / 评审 / 平衡` 四种角色的职责与切换时机，长任务可让多个 agent 协作（设计先出 ADR 草稿 → 实现按 ADR 编码 → 评审按规则自检 → 平衡跑回放）。
- **引擎 MCP/Bridge 接入指南（`docs/AI协作/引擎集成.md`）**：当 Godot MCP 等服务可用时，AI 可直接查询场景树/读 Resource/触发引擎命令，从"读写文件"升级到"操作引擎"；本指南列出推荐工具与配置方式。
- **本地实时验证回路**：通过 pre-commit hook（参见 11.A）在**秒级**反馈契约 / 数据 / 类型违规，AI 改完即可知错即改，避免"push → CI → 等待"的分钟级回路。
- **依赖关系可视化**：`docs/AI导航.md` 第 5 节维护一份 Mermaid 系统依赖图，AI 改某模块前能"看见"上下游影响。

> 设计要点：把"AI 怎么协作"也当成一条与本地化/设置/埋点/回放并列的基础设施——它决定了项目跨人/跨 agent/跨时间的可维护性。

### 9.12 游戏状态机（GameState，统一流程管理）
游戏全局流程「主菜单 → 开局 → 进行中 → 暂停 → 升级选择 → 结算 → 死亡 → 回主菜单」必须由**单一状态机统一管理**，不得散落在各个场景脚本里。

- **统一状态管理器（autoload 单例 `GameState`）**：维护当前状态枚举与上下文数据；状态切换通过 `change_state(new_state, context)` 唯一入口。
- **核心状态**：`MAIN_MENU` / `LOADING` / `PLAYING` / `PAUSED` / `LEVEL_UP` / `GAME_OVER` / `RESULT` / `META_PROGRESSION`。
- **信号广播**：`state_changed(old, new)` / `state_entered(state)` / `state_exited(state)` —— 暂停联动、UI 显示、埋点上报、回放控制都订阅。
- **与现有系统的关系**：
  - `PauseMenu`（9.8）改为 `GameState` 的一个状态消费者：进入 `PAUSED` 时 `get_tree().paused = true`，退出时还原；不再自管。
  - `Replay`（9.9）订阅 `state_changed`，在 `PLAYING` 入口开始录制，`GAME_OVER`/`RESULT` 出口落盘。
  - `Analytics`（9.6）的 `run_start` / `run_end` 改为状态触发，业务代码不再自己埋。
- **禁止**：业务代码直接读写 `get_tree().paused` 或自管"是否在游戏中"的布尔变量；一律走 `GameState`。

> 设计要点：状态机是**全局协调中枢**，把暂停 / 回放 / 埋点 / UI 切换 / 难度计时全部挂在它的信号上，避免各系统各自判断。

### 9.13 通用对象池（PoolManager）
规则 10 已立"高频实体必须对象池"，但仅 `BulletPool` 不够——**敌人 / 掉落物 / 伤害飘字 / 命中特效 / 子弹**全部应统一池化。

- **统一池管理器（autoload `PoolManager`）**：按场景类型 id 分池：`PoolManager.acquire(pool_id)` / `PoolManager.release(node)`。
- **池类型 id 进白名单**：在 `docs/词表与契约.md` 第 8 节登记（如 `bullet_basic` / `enemy_chaser` / `damage_number` / `hit_spark`）；新池**先登记再使用**。
- **生命周期约定**：被池化节点必须实现 `_pool_reset() -> void`（重置状态）与 `_pool_release() -> void`（释放钩子，可选）。
- **预热**：开局或关卡切换时按数据预热池容量（防战斗中扩容卡顿）。
- **诊断**：debug build 暴露 `PoolManager.stats()` 返回各池水位 / 命中率，配合健康度 dashboard。

> 设计要点：把"对象池"从规则约束升级为**统一接口**——避免每个系统自己写池逻辑导致漂移。

### 9.14 UI 界面栈管理器（UIManager）
暂停 / 设置 / 升级选择 / 结算 / 商店等都是叠加弹窗，且大多需要联动 `GameState` 切换暂停。**没有界面栈则层级、开关、暂停联动会散落到处都是**。

- **统一 UI 管理器（autoload `UIManager`）**：维护 UI 栈，提供 `push(scene, ctx)` / `pop()` / `replace(scene)` / `clear()` 接口；自动处理：
  - 焦点（栈顶接收输入）
  - z 序（按入栈顺序）
  - 暂停联动（标记为"模态"的 UI 入栈时切 `GameState` 到 `PAUSED`，出栈还原）
  - `process_mode = PROCESS_MODE_ALWAYS`（暂停时仍响应）
- **场景定义其行为**：UI 场景根节点暴露 `@export var modal: bool` / `@export var pauses_game: bool` / `@export var music_duck: bool` 等元数据，由 `UIManager` 统一处理。
- **禁止**：业务代码直接 `add_child` UI 弹窗或自管显隐；一律走 `UIManager`。
- **本地化联动**：`UIManager` 在 `notification(NOTIFICATION_TRANSLATION_CHANGED)` 时刷新栈中所有节点。
- **文本尺寸验收**：UIManager 管理的按钮、面板、升级选择、结算和 HUD 布局默认按英文 `en` 文案长度设计；中文通过不代表英文通过。

> 设计要点：UI 栈是**用户交互的协调中枢**，把"该不该暂停"、"该不该收输入"、"该不该刷新文本"都从业务里收上来。

### 9.15 伤害结算管线与状态效果（DamageInfo / StatusEffect）
6.4 的 `modifiers` 模型解决了"玩家属性如何聚合"，但**一次伤害怎么算、点燃/中毒/冰冻这类持续效果如何叠加**目前糊在 `effect` 里。本节把它们从 `effect` 原语中拎出来，抽象为可复用的伤害管线与状态效果系统。

#### 9.15.1 统一伤害结算（DamageInfo）
- 所有伤害（玩家受到 / 敌人受到 / 机关 / DoT）走单一入口 `Combat.apply_damage(target, damage_info: DamageInfo)`。
- `DamageInfo` 字段（白名单见 `docs/词表与契约.md` 第 9 节）：
  - `amount: float` —— 基础数值
  - `damage_type: String` —— `physical` / `fire` / `poison` / `lightning` / `true` 等（白名单登记）
  - `source: Node` —— 伤害来源（用于击退方向、击杀归属、埋点）
  - `source_actor_id` / `source_team_id`（可空）—— 未来多人 / 召唤物 / 队伍归属预留；单人模式可由 source 推导
  - `flags: int` —— 位掩码（`is_crit` / `is_dot` / `pierce_armor` / `cannot_kill` 等）
  - `crit_chance` / `crit_mult`（可空，使用默认走源属性）
- **结算顺序**：阵营 / 友伤 / 模式规则校验 → 基础数值 → 暴击判定（用 `RNG.combat`）→ 类型抗性 → 防御 → 写血 → 触发 `on_hit` 链 → 触发 `StatusEffect` 注入。
- **禁止**：业务代码绕过 `Combat.apply_damage` 直接 `target.hp -= n`。

#### 9.15.2 状态效果（StatusEffect）
- 类型化资源（`StatusEffect.gd : Resource`），字段：
  - `id: StringName` —— 白名单（`burn` / `poison` / `freeze` / `slow` / `bleed` / `mark` 等）
  - `duration: float`（秒）
  - `tick_interval: float`（DoT 用）
  - `magnitude: float` —— 单 tick 伤害或减速比例等
  - `damage_type: String`（DoT 用）—— 例如 `burn` 使用 `fire`
  - `stack_rule: enum { REPLACE, REFRESH, ADD_DURATION, INDEPENDENT, MAX_MAGNITUDE }`
  - `granted_ability_tags: Array[StringName]` —— 状态存在期间授予的 ability tags，如沉默授予 `ability_tag_silenced`
  - `source: Node`
  - `source_team` / `target_team`（快照用）—— DoT 续局后仍保留击杀归因
- 挂载到目标的 `StatusEffectComponent`（每个可被状态影响的实体加一个）；该组件负责：tick / 过期清理 / ability tag 授予与释放 / 视觉表现（特效池 + 颜色叠加）/ 与 `ModifierEngine` 联动（如减速注入 `move_speed` 乘法修正）。当前已落地 Player / Enemy / SkillSystem 自身的 ability tag 生命周期、run 快照恢复和 DoT tick；旧 payload 缺少状态字段时按无状态兼容，视觉和 ModifierEngine 联动后续补。
- **效果原语 `ignite` / `chain` 等改为薄包装**：内部调用 `StatusEffectComponent.apply(StatusEffect.new("burn", { damage_type, magnitude, tick_interval, duration }))`，不再各自实现。
- **回放确定性**：所有 tick 走 `GameClock`，所有判定走 `RNG`（见 9.18）。

> 设计要点：伤害与状态效果是刷宝构筑表达力的核心。把"怎么算" / "怎么叠"统一到管线 + 资源对象上，新效果原语只是注入新 `StatusEffect`，不再各自管 tick 与叠加。

### 9.16 存档管理器（SaveManager）
玩家存档是完整项目核心能力，必须同时支持**局外成长长期保存**和**局内暂停退出后加载续局**。完整项目早期必须先落地 `SaveManager`，否则深局外成长和暂停退出都会缺少可靠持久化。**没有版本号 + 迁移逻辑 + 原子写入的存档格式一旦变更就是灾难**——本节把它框架化。

- **统一存档管理器（autoload `SaveManager`）**：对外 `save(slot, kind, payload)` / `load(slot, kind)` / `delete(slot, kind)` / `list_slots()` / `has_save(slot, kind)`；落到 `user://saves/<slot>/<kind>.save`。
- **槽位（slot）**：默认至少支持 `slot_0` 一个主槽；UI 可扩展多个槽位。槽位只表示玩家档案，不表示局内关卡。
- **存档种类（kind，白名单见 `docs/词表与契约.md` §14）**：
  - `meta`：局外成长长期档案，保存货币余额、账号等级、已购升级、已解锁内容、统计累计，设置除外。
  - `run`：局内暂停退出续局档案，保存当前一局的可恢复快照；这是正式需求，不是可选项。
  - `replay_index`：回放索引；具体回放仍走 `Replay` 文件。
- **格式**：JSON（人和 AI 易读）。**强制头字段**：
  - `version: int` —— 当前 kind 的 schema 版本号。
  - `created_at` / `updated_at` —— ISO 字符串或统一 wall time，仅用于诊断和 UI 排序，不参与玩法判定。
  - `game_version` —— `游戏设计文档.md` 版本号或构建版本。
  - `kind` / `slot` —— 与文件路径一致，加载时交叉校验。
  - `data_hash` —— payload 校验哈希，用于发现截断 / 半写入 / 人工破坏。
- **原子写入**：保存必须先写 `*.tmp`，flush 后再替换正式文件；替换前保留 `*.bak` 最近备份。禁止直接覆盖正式存档。
- **迁移**：`SaveManager.register_migration(kind, from, to, fn)` 链式注册；加载时若 `version < CURRENT[kind]` 自动跑迁移链；迁移失败时 fail-fast 并备份原文件到 `user://saves/.broken/`。
- **损坏恢复**：加载失败时按顺序尝试正式文件 → `.bak`；仍失败则把坏文件移动到 `.broken/`，UI 提示新建 / 重试 / 查看错误，不静默吞掉。
- **写入时机**：
  - `meta`：结算获得货币 / 账号经验、购买永久升级、授予解锁后立即写入。
  - `run`：玩家在暂停菜单选择“保存并退出”时写入；可在安全节点（进入关卡、升级选择结束、Boss 前）自动写 checkpoint，但不能每帧写。
- **局内续局快照（`run` payload）**：必须保存足以恢复当前一局的确定性状态，包括 `mode_id`、participants / teams 快照、主 seed、`RNG` 子流状态、`GameClock` tick / elapsed、`GameState`、玩家状态、当前武器 / 遗物 / 主动道具、经验 / 等级、有限地图状态、PCG / 人工机关摆放、Spawner 波次、敌人 / 子弹 / 机关 / 掉落物的可恢复快照、当前 UI 恢复点（如升级选择候选 id 和已暂停状态）。
- **退出与加载流程**：暂停菜单提供“保存并退出”；流程为 `GameState.PAUSED` → `SaveManager.save(slot, "run", snapshot)` → 回主菜单。主菜单若发现 `run` 存档，提供“继续游戏”；加载成功后恢复到 `PLAYING` 或保存时的暂停 / 升级选择状态。
- **通关 / 死亡清理**：一局正常结算、死亡或放弃后，`run` 存档必须删除或标记 consumed，防止重复结算刷局外奖励。`meta` 写入必须由结算流程单独完成，不能依赖 `run` 存档残留。
- **回放边界**：`run` 存档是玩家续局用，`Replay` 是回归 / 分析用；两者都记录 seed 和关键决策，但文件格式与用途分开。
- **隐私 / 反作弊**：默认不加密、不做反作弊；如需云存档、校验签名或防改档，后续单独 ADR。
- **与 `Settings` 的边界**：玩家偏好仍走 `Settings` 的 `user://settings.cfg`，本系统**只管游戏内进度**，避免职责重叠。
- **与局外成长配置的关系**：`meta_progression.json` 定义可购买 / 可解锁内容，`meta` 存档只保存玩家状态（余额、等级、购买级数、已解锁 id、统计），加载时由 `SaveManager` 迁移后交给 `MetaProgressionSystem` 校验。

> 设计要点：存档系统第一天就**必须有标准头字段（含 `data_hash`）+ 迁移 + 原子写入 + `.bak` 回退 + `.broken` 损坏隔离**。局外成长和暂停退出续局都依赖它；事后补会牵动几乎所有玩法系统。

### 9.17 音频管理器（AudioManager）
9.5 的 `Settings` 已有音量总线（`audio.master/music/sfx`），但缺少**统一播放接口**——业务代码若各自 `play()`，音效 id 散落、无法做"同 id 节流" / "Bus 路由" / "音频压缩"等。

- **统一音频管理器（autoload `AudioManager`）**：
  - `play_sfx(id, opts)`：按 id 取池中可用 `AudioStreamPlayer`（与 `PoolManager` 集成）；自动接 SFX bus；支持 `pitch_range` / `volume_db` / `max_polyphony`（同 id 同时最多 N 路，超出 ducking）。
  - `play_music(id, fade)`：交叉淡入淡出，绑 Music bus。
  - `stop_all()` / `pause_all()`（受 `GameState` 控制）。
- **音频 id 进白名单**：`docs/词表与契约.md` 第 10 节登记（如 `sfx_player_shoot` / `sfx_enemy_hit` / `music_run_loop`）；新音效**先登记再实现**。
- **Bus 路由**：默认 `Master → { Music, SFX, UI }`；`AudioManager` 启动时校验 Bus 配置，缺失则 fail-fast。
- **暂停联动**：暂停时音乐降低 -6dB、SFX 静音（除标记 `play_when_paused` 的 UI 音效）。
- **本地化（可选）**：未来若有配音，按 locale 切换音轨，键统一走 `AudioManager.play_voice(key)`，避免 if-else 判语言。

> 设计要点：音频也是**契约 + 池化 + 单一接口**，避免每个系统自己 `play()`。

### 9.18 确定性基础设施：RNG 与 GameClock（J / P 正式落地）
9.9 已要求"随机走 `RNG`、时间走 `GameClock`"作为录制回放的硬约束。本节是这两个 autoload 的正式设计章节（对应原修改建议 J 与 P）。

#### 9.18.1 RNG（种子化随机源）
- **autoload 单例 `RNG`**：维护多个分流子流，每个有独立 `RandomNumberGenerator`：
  - `RNG.spawn` —— 刷怪
  - `RNG.drop` —— 掉落
  - `RNG.combat` —— 暴击 / 命中 / 伤害浮动
  - `RNG.ui_choice` —— 升级候选数量判定与候选池抽取
  - `RNG.world` —— 地图 / 机关
  - 子流 id 进白名单（词表第 11 节）；新增需登记。
- **种子规则**：
  - 一局一个**主 seed**（`RNG.set_run_seed(seed)`）；各子流 seed 由带项目域名的 SHA-256 mixer 从主 seed + 子流 id 派生，禁止依赖 Godot `hash()` 或简单相邻 seed。
  - 修改 seed 派生或子流集合时必须跑跨子流相关性审计 `rng-audit`，避免一个随机结果泄漏或预测另一子流结果。
  - 支持「输入种子复现同一局」与「每日挑战」（共享主 seed）。
- **统一 API**：`RNG.<stream>.randi()` / `randf()` / `randf_range(a,b)` / `pick(arr)` / `weighted_pick(arr, weights, luck_bias)`；`luck_bias` 接 `Player.luck` 属性。
- **禁区**：业务代码**禁止**直接调用 `randi()` / `randf()` / `randi_range()` / `Time` 派生伪随机（规则 12-A 已立）；CI/hook 扫描裸调用 fail-fast。

#### 9.18.2 GameClock（统一时间源）
- **autoload 单例 `GameClock`**：
  - `now() -> float` —— 受 `GameState` 暂停冻结的"游戏时间"（秒）
  - `tick() -> int` —— 录制/重放用的整数 tick（与物理 tick 对齐）
  - `delta_scaled(delta) -> float` —— 业务在 `_process(delta)` 中应使用此值（受暂停 / 慢动作影响）
  - `wall_now() -> float` —— 真实时间（仅 UI 计时器、Analytics 时间戳用，不参与玩法）
- **暂停冻结**：`GameClock` 订阅 `GameState.state_changed`，进入 `PAUSED` 时停 `now/tick/delta_scaled` 自增；`wall_now` 不冻结。
- **慢动作 / 加速**：通过 `GameClock.set_time_scale(s)`（不是 `Engine.time_scale`，避免影响物理与 UI），业务系统乘 `delta_scaled`。
- **禁区**：业务代码**禁止**直接调用 `Time.get_ticks_msec()` / `Time.get_ticks_usec()` / `OS.get_unix_time()` 等参与玩法判定（规则 12-A 已立）；只允许在 `Analytics` / UI 计时器等非玩法路径使用。
- **测试**：`GameClock` 必须可被注入假时钟（用于单测与 headless 模拟）。

> 设计要点：RNG 与 GameClock 是**回放确定性的两根支柱**；它们的 API 与禁区一旦立死，整局可重现性、平衡测试、回归回放才有可能成立。

### 9.19 词表与代码常量的单一来源（R 落地：脚本生成）
规则 15 要求"约定字符串走白名单 + 代码用常量引用"。落地后代码端必有 `Stats.gd` / `Effects.gd` / `Events.gd` 等常量类——**md 与代码两边手写迟早漂移**。本节定 R 项的最终方案。

- **决策**：md 为**人类可读**入口；机器可读副本是 `client/data/_contracts.json`（由脚本从 md 生成）；GDScript 常量类同样由脚本生成。
- **流水线**：
  ```
  docs/词表与契约.md        ← 人手维护（唯一权威）
        │ tools/sync_contracts.py
        ▼
  client/data/_contracts.json   ← 机器可读，DataLoader 校验也读它
        │ tools/sync_contracts.py
        ▼
  client/scripts/contracts/
    stats.gd  effects.gd  events.gd  analytics_events.gd  settings_keys.gd
    actions.gd  pool_ids.gd  damage_types.gd
    status_effects.gd  audio_ids.gd  rng_streams.gd
    character_ids.gd  capabilities.gd  content_tags.gd
    meta_currencies.gd  meta_upgrades.gd  meta_unlocks.gd
    save_kinds.gd
  ```
- **生成的常量类约定**：
  - `class_name` + `const` 字段；命名为 SCREAMING_SNAKE_CASE
  - 文件头自动加注释「**自动生成，请勿手改；改 `docs/词表与契约.md` 后跑 `tools/sync_contracts.py`**」
  - `sync_contracts.py --check` 通过重新生成并对比文件内容诊断漂移
- **CI/hook 强制**：
  - `pre-commit`：md 改动 → 自动跑 `sync_contracts.py`，结果一并入 commit；若开发者手动改了生成文件 → fail
  - CI：跑一遍生成 → 与 commit 内容 diff，不一致 fail
- **DataLoader 校验**：直接读 `_contracts.json` 而不是解析 md（避免重复解析逻辑）
- **当前落地**：`tools/sync_contracts.py` 已生成 `_contracts.json` 与 `client/scripts/contracts/*.gd`，CI 通过 `--check` 防止漂移。

> 设计要点：单一来源 + 自动生成 = 漂移不可能。md 改完什么都不用做、代码常量自动同步——这是把规则 15 从"人靠自觉"升级为"工具兜底"的最终形态。

### 9.20 GM 指令与调试工具（debug/dev_tools 构建专用）
调试工具是框架级开发基础设施，用来提高调参、试玩复盘、问题复现和自动化诊断效率，但必须与正式导出隔离。

- **构建边界**：GM 指令、调试控制台、调试 overlay、刷怪 / 给道具 / 改 HP / 跳关 / 重载数据等工具只在 `debug` 或自定义 `dev_tools` feature 构建中启用；正式 release preset 不得启用 `dev_tools`，并应排除调试脚本 / 场景资源。
- **运行时双保险**：调试入口初始化前必须检查 `OS.is_debug_build()` 或 `OS.has_feature("dev_tools")`；release 构建即使保留了快捷键绑定，也不得创建调试 UI 或注册 GM 指令。
- **集中入口**：完整项目应收口到 `DebugConsole` / `DebugOverlay` / `GMCommandRegistry` / `DebugSpawner` / `DebugSaveTools` 等独立模块；历史 MVP 的轻量 `debug_tools.gd` 只作为已完成验证经验，不得把 GM 分支散落到正式玩法代码。
- **调用正式 API**：GM 指令必须通过正式系统 API 改状态，例如刷怪走 `Spawner`，改存档走 `SaveManager`，给道具走 `ItemSystem` / 数据 id，伤害走 `Combat.apply_damage`。禁止为调试方便直接绕过 autoload、数据契约或状态机。
- **InputMap action**：调试快捷键使用 `debug_*` action，并登记到 `docs/词表与契约.md`；正式设置 UI 不暴露这些 action。
- **导出 / CI 检查**：发版前检查 release preset 未启用 `dev_tools`，且导出资源不包含调试脚本、GM 命令表或开发-only overlay。后续 CI 可加扫描，发现 release preset 包含 `debug_tools` / `GMCommandRegistry` 等路径即 fail。
- **日志与埋点边界**：GM 指令可写本地开发日志，但不得伪造正式 analytics；需要记录调试行为时使用带 `debug_` 前缀的开发日志或本地 overlay，不进入玩家数据。

> 设计要点：调试能力越强，越要集中、可关闭、可排除。正式版没有入口、没有资源、没有可被玩家触发的 GM 命令。

### 9.21 本地 Mod 接口（创意工坊前置边界）

项目长期需要支持玩家制作 mod，并为未来 Steam 创意工坊 / 其他分发平台预留接口；当前阶段只落地**本地数据包式 mod**，不接创意工坊、不联网、不执行玩家脚本。

- **统一入口（autoload `ModLoader`）**：扫描 `user://mods/<mod_id>/mod.json`，读取 manifest、排序、诊断和启用状态；manifest `id` 必须等于目录名且重复 id 只启用第一个；业务系统不得自行读取 mod 目录。
- **分发与加载解耦**：未来创意工坊只负责把订阅内容同步到 `user://mods/<mod_id>/`；进入游戏后的加载、校验、禁用、排序和错误提示仍走 `ModLoader` + `DataLoader`。
- **当前支持的数据 patch**：`json_array_append` 用于给 `relics.json`、`weapons.json`、`game_modes.json` 等 JSON 顶层数组追加条目；`csv_append` 用于给 `enemies.csv`、`spawn_waves.csv`、`strings.csv` 等 CSV 追加行。当前不支持覆盖、删除或按 id 替换基础数据。
- **动态契约扩展**：玩家 mod 可以在 manifest 中声明少量运行时扩展 id（当前为 `character_ids`、`game_modes`、`content_tags`、`locale_prefixes`），且必须以 `mod_<mod_id>_` 开头；代码仍然只引用内置生成常量，mod id 只作为数据内容被校验和使用。
- **核心契约不可由 mod 擅自扩展**：`stats`、`effects`、`events`、`damage_types`、`pool_ids`、`audio_prefixes`、`rng_streams`、`save_kinds` 等需要代码、资源、确定性或存档同步的类别不得由玩家 manifest 打开。需要新原语时走正式项目开发流程。
- **安全边界**：mod manifest 只能引用自身目录内的安全相对路径，禁止 `..`、绝对路径和 `://`；当前不加载玩家 GDScript、动态库、可执行文件或远端资源。
- **校验边界**：`DataLoader` 对合并后的数据继续执行 schema / locale / 词表校验；无效 mod 应 fail-fast 输出 `[ModLoader]` / `[DataLoader]` 诊断，而不是静默吞掉。
- **回放 / 存档边界**：后续回放与 `run` 存档必须记录数据指纹，避免缺失 mod 或 mod 版本变化时静默恢复旧局；本切片不改变存档 schema。

> 设计要点：mod 是“玩家可分发的数据包”，不是绕过项目架构的脚本插件。先守住 manifest、数据 patch、安全路径和 DataLoader 校验边界，未来再把创意工坊作为分发层接上。

### 9.22 平台服务接口（Steam 优先，多平台预留）

项目后续需要接 Steam API，包括 Steam 成就、Steam 富状态 / 状态显示、overlay、好友邀请、Lobby / 联机入口等；也会接其他平台。当前阶段只落地**统一平台服务接口**，不接 Steamworks SDK、不联网、不创建真实大厅、不解锁真实成就。

- **统一入口（autoload `PlatformServices`）**：业务系统只调用 `PlatformServices`，不得直接调用 Steamworks / GodotSteam / Epic / GOG / 主机平台 SDK。后续平台差异由 provider adapter 吸收。
- **Steam 优先 provider**：当前 `preferred_provider()` 固定为 `steam`，启动时没有 SDK adapter 则 `active_provider()` 为 `none`、`is_available()` 为 false。这样先把业务调用面稳定下来，再接 Steamworks。
- **能力查询**：平台能力以 capability 暴露，当前预留 `achievements`、`stats`、`rich_presence`、`overlay`、`multiplayer`、`lobbies`、`user_identity`。调用前可用 `supports(capability)` 判断。
- **成就 / 统计边界**：`unlock_achievement()` 与 `store_stats()` 先进入统一门面。正式接 Steam 成就前，应建立成就 / 统计数据源与校验，避免业务层散落临时字符串。
- **状态显示边界**：`set_rich_presence()` 保存期望状态；当前没有真实后端时只本地记录。接 Steam 后再映射到 Steam Rich Presence / 状态显示字段。
- **联机边界**：`create_lobby()`、`join_lobby()`、`leave_lobby()`、`invite_friend()` 只代表平台大厅 / 邀请入口，不等于游戏同步协议。真正 PvE / PvP 仍需后续网络同步、服务器权威、断线恢复和反作弊设计。
- **overlay / 用户身份边界**：overlay、好友邀请、平台 user id / display name 都由 `PlatformServices` 提供；UI 不直接依赖具体平台 SDK。
- **离线与非 Steam 平台**：平台不可用时游戏必须仍能启动、游玩、保存和回放；平台 API 调用应安全返回 false 并给 diagnostics，不得崩溃。
- **云存档边界**：未来若接 Steam Cloud 或其他平台云存档，只能作为 `SaveManager` 的同步 / 分发层，不得绕过 `SaveManager` 的 envelope、迁移、原子写入和损坏隔离。

> 设计要点：Steam 是优先落地平台，但不是业务架构的唯一平台。先把 `PlatformServices` 作为“平台插座”固定下来，后续 Steamworks adapter 接在背后；其他平台接入时复用同一套业务调用面。

## 12. 风险与注意事项

| 风险 | 说明 | 缓解 |
|------|------|------|
| 性能 | 大地图 + 海量子弹/敌人易卡顿 | 对象池、空间分区、视野外卸载 |
| 平衡 | 协同过强或过弱 | 数据驱动 + 快速迭代 |
| 操作手感 | 鼠标瞄准提高命中直觉后，手柄 / 方向键兜底仍可能空打 | 瞄准辅助、提高子弹速度、后续补自动瞄准模式 |
| 内容量 | 射击刷宝生存需要大量装备、词条、遗物、武器模组技能和敌巢主题 | 数据驱动让 AI 量产候选内容，再由人工筛选和工具验证 |
| 本地化返工 | 后期补本地化成本极高，且中文短文本容易掩盖英文 UI 溢出 | 框架阶段即用文本键，杜绝硬编码文本；UI 尺寸按英文 `en` 长度验收 |
| 埋点缺失 | 后期补数据收集需大改业务代码 | 框架阶段即建 `Analytics` 统一接口，关键节点预留钩子 |

## 附录：术语表
- **Build（构筑）**：本局所获遗物/道具的组合形成的玩法流派。
- **Synergy（协同）**：多个遗物组合产生的强化/质变效果。
- **Modifier（修正器）**：对玩家属性的加法/乘法修改项。
- **Spawner（刷怪器）**：按规则在地图上生成敌人的系统。
- **Object Pool（对象池）**：复用实体以提升性能的技术。
