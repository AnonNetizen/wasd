# Roguelike 游戏设计文档（GDD）

> 版本：v1.3
> 类型：俯视角 Roguelike 弹幕生存
> 引擎：Godot 4.6.3（GDScript）
> 核心灵感：《以撒的结合》（道具/机关/自动射击） + 《吸血鬼幸存者》（开放大地图/生存）
> 配套文档：`AI导航.md`（项目索引）、`词表与契约.md`（字符串白名单）、`决策记录.md`（ADR）、`修改建议.md`（待决策项）；编码规则见 `.codebuddy/rules/game-coding-rules.md`

## 1. 游戏概述

### 1.1 一句话定位
一款俯视角 Roguelike 游戏：玩家在开放大地图中用 WASD 移动、方向键控制射击朝向，角色全自动开火，通过收集主动道具、消耗品与被动遗物不断变强，在机关与敌群中尽可能久地存活并推进。

### 1.2 设计支柱
- **操作简单、深度自由**：移动与瞄准分离，自动射击降低操作压力，把策略重心放在 build 构筑与走位上。
- **构筑驱动（Build-driven）**：被动遗物之间产生协同（synergy），形成滚雪球式成长。
- **美术极简**：几何/像素风格，便于 AI 批量生成素材，也便于程序化生成。
- **数据与逻辑分离**：所有可调数值集中在独立配置文件，策划/玩家可零代码调整。
- **框架级基础设施**：从设计阶段就内建**多语言本地化**与**可扩展的设置系统**，所有玩家可见文本走本地化键、所有玩家偏好走统一设置管理器。

### 1.3 目标平台与引擎
- 平台：PC（键盘操作优先）。
- 引擎：**Godot 4.6.3（GDScript）**，详见第 9 节。

## 2. 核心玩法循环（Core Loop）

```
出生 → 移动探索大地图 → 自动射击清怪 → 拾取掉落（经验/金币/道具）
   → 变强（升级/遗物/主动道具） → 遭遇更强敌群与机关 → 死亡/通关 → 结算 → 再次开局
```

- **短循环（秒级）**：移动走位 → 自动射击 → 躲机关/弹幕 → 拾取。
- **中循环（分钟级）**：升级选择 → 获取遗物 → 充能并释放主动道具。
- **长循环（一局/跨局）**：难度随时间递增 → 死亡结算 → 解锁新道具/角色（元进度）。

## 3. 玩家系统

### 3.1 移动
- WASD 控制 8 方向移动，移动速度由 `moveSpeed` 决定。
- 建议加入轻微加速度/惯性（可配置），手感更顺滑；初版可先做即时速度。

### 3.2 射击（自动开火）
- **朝向**：方向键（↑↓←→）控制射击方向，支持上下左右 **4 方向**；松开方向键时保持上一次朝向（可配置为停火）。
- **开火**：自动进行，间隔由 `fireRate`（射速）决定，玩家不可手动控制开火与否。
- **移动与瞄准解耦**：可以朝右移动同时向左射击（双摇杆式手感的键盘版本）。

### 3.3 核心属性（全部可在数值文件中调整）
| 属性 | 字段名 | 说明 |
|------|--------|------|
| 生命上限 | `maxHp` | 心数/血量上限 |
| 移动速度 | `moveSpeed` | 单位/秒 |
| 射速 | `fireRate` | 每秒射击次数（或射击间隔） |
| 伤害 | `damage` | 单发子弹伤害 |
| 子弹速度 | `bulletSpeed` | 子弹飞行速度 |
| 子弹射程 | `bulletRange` | 子弹存活距离/时间 |
| 子弹数量 | `bulletCount` | 一次开火发射的子弹数 |
| 暴击率/暴击伤害 | `critChance` / `critMult` | 可选扩展属性 |
| 拾取范围 | `pickupRange` | 自动吸附经验/金币的范围 |
| 幸运值 | `luck` | 影响掉落与稀有度（可选） |

> 属性采用「基础值 + 加法修正 + 乘法修正」的计算模型，便于遗物叠加（见 6.3）。

## 4. 子弹系统

- 子弹是独立实体，拥有位置、速度向量、伤害、存活时间/射程。
- **有飞行速度**，非瞬间命中：敌人移动时需要预判，玩家也可用走位躲避敌方子弹。
- 子弹与敌人/墙体/机关碰撞后结算（命中、穿透、反弹等行为可由遗物改写）。
- 性能要点：使用**对象池**复用子弹实体，避免频繁创建销毁导致卡顿（大地图弹幕量可能很大）。

## 5. 大地图与世界

### 5.1 地图设计
- 取消《以撒》的房间制，改为《吸血鬼幸存者》式的**开放大地图**。
- 建议方案：**无限/超大平铺地图**，以玩家为中心动态生成/卸载区块（Chunk）。
- 视角：俯视角。

### 5.2 摄像机（玩家始终居中）
- **核心规则**：与《吸血鬼幸存者》一致，**玩家角色始终固定显示在屏幕正中央**，移动时是世界（地图/敌人/子弹）相对玩家滚动，而非角色在屏幕内移动。
- Godot 实现：在 `Player` 节点下挂一个 `Camera2D` 并设为 `enabled`/当前相机，相机随玩家移动即可让玩家恒居屏幕中心。
  - **不要**开启相机的 `limit`（边界限制），否则地图边缘会破坏居中效果（开放/无限地图无需边界）。
  - **不要**使用 `drag margin`（拖拽边距），将其关闭以保证严格居中（拖拽边距会让玩家在中心附近偏移）。
  - 可选：开启 `position_smoothing` 做轻微平滑跟随；若要绝对锁定中心则关闭平滑。
  - 背景（地块/网格）随相机移动产生滚动感，建议用 `TileMap` 或可平铺纹理表现移动参照。

### 5.3 敌人与刷怪（Spawner）
- 敌人在玩家视野外围**按波次/时间曲线**刷新，随时间提高数量、血量与种类。
- 敌人类型示例：
  - **追击型**：直线冲向玩家。
  - **远程型**：保持距离并发射子弹（需要躲弹幕）。
  - **群集型**：数量多、单体弱，制造压迫感。
  - **精英/Boss**：定时出现，掉落更好的道具。

### 5.4 机关（Hazards）
沿用《以撒》的环境伤害机关，置于大地图中：
- 尖刺地块（踩上扣血）、火焰/喷火口（周期性）、毒池/减速带、可推动的石块、激光等。
- 机关数据化配置（伤害、触发周期、范围），便于程序化撒布与 AI 生成关卡。

## 6. 道具系统

### 6.1 主动道具（可充能，Active Items）
- 占用一个主动栏位，通过**充能**后手动释放（如击杀/拾取/时间充能）。
- 示例：全屏清屏、短时无敌冲刺、范围爆炸、临时召唤物。
- 关键字段：`chargeType`（充能方式）、`chargeMax`（所需充能）、`effect`（效果 id）。

### 6.2 一次性消耗品（Consumables）
- 拾取即用或可携带后使用，用完消失。
- 示例：药水（回血）、临时增益（限时加攻速）、炸弹（手动放置炸开机关/敌群）。

### 6.3 被动遗物（Passive Relics）
- 拾取后**永久生效**（本局内），可叠加，是 build 构筑的核心。
- 效果类型：
  - **数值修正**：+伤害、+射速、+移速等（加法/乘法）。
  - **行为改写**：子弹穿透、追踪、分裂、反弹、变更弹道（环形/扇形）。
  - **触发型**：命中时几率点燃/冰冻/中毒；受伤时反弹等。
- **协同（Synergy）**：遗物组合产生质变效果（如「分裂」+「子弹数量」=弹幕海），是 Roguelike 乐趣核心。

### 6.4 修正叠加模型（建议）
最终属性 = （基础值 + Σ加法修正） × Π乘法修正。
所有遗物通过往一个「修正器列表」注册修正项来生效，便于动态增删与 AI 生成新遗物。

## 7. 成长与难度

- **经验/升级**：击杀掉落经验，升级时三选一获取遗物/属性（吸血鬼幸存者式）。
- **金币/商店**：可选，掉落金币用于在地图刷新点购买道具。
- **难度曲线（数据化）**：随存活时间推进，刷怪强度、密度、机关复杂度递增。**难度曲线、刷怪规则、掉落概率全部外置为数据**（`spawn_waves.json` 等），留出快速调参空间，平衡通过实际体验迭代调整，避免硬编码。
- **元进度（Meta）**：跨局解锁新角色、新遗物池、起始增益，提升重复可玩性。存档使用 Godot `ConfigFile` 或 `JSON` 写入 `user://`（与 9.5 设置存档区分文件）。作为 MVP 之后的扩展模块。

## 8. 美术与音效（极简、AI 友好）

### 8.1 美术风格
- 推荐**几何形状 / 简约像素**风格：圆形=玩家，三角=敌人，小圆点=子弹等。
- 优点：AI 易生成、文件小、辨识度高、性能好；后期可逐步替换为更精细素材。
- 统一规范：固定调色板、固定贴图尺寸（如 32×32），便于批量生成与替换。

### 8.2 美术规范先行（工作流）
- **流程**：先制定**统一调色板 + 固定贴图尺寸 + 资源命名规范**，再让 AI 批量生成素材，保证风格统一、可随时替换。
- **占位图先行**：开发初期一律用几何占位图，把玩法跑通后再替换为正式美术，避免美术返工阻塞玩法验证。

### 8.3 音效
- 简短电子音效（射击、命中、拾取、升级、受伤），可由 AI/合成器批量生成。

## 9. 技术架构（AI 友好 + 数据驱动）

### 9.1 引擎/语言
- **确定使用 Godot 4.6.3 + GDScript**：开源、轻量、场景/脚本均为文本化资源，AI 易读写与生成，自带 2D 节点（`Node2D` / `Camera2D` / `TileMap` / `Area2D` 等）非常契合本项目需求。
- 项目目录约定：`res://scenes/`（场景）、`res://scripts/`（脚本）、`res://data/`（数值配置）、`res://locale/`（翻译表）、`res://assets/`（美术/音效）；玩家设置存于 `user://settings.cfg`。

### 9.2 架构原则（让代码易于 AI 分析/生成）
- **数据与逻辑分离**：数值放配置文件，逻辑读取配置（见 9.3）。
- **模块清晰、单一职责**：Player / Bullet / Enemy / Item / Hazard / Spawner / Map 各自独立场景与脚本。
- **组件化或实体系统**：用「修正器/效果」组合实现遗物，而非为每个遗物写硬编码分支。
- **善用 Godot 机制**：信号（`signal`）解耦事件、`Resource`（`.tres`）做强类型配置、对象池复用 `Bullet` 节点。
- **命名规范统一**、函数短小、注释明确，便于 AI 上下文理解。

### 9.3 数值配置文件（核心需求 #10）
所有需人工调整的数值集中放在独立文件（推荐 `JSON`，比代码更易被人和 AI 编辑）。在 Godot 中放于 `res://data/`，用 `FileAccess.open()` + `JSON.parse_string()` 读取；如需强类型也可用 `Resource`（`.tres`）。

- **配置热重载**：支持运行时重新读取 `/data` 配置即时生效（如监听文件变化或提供调试热键重载），调参无需重启，效率极高。
- **字段说明文档**：随 `/data` 附一份字段说明（每个字段的含义、单位、取值范围），方便人和 AI 正确填写。

建议结构：
```
/data
  player.json       # 玩家基础属性
  relics.json       # 被动遗物定义
  active_items.json # 主动道具定义
  consumables.json  # 消耗品定义
  enemies.json      # 敌人属性
  hazards.json      # 机关属性
  spawn_waves.json  # 刷怪/难度曲线
```

`player.json` 示例：
```json
{
  "maxHp": 6,
  "moveSpeed": 4.0,
  "fireRate": 2.5,
  "damage": 3.5,
  "bulletSpeed": 8.0,
  "bulletRange": 6.0,
  "bulletCount": 1,
  "pickupRange": 2.0,
  "luck": 0
}
```

`relics.json` 示例（数据驱动的遗物，无需改代码即可新增）：
```json
[
  {
    "id": "relic_sharp_rounds",
    "name": "锋利弹头",
    "desc": "伤害 +1.5",
    "modifiers": [
      { "stat": "damage", "type": "add", "value": 1.5 }
    ]
  },
  {
    "id": "relic_rapid_fire",
    "name": "急速射击",
    "desc": "射速 ×1.3",
    "modifiers": [
      { "stat": "fireRate", "type": "mult", "value": 1.3 }
    ]
  },
  {
    "id": "relic_split_shot",
    "name": "分裂弹",
    "desc": "命中后分裂出 2 颗子弹",
    "behaviors": [
      { "event": "onHit", "effect": "split", "params": { "count": 2 } }
    ]
  }
]
```

> 关键：遗物用「modifiers（数值）+ behaviors（行为）」描述。新增遗物 = 加一条数据，逻辑层只需实现有限的 effect/behavior 种类，极利于 AI 生成与扩展。在 Godot 中，「效果原语」（加成 / 穿透 / 分裂 / 点燃……）可实现为脚本方法或小型 `Node`/`Resource`，由数据中的 `effect` id 映射调用，严禁为每个遗物写独立硬编码分支。

### 9.4 多语言本地化（框架级需求）
本地化必须从框架设计阶段就纳入，而非后期补丁，否则大量硬编码文本难以返工。

**核心原则：代码与配置中不出现面向玩家的硬文本，一律用「文本键（key）」引用。**

- **统一文本键**：所有 UI、道具名、描述、提示等使用 key（如 `relic_sharp_rounds_name`），运行时按当前语言查表得到译文。
- **Godot 实现**：优先使用 Godot 内建本地化系统。
  - 翻译表用 `CSV`（`res://locale/strings.csv`）导入为 `.translation` 资源，调用 `tr("key")` 取译文。
  - 在 `Project Settings → Localization` 注册各语言；运行时用 `TranslationServer.set_locale("en")` 切换。
  - UI 控件文本绑定 key，切换语言时通过 `notification(NOTIFICATION_TRANSLATION_CHANGED)` 或重载界面刷新。
- **数据文件与本地化解耦**：`relics.json` 等只存 `name_key` / `desc_key`（文本键），具体译文放翻译表，新增语言不必改数据文件。
- **格式化与占位符**：动态数值用占位符（如 `"伤害 +{value}"` / `"Damage +{value}"`），避免拼接句子；注意数字、单位顺序在不同语言的差异。
- **字体**：选用覆盖目标语言字符集的字体（至少中、英；如需 CJK 用支持中日韩的字体），避免缺字方块。
- **首批语言建议**：简体中文、English；预留扩展（繁中、日、韩等）。

翻译表 `strings.csv` 示例：
```csv
keys,zh_CN,en
relic_sharp_rounds_name,锋利弹头,Sharp Rounds
relic_sharp_rounds_desc,伤害 +{value},Damage +{value}
ui_settings,设置,Settings
ui_resume,继续,Resume
```

数据文件改为引用文本键（与 9.3 配合）：
```json
{
  "id": "relic_sharp_rounds",
  "name_key": "relic_sharp_rounds_name",
  "desc_key": "relic_sharp_rounds_desc",
  "modifiers": [ { "stat": "damage", "type": "add", "value": 1.5 } ]
}
```

### 9.5 设置系统（框架级需求）
设置系统同样需在框架阶段搭好，提供统一的「读取 / 修改 / 持久化 / 即时生效」通路，方便后续不断扩充设置项。

- **统一设置管理器（autoload 单例 `Settings`）**：集中管理所有设置项的默认值、当前值、读写与广播。
- **持久化**：存档到 `user://settings.cfg`（Godot `ConfigFile`）或 JSON；启动时加载，修改时保存。
- **即时生效**：设置变更通过信号（如 `setting_changed(key, value)`）广播，相关系统订阅后实时响应，无需重启。
- **可扩展**：每个设置项以「键 + 类型 + 默认值 + 取值范围」描述，新增设置=加一条配置 + 一个 UI 控件，逻辑统一走管理器。
- **设置界面本地化**：设置项标签全部使用文本键（与 9.4 联动），其中包含「语言选择」项。

建议覆盖的设置分类：
| 分类 | 示例设置项 |
|------|-----------|
| 语言 | 界面语言切换（联动 9.4） |
| 画面 | 全屏/窗口、分辨率、垂直同步、画面缩放、特效/伤害数字开关 |
| 音频 | 主音量、音乐音量、音效音量（分总线 Bus 控制） |
| 操作 | 按键重绑定（移动/瞄准/主动道具/暂停）、是否「松开方向键停火」、瞄准模式（4 方向/自动瞄准） |
| 辅助功能 | 屏幕震动开关、高对比/色盲友好配色、字体大小 |
| 游戏 | 难度倾向（若开放）、伤害数字显示、教程提示开关 |

`settings.cfg` 示例：
```ini
[general]
locale="zh_CN"

[video]
fullscreen=false
vsync=true

[audio]
master=1.0
music=0.8
sfx=0.9

[gameplay]
fire_on_release=false   ; 松开方向键是否停火
aim_mode="8dir"          ; 8dir / auto
screen_shake=true
```

> 设计要点：**所有面向玩家的文本走本地化键（9.4），所有可调玩家偏好走设置管理器（9.5）**，二者构成框架的两条横向基础设施。

### 9.6 数据埋点 / 玩家数据收集（框架级需求）
项目需收集玩家数据用于后续分析与平衡迭代，**必须从框架阶段就预留统一的数据收集接口**，避免后期在业务代码里临时硬塞导致难以维护。

- **统一埋点管理器（autoload 单例 `Analytics`）**：对外暴露简洁接口，如 `track_event(event_name, params)`，由它统一缓冲、批量上报或落盘。
- **埋点与业务解耦**：各系统通过 `signal` 或调用 `Analytics.track_event()` 上报，**不把上报细节散落进业务逻辑**；事件名与字段集中定义为常量/配置，避免裸字符串。
- **上报实现可插拔**：通过接口隔离具体上报方式（本地落盘 JSON / HTTP 上报后端等），切换后端不改业务代码；即使后端暂未就绪，也先把接口与事件留好（先本地落盘）。
- **隐私合规**：在 `Settings` 中提供「数据收集开关」，不收集敏感个人信息；遵守目标平台与地区的隐私要求。

建议预留的关键埋点事件：
| 事件 | 触发时机 | 关键字段（示例） |
|------|---------|-----------------|
| `run_start` | 开局 | 角色、起始遗物、版本、locale |
| `run_end` | 一局结束 | 存活时长、击杀数、原因（死亡/退出）、最终等级 |
| `player_death` | 玩家死亡 | 位置、时间、击杀数、当前波次、死于何种敌人/机关 |
| `level_up` | 升级 | 等级、提供的三选一选项、玩家选择 |
| `relic_pick` | 获得遗物 | 遗物 id、当前已有遗物组合 |
| `item_use` | 使用主动道具/消耗品 | 道具 id、剩余充能 |
| `wave_milestone` | 难度/波次节点 | 波次、敌人密度、当前 DPS/承伤 |

`track_event` 调用示例：
```gdscript
Analytics.track_event("player_death", {
    "pos": player.global_position,
    "time": run_time,
    "kills": kill_count,
    "wave": current_wave,
})
```

> 设计要点：埋点是框架第三条横向基础设施（与本地化、设置并列）——**所有数据上报走 `Analytics` 统一接口，关键节点从第一行代码起就留好钩子**。

### 9.7 AI 友好工程实践（让 agent 易读、易写、易扩展）
本项目目标之一是让 AI agent 能高质量续写/生成代码，为此除"数据驱动 + 单一职责 + 信号解耦"外，还约定以下工程实践（强制规则见 `.codebuddy/rules/game-coding-rules.md` 第 14~18 条）：

- **项目索引（`AI导航.md`）**：项目地图，列出"加某类内容该改哪些文件、扩展点在哪"。agent 开工前先读它定位，避免盲目全仓搜索。
- **词表与契约（`词表与契约.md`）**：集中维护所有"约定字符串"白名单（`stat` / `effect` / `behavior.event` / 埋点 `event_name` / 设置 `key` / 本地化 key 前缀）。代码以**常量/枚举**引用，**禁止裸字符串与编造 id**；新增 id 先登记再实现再使用。这是数据驱动可靠性的关键——防止 AI 写错字段名。
- **数据校验 + fail-fast**：`DataLoader` 加载时校验字段/类型/取值范围，出错打印**具体文件名+字段+期望值**，让 AI 能据报错自我纠错。
- **黄金样例**：每类数据文件保留一条带注释的范例条目，AI 照结构填写最可靠。
- **类型化 GDScript**：变量/参数/返回值都标类型，配合静态检查，帮助 AI 推断用法、减少错误。
- **脚手架模板（`res://templates/`）**：新增同类内容复制模板改名，结构统一、便于模仿。
- **决策记录（`决策记录.md`）**：轻量 ADR，记"决策+理由"，避免 agent 误改既定约束。
- **日志规范 / README**：日志带模块前缀；`README.md` 写清运行调试方式，便于 agent 自行验证。
- **小文件**：脚本保持小而聚焦，便于 AI 整体读入上下文。

> 配套文档：`AI导航.md`（索引）、`词表与契约.md`（白名单）、`决策记录.md`（ADR）。

### 9.8 暂停与游戏状态管理
游戏需提供**暂停功能**：玩家可随时暂停以打开暂停菜单（继续 / 设置 / 退出），暂停期间整局游戏世界冻结。

- **统一暂停机制**：用 Godot 内建 `get_tree().paused = true/false` 暂停整棵场景树。暂停时移动、自动开火、子弹飞行、刷怪、机关、计时等业务节点全部停止（默认 `process_mode = PROCESS_MODE_PAUSABLE`）。
- **暂停菜单仍可交互**：暂停菜单及其下需要响应的 UI 节点设 `process_mode = PROCESS_MODE_ALWAYS`，保证暂停时仍能接收输入。
- **暂停键可重绑定**：通过 `Settings` 重绑定的 InputMap action `pause`（默认 `Esc`）触发，**不硬编码按键**；同一键在暂停态再次按下=继续。
- **菜单内容**：继续（`ui_resume`）、设置（`ui_settings`，复用 9.5 设置界面）、退出到主菜单（`ui_quit`）。所有文本走本地化键（9.4），**不硬编码**。
- **联动设置**：打开设置菜单时自动暂停；可提供「窗口失焦自动暂停」设置项（`gameplay.pause_on_focus_loss`）。
- **解耦实现**：由独立的 `PauseMenu`/状态管理节点统一控制暂停，通过 `signal`（如 `game_paused(bool)`）广播，其它系统按需订阅，不在各业务脚本里散落暂停判断。
- **埋点（可选）**：可在暂停/继续时上报 `game_pause` / `game_resume`（见 9.6 与词表），用于分析玩家中断行为。

> 设计要点：暂停是**游戏状态层**的统一能力——靠 `get_tree().paused` + `process_mode` 实现，按键走设置、文本走本地化，避免散落硬编码。

## 11. MVP（最小可玩版本）开发里程碑

1. **M1 基础移动与射击 + 框架地基**：WASD 移动 + 方向键瞄准 + 自动开火 + 子弹飞行；同时搭建 `Localization`、`Settings`、`Analytics` 三个 autoload 单例（哪怕只有 1~2 个语言/设置项、埋点先本地落盘），让本地化键、设置通路与数据埋点接口从第一行代码起就到位。
2. **M2 敌人与大地图**：开放地图 + 摄像机跟随 + 一种追击敌人 + 刷怪。
3. **M3 成长系统**：经验/升级三选一 + 数值修正器引擎 + 3~5 个被动遗物（道具文本走本地化键）。
4. **M4 道具与机关**：1 个主动道具 + 1 个消耗品 + 2 种机关。
5. **M5 难度与结算**：难度曲线 + 死亡结算 + 数据全部外置到 `/data`。
6. **M6 打磨**：协同效果、更多敌人/遗物、音效、完整设置菜单与多语言、UI。

## 12. 风险与注意事项

| 风险 | 说明 | 缓解 |
|------|------|------|
| 性能 | 大地图 + 海量子弹/敌人易卡顿 | 对象池、空间分区、视野外卸载 |
| 平衡 | 协同过强或过弱 | 数据驱动 + 快速迭代 |
| 操作手感 | 4 方向瞄准空打 | 瞄准辅助、提高子弹速度 |
| 内容量 | Roguelike 需大量道具 | 数据驱动让 AI 量产内容 |
| 本地化返工 | 后期补本地化成本极高 | 框架阶段即用文本键，杜绝硬编码文本 |
| 埋点缺失 | 后期补数据收集需大改业务代码 | 框架阶段即建 `Analytics` 统一接口，关键节点预留钩子 |

## 附录：术语表
- **Build（构筑）**：本局所获遗物/道具的组合形成的玩法流派。
- **Synergy（协同）**：多个遗物组合产生的强化/质变效果。
- **Modifier（修正器）**：对玩家属性的加法/乘法修改项。
- **Spawner（刷怪器）**：按规则在地图上生成敌人的系统。
- **Object Pool（对象池）**：复用实体以提升性能的技术。
