# 测试策略

> 本文档收口项目所有测试相关约定，作为**唯一的测试方法权威**。
> 配套：CICD 路线图（`docs/CICD规划.md`）、编码规则（见 `AGENTS.md` 第 3 步的当前平台规则入口）、回放设计（`游戏设计文档.md` 9.9）、平衡测试（9.10）。
>
> **核心理念**：测试是**多层冗余**的，不是单点。每层抓不同 bug，缺哪一层都会出事。
>
> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。本文档是测试义务权威；改测试层级 / 覆盖率 / 里程碑矩阵时，常见联动为 `docs/CICD规划.md`、`docs/AI协作/实时验证回路.md`、规则自检清单、`docs/决策记录.md`、`docs/AI记忆/项目记忆.md`。

## 1. 五层测试金字塔（总览）

```
                  ▲
                 ╱ ╲
                ╱手 ╲       L5 手动回归 / 探索性测试（发版前）
               ╱──── ╲
              ╱平衡  Sim╲     L4 平衡 sim（批量 AI 玩家，统计平衡性）
             ╱──────── ╲
            ╱黄金回放回归 ╲   L3 整局回归（确定性回放 vs 当前代码）
           ╱──────────── ╲
          ╱  Headless 启动 ╲ L2 烟雾测试（项目能起来）
         ╱────────────────── ╲
        ╱ GUT 单元测试        ╲ L1 单元测试（核心模块逻辑）
       ╱────────────────────── ╲
      ╱  静态校验 + 契约校验      ╲ L0 编译前（pre-commit + CI 阶段 1）
     ╱──────────────────────────── ╲
    ──────────────────────────────────
```

| 层 | 名称 | 工具 | 反馈延迟 | 抓什么 bug |
|----|------|------|---------|----------|
| L0 | 静态/契约校验 | pre-commit hook + CI | **秒** | 裸字符串、未登记 id、schema 错误、字段文档漂移、GDScript 红线违规、死链、版本不同步 |
| L1 | 单元测试 | GUT | 秒~分钟 | 核心算法/边界/状态机正确性 |
| L2 | Headless 启动 | `godot --headless --quit` | 分钟 | 脚本编译错、缺 autoload、场景引用断链 |
| L3 | 黄金回放回归 | `Replay.play(golden_*)` + diff | 分钟 | 数值漂移、行为意外改变、确定性破坏 |
| L4 | 平衡 sim | `tools/sim.gd --runs N` | 小时 | build 强度失衡、过强/过弱组合、胜率异常 |
| L5 | 手动回归 / 探索 | 人工按 checklist | 数十分钟 | UI 体验、手感、未预料场景 |

> 每层缺一不可。L0~L2 是日常守门，L3~L4 是迭代质量，L5 是发版门槛。

## 2. 各层详细约定

### 2.1 L0：静态 / 契约校验
**实施**：见 `docs/CICD规划.md` 阶段 1（A 文档 / 数据校验 / B 词表契约 / E 本地 pre-commit；D commitlint 后续）+ `docs/AI协作/实时验证回路.md`。

**必查**：
- 文档死链 / `修改建议.md` 编号唯一 / `决策记录.md` ADR 连续 / 版本号同步
- 词表白名单（stat / effect / event / setting / locale / action / pool_id / damage_type / status_effect / audio_id / rng_stream）—— 当前由 `tools/sync_contracts.py --check` 与 `tools/validate_data.py` 守门，任何未登记 id 即 fail
- JSON / 数据 schema 校验（当前覆盖 `player.json`、`characters.json`、`weapons.json`、`skills.json`、`enemy_ai_profiles.json`、`enemies.csv`、`gear_mods.json`、`gear_mod_drop_tables.csv`、`gear_mod_fusion_costs.csv`、`hazards.csv`、`map_layouts.json`、`spawn_waves.csv`、`relics.json`、`active_items.json`、`consumables.json`、`credits.json`、`game_modes.json`、`growth.csv`、`growth_pools.json`）
- DataLoader schema 回归测试：`tools/test_data_loader_schema.py` 用临时数据副本覆盖黄金数据、未登记 id、缺失 locale key、类型 / 范围错误与跨文件引用错误；在 GUT runner 完整接入前作为 DataLoader 的 L1 等价守门
- 第一档 GDScript 项目 lint：`tools/lint_gdscript_rules.py` 检查可低误报自动化的官方段落顺序、危险 `:=`、中文硬编码字符串、裸随机 / 时间 / 暂停 API
- 第二档项目规则 lint：`tools/lint_project_rules.py` 检查新增数据字段是否登记到 `client/data/README.md`、locale `zh_CN` / `en` 双语是否保留、release preset 是否误带 debug/dev_tools；`tools/test_project_rules_lint.py` 固定坏样例
- 第三档语义 advisory lint：`tools/lint_semantic_rules.py` 扫描特殊 id 分支、业务脚本绕过 autoload、缺类型签名、长期脚本缺 `# Doc:`、引用未知 contract 常量；默认非阻塞，`tools/test_semantic_rules_lint.py` 固定坏样例
- 本地 pre-commit：`.pre-commit-config.yaml` 串联 Stage 1 主要脚本与 `tools/check_staged_whitespace.py`；该脚本会自动修 staged EOF 多空行，其他 whitespace 错误仍失败；正式 review 前优先跑 `pre-commit run --all-files`，缺少 pre-commit 时散跑等价命令
- Conventional Commits 格式
- 完整 GDScript lint + format（gdtoolkit 等第三方工具，代码规模扩大后）
- 强制类型化（无 untyped 函数签名进 main）
- **代码常量未手改**（`client/scripts/contracts/*.gd` 自动生成，与 md 一致）
- release 导出 preset 不启用 `dev_tools`，且不包含调试脚本、GM 命令表或开发-only overlay 资源；debug/dev_tools 构建才允许创建调试入口

**反馈方式**：fail-fast，输出**「文件 + 字段路径 + 期望值」**，让 AI/人据此自我纠错。

### 2.3 L2：Headless 启动验证
**实施**：见 `docs/CICD规划.md` 2.G。

**做什么**：每个 PR 跑 `godot --headless --quit`，2 分钟内能干净退出 = 通过。
**抓什么**：脚本语法/类型错、缺 autoload 注册、`preload` 路径错、循环依赖、场景节点路径断链。
**升级版**：未来可加 `tools/smoke.gd`，跑通 1 分钟最小循环（开局 → 移动 5 秒 → 主动结束）。

### 2.4 L3：黄金回放回归
**实施**：见 GDD 9.9 + ADR #16 + CICD 4.M。

**流程**：
1. 在关键里程碑录一组 golden replay（玩家或 AIPlayer 跑 30~60 秒）
2. 存到 `tests/replays/golden_<scenario>.replay`，进 git LFS
3. CI 跑 `tools/replay_runner.gd <golden_files>`：
   - 用录制时的 seed + 输入序列重放
   - 对比帧状态（玩家位置、敌人血量、子弹位置、score）
   - 任意 diff > 容差（默认 0）→ fail，输出**首个 diff 帧 + 字段**

**何时更新黄金样例**：
- 改了数值/原语**有意改变行为** → 重录一组（commit message 注明影响）
- 修 bug 不应改变行为 → 黄金样例不动

F8 已提供 `python tools/godot_bridge.py --project client replay-smoke` 验证最小 `.replay` 文件 envelope、recording hash、data fingerprint 和摘要 roundtrip，并提供 `python tools/godot_bridge.py --project client replay-runner` 作为当前 L3 runner 外壳：它读取 `.replay`，校验 envelope，并比较内嵌 summary 或外部 expectation JSON，失败时输出首个字段级 diff。F8 也已入库四条运行时摘要 + 扩展稳定帧样本 / 场景语义字段 golden：`client/tests/replays/golden_basic_run.replay`、`client/tests/replays/golden_pause_resume.replay`、`client/tests/replays/golden_full_death.replay` 和 `client/tests/replays/golden_level_up_choice.replay`。basic 由 `python tools/godot_bridge.py --project client capture-golden-replay` 生成；pause/resume、full-death 与 level-up choice 分别由 `--golden-scenario golden_pause_resume` / `--golden-scenario golden_full_death` / `--golden-scenario golden_level_up_choice` 生成，并通过对应 `replay-runner --replay-file ... --rerun-runtime-summary` 重跑真实 `GameplayRunLoop` 摘要和 `run_summary.frame_samples` 对照。当前帧样本覆盖玩家生命、右移语义、瞄准方向、武器冷却是否就绪、敌人类型、掉落存在性和场景状态；精确玩家坐标仍不进入稳定样本。`golden_full_death` 使用 replay `runtime_events` 在 runner 工具层调用正式 `Combat` 死亡路径，覆盖 GameOverPanel、run 存档删除和失败页摘要；旧 meta 结算字段已随 ADR #117 移除。`golden_level_up_choice` 由测试 harness 显式调用 `debug_enable_level_up_growth()` 后使用真实经验球和 `LevelUpPanel.choose_index()` 覆盖升级候选 decision 记录、选择 id 与修正应用；默认标准模式仍由 `runtime-smoke` 断言不进入 `LEVEL_UP`。`python tools/godot_bridge.py --project client replay-input-smoke` 当前覆盖移动 / 兜底瞄准 action / pause / ui_back 的 gameplay 输入录制首片；F9 默认鼠标瞄准的连续向量还不在 replay 输入 payload 中，后续若要回放真实键鼠局必须先扩展 aim intent / mouse vector 事件。`python tools/godot_bridge.py --project client replay-runner --rerun-runtime-summary` 在未指定 replay 文件时会生成临时输入播放 replay，并验证 runner 播放输入后能得到稳定摘要和帧样本。完整逐帧对照仍按本节流程后续扩展。

**黄金样例必备组**（最小集，落地时录）：

| `client/tests/replays/golden_level_up_choice.replay` | F8 第四条已入库：测试 harness 显式启用成长池后，通过真实经验球触发升级选择，记录 `level_up` decision，验证候选 id 顺序、选中 id 和属性修正应用 | ~3s |

### 2.5 L4：平衡 Sim
**实施**：见 GDD 9.10 + ADR #17 + CICD 4.N。

**何时启用**：M7（核心循环跑通后）。
**做什么**：headless 跑 1000 局 × 不同 seed，输出 CSV：
- 存活时长分布、击杀数、最终等级
- 每个遗物的"被选率 / 持有时胜率"
- 过强组合 top-10、无人选遗物清单
**怎么用**：每周/每次大改后跑一遍，diff 上一次的报告。

### 2.6 L5：手动回归 / 探索性测试
**何时**：每个 milestone 末尾 + 发版前。
**Checklist** 见 §5。

### 2.7 测试隔离与成功协议

- 任何可能读写 `user://` 的自动测试，必须在目标节点进入场景树、触发 `_ready()` 前注入专用 fixture 路径，或由权威 runner 为进程提供独立 `APPDATA` / XDG / `HOME`。备份、删除、迁移再恢复玩家真实文件不能作为测试方案。
- 自动 smoke 的成功不能只看退出码：还必须匹配该测试约定的精确 `ALL PASS` 标志，并拒绝 `SCRIPT ERROR`、`ERROR:` 等致命日志。双进程测试还要有明确的就绪标志、超时和统一终止责任。
- 先跑变更直接触及的目标 suite，确认 fixture 与兼容边界后再跑昂贵全套回归；不得为赶进度临时拼接未经仓库固化的多进程命令。

## 4. 性能预算（按需参考）

性能预算保留为优化目标，但性能测试不属于日常修改、提交前检查、常规回归或交付的默认义务。AI 只有在用户当次明确要求性能测试时，才运行 `startup-probe`、`perf-probe`、Profiler 或同类采样；未收到明确要求时不得因为地图、实体数量、流式加载或“性能敏感”改动而自行追加性能测试。

当用户明确要求性能测试时，超过下列预算才视同该次性能测试失败。预算先粗后细，后续用按需采样结果校准。

| 指标 | 目标（PC，1080p） | 触底警戒 |
|------|------------------|---------|
| 平均 FPS | ≥ 60 | < 50 时拉警 |
| 99% 帧时间 | ≤ 20 ms | > 33 ms 即 fail |
| 子弹峰值数（屏幕内） | ≤ 500 | > 800 必查池溢出 |
| 敌人峰值数（屏幕内） | ≤ 200 | > 300 |
| 内存（启动 5 分钟） | ≤ 512 MB | > 800 MB |
| 首帧到可玩 | ≤ 2 s | > 5 s |
| 数据热重载（一次） | ≤ 100 ms | — |

**度量方式**：
- 用户明确要求时：可用 Godot Profiler、`PoolManager.stats()`、`python tools/godot_bridge.py --project client startup-probe` 和 `python tools/godot_bridge.py --project client perf-probe`。`startup-probe` 测量正式主场景首帧 `BOOT_BEGIN` 到模块世界进入 `PLAYING` 的耗时；`perf-probe` 输出 schema v2 可比较 JSON，30 帧 warmup 后采样 180 帧，并在采样期强制从中心跨到相邻模块。
- `perf-probe` 的 `budget_status` 在被明确要求运行时，对 p99≤20 ms、跨模块 max<33 ms、静态内存≤512 MB、敌人峰值≤200、子弹峰值≤500 做硬失败，同时输出 3×3 active slots、map hash、实体 / 池峰值和 GameClock。
- pre-commit、默认 CI、日常回归和 AI 常规交付不得自动运行性能测试；未来若接 CI，也只能由用户明确触发的手动 workflow 承载。
