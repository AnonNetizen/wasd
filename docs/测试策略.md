# 测试策略

> 本文档收口项目所有测试相关约定，作为**唯一的测试方法权威**。
> 配套：CICD 路线图（`docs/CICD规划.md`）、编码规则（见 `AGENTS.md` 第 3 步的当前平台规则入口）、回放设计（`游戏设计文档.md` 9.9）、平衡测试（9.10）。
>
> **核心理念**：测试是**多层冗余**的，不是单点。每层抓不同 bug，缺哪一层都会出事。
>
> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。本文档是测试义务权威；改测试层级 / 覆盖率 / 里程碑矩阵时，常见联动为 `docs/CICD规划.md`、`docs/AI协作/实时验证回路.md`、规则自检清单、`docs/决策记录.md`、`docs/AI记忆/项目记忆.md`。

## 1. 五层测试金字塔（总览）

```
                  ▲
                 ╱ ╲
                ╱手 ╲       L5 手动回归 / 探索性测试（发版前）
               ╱──── ╲
              ╱平衡  Sim╲     L4 平衡 sim（批量 AI 玩家，统计平衡性）
             ╱──────── ╲
            ╱黄金回放回归 ╲   L3 整局回归（确定性回放 vs 当前代码）
           ╱──────────── ╲
          ╱  Headless 启动 ╲ L2 烟雾测试（项目能起来）
         ╱────────────────── ╲
        ╱ GUT 单元测试        ╲ L1 单元测试（核心模块逻辑）
       ╱────────────────────── ╲
      ╱  静态校验 + 契约校验      ╲ L0 编译前（pre-commit + CI 阶段 1）
     ╱──────────────────────────── ╲
    ──────────────────────────────────
```

| 层 | 名称 | 工具 | 反馈延迟 | 抓什么 bug |
|----|------|------|---------|----------|
| L0 | 静态/契约校验 | pre-commit hook + CI | **秒** | 裸字符串、未登记 id、schema 错误、字段文档漂移、GDScript 红线违规、死链、版本不同步 |
| L1 | 单元测试 | GUT | 秒~分钟 | 核心算法/边界/状态机正确性 |
| L2 | Headless 启动 | `godot --headless --quit` | 分钟 | 脚本编译错、缺 autoload、场景引用断链 |
| L3 | 黄金回放回归 | `Replay.play(golden_*)` + diff | 分钟 | 数值漂移、行为意外改变、确定性破坏 |
| L4 | 平衡 sim | `tools/sim.gd --runs N` | 小时 | build 强度失衡、过强/过弱组合、胜率异常 |
| L5 | 手动回归 / 探索 | 人工按 checklist | 数十分钟 | UI 体验、手感、未预料场景 |

> 每层缺一不可。L0~L2 是日常守门，L3~L4 是迭代质量，L5 是发版门槛。

## 2. 各层详细约定

### 2.1 L0：静态 / 契约校验
**实施**：见 `docs/CICD规划.md` 阶段 1（A 文档 / 数据校验 / B 词表契约 / E 本地 pre-commit；D commitlint 后续）+ `docs/AI协作/实时验证回路.md`。

**必查**：
- 文档死链 / `修改建议.md` 编号唯一 / `决策记录.md` ADR 连续 / 版本号同步
- 词表白名单（stat / effect / event / setting / locale / action / pool_id / damage_type / status_effect / audio_id / rng_stream）—— 当前由 `tools/sync_contracts.py --check` 与 `tools/validate_data.py` 守门，任何未登记 id 即 fail
- JSON / 数据 schema 校验（当前覆盖 `player.json`、`characters.json`、`weapons.json`、`skills.json`、`enemy_ai_profiles.json`、`enemies.csv`、`gear_mods.json`、`gear_mod_drop_tables.csv`、`gear_mod_fusion_costs.csv`、`hazards.csv`、`map_layouts.json`、`spawn_waves.csv`、`relics.json`、`active_items.json`、`consumables.json`、`credits.json`、`game_modes.json`、`meta_progression.json` legacy、`growth.csv`、`growth_pools.json`）
- DataLoader schema 回归测试：`tools/test_data_loader_schema.py` 用临时数据副本覆盖黄金数据、未登记 id、缺失 locale key、类型 / 范围错误与跨文件引用错误；在 GUT runner 完整接入前作为 DataLoader 的 L1 等价守门
- 第一档 GDScript 项目 lint：`tools/lint_gdscript_rules.py` 检查可低误报自动化的官方段落顺序、危险 `:=`、中文硬编码字符串、裸随机 / 时间 / 暂停 API
- 第二档项目规则 lint：`tools/lint_project_rules.py` 检查新增数据字段是否登记到 `client/data/README.md`、locale `zh_CN` / `en` 双语是否保留、release preset 是否误带 debug/dev_tools；`tools/test_project_rules_lint.py` 固定坏样例
- 第三档语义 advisory lint：`tools/lint_semantic_rules.py` 扫描特殊 id 分支、业务脚本绕过 autoload、缺类型签名、长期脚本缺 `# Doc:`、引用未知 contract 常量；默认非阻塞，`tools/test_semantic_rules_lint.py` 固定坏样例
- 本地 pre-commit：`.pre-commit-config.yaml` 串联 Stage 1 主要脚本与 `tools/check_staged_whitespace.py`；正式 review 前优先跑 `pre-commit run --all-files`，缺少 pre-commit 时散跑等价命令
- Conventional Commits 格式
- 完整 GDScript lint + format（gdtoolkit 等第三方工具，代码规模扩大后）
- 强制类型化（无 untyped 函数签名进 main）
- **代码常量未手改**（`client/scripts/contracts/*.gd` 自动生成，与 md 一致）
- release 导出 preset 不启用 `dev_tools`，且不包含调试脚本、GM 命令表或开发-only overlay 资源；debug/dev_tools 构建才允许创建调试入口

**反馈方式**：fail-fast，输出**「文件 + 字段路径 + 期望值」**，让 AI/人据此自我纠错。

### 2.3 L2：Headless 启动验证
**实施**：见 `docs/CICD规划.md` 2.G。

**做什么**：每个 PR 跑 `godot --headless --quit`，2 分钟内能干净退出 = 通过。
**抓什么**：脚本语法/类型错、缺 autoload 注册、`preload` 路径错、循环依赖、场景节点路径断链。
**升级版**：未来可加 `tools/smoke.gd`，跑通 1 分钟最小循环（开局 → 移动 5 秒 → 主动结束）。

### 2.4 L3：黄金回放回归
**实施**：见 GDD 9.9 + ADR #16 + CICD 4.M。

**流程**：
1. 在关键里程碑录一组 golden replay（玩家或 AIPlayer 跑 30~60 秒）
2. 存到 `tests/replays/golden_<scenario>.replay`，进 git LFS
3. CI 跑 `tools/replay_runner.gd <golden_files>`：
   - 用录制时的 seed + 输入序列重放
   - 对比帧状态（玩家位置、敌人血量、子弹位置、score）
   - 任意 diff > 容差（默认 0）→ fail，输出**首个 diff 帧 + 字段**

**何时更新黄金样例**：
- 改了数值/原语**有意改变行为** → 重录一组（commit message 注明影响）
- 修 bug 不应改变行为 → 黄金样例不动

F8 已提供 `python tools/godot_bridge.py --project client replay-smoke` 验证最小 `.replay` 文件 envelope、recording hash、data fingerprint 和摘要 roundtrip，并提供 `python tools/godot_bridge.py --project client replay-runner` 作为当前 L3 runner 外壳：它读取 `.replay`，校验 envelope，并比较内嵌 summary 或外部 expectation JSON，失败时输出首个字段级 diff。F8 也已入库四条运行时摘要 + 扩展稳定帧样本 / 场景语义字段 golden：`client/tests/replays/golden_basic_run.replay`、`client/tests/replays/golden_pause_resume.replay`、`client/tests/replays/golden_full_death.replay` 和 `client/tests/replays/golden_level_up_choice.replay`。basic 由 `python tools/godot_bridge.py --project client capture-golden-replay` 生成；pause/resume、full-death 与 level-up choice 分别由 `--golden-scenario golden_pause_resume` / `--golden-scenario golden_full_death` / `--golden-scenario golden_level_up_choice` 生成，并通过对应 `replay-runner --replay-file ... --rerun-runtime-summary` 重跑真实 `GameplayRunLoop` 摘要和 `run_summary.frame_samples` 对照。当前帧样本覆盖玩家生命、右移语义、瞄准方向、武器冷却是否就绪、敌人类型、掉落存在性和场景状态；精确玩家坐标仍不进入稳定样本。`golden_full_death` 使用 replay `runtime_events` 在 runner 工具层调用正式 `Combat` 死亡路径，覆盖 GameOverPanel、run 存档删除与 meta 结算存档；`golden_level_up_choice` 使用真实经验球和 `LevelUpPanel.choose_index()` 覆盖升级候选 decision 记录、选择 id 与修正应用。`python tools/godot_bridge.py --project client replay-input-smoke` 当前覆盖移动 / 兜底瞄准 action / pause / ui_back 的 gameplay 输入录制首片；F9 默认鼠标瞄准的连续向量还不在 replay 输入 payload 中，后续若要回放真实键鼠局必须先扩展 aim intent / mouse vector 事件。`python tools/godot_bridge.py --project client replay-runner --rerun-runtime-summary` 在未指定 replay 文件时会生成临时输入播放 replay，并验证 runner 播放输入后能得到稳定摘要和帧样本。完整逐帧对照仍按本节流程后续扩展。

**黄金样例必备组**（最小集，落地时录）：
| 文件 | 场景 | 时长 |
|------|------|------|
| `client/tests/replays/golden_basic_run.replay` | F8 首条已入库：固定 seed 出生后采样 180 帧，比较运行时摘要和 30 帧间隔扩展稳定帧样本；尚非击杀 10 敌人 / 升级一次 | ~3s |

| `client/tests/replays/golden_pause_resume.replay` | F8 第二条已入库：pause 输入打开暂停菜单，`ui_back` 恢复运行，帧样本验证 `paused/ui_stack=1` 后回到 `playing/ui_stack=0` | ~3s |
| `client/tests/replays/golden_full_death.replay` | F8 第三条已入库：第 75 帧通过 runtime event 调用正式 Combat 伤害，帧样本验证 `playing` 进入 `game_over/ui_stack=1`，并检查 meta 结算存档与 run 存档删除 | ~3s |
| `client/tests/replays/golden_level_up_choice.replay` | F8 第四条已入库：通过真实经验球触发升级选择，记录 `level_up` decision，验证候选 id 顺序、选中 id 和属性修正应用 | ~3s |
| `golden_relic_synergy.replay` | 拾取 2 个有协同的遗物，验证修正叠加 | 60s |

### 2.5 L4：平衡 Sim
**实施**：见 GDD 9.10 + ADR #17 + CICD 4.N。

**何时启用**：M7（核心循环跑通后）。
**做什么**：headless 跑 1000 局 × 不同 seed，输出 CSV：
- 存活时长分布、击杀数、最终等级
- 每个遗物的"被选率 / 持有时胜率"
- 过强组合 top-10、无人选遗物清单
**怎么用**：每周/每次大改后跑一遍，diff 上一次的报告。

**何时**：每个 milestone 末尾 + 发版前。

## 4. 性能预算（与 L4 联动）

性能不达标视同测试失败。预算先粗后细，落地后用 sim + Profiler 校准。

| 指标 | 目标（PC，1080p） | 触底警戒 |
|------|------------------|---------|
| 平均 FPS | ≥ 60 | < 50 时拉警 |
| 99% 帧时间 | ≤ 20 ms | > 33 ms 即 fail |
| 子弹峰值数（屏幕内） | ≤ 500 | > 800 必查池溢出 |
| 敌人峰值数（屏幕内） | ≤ 200 | > 300 |
| 内存（启动 5 分钟） | ≤ 512 MB | > 800 MB |
| 首帧到可玩 | ≤ 2 s | > 5 s |
| 数据热重载（一次） | ≤ 100 ms | — |

**度量方式**：
- 开发期：Godot Profiler + `PoolManager.stats()` 每帧水位
- F8 首片：`python tools/godot_bridge.py --project client perf-probe` 输出 schema v2 可比较 JSON，包含 30 帧 warmup 后 180 帧 avg / p95 / p99 / max 帧时间、active / peak entity counts、pool final stats / peak active、等级、击杀、GameClock、状态和 `budget_status`
- CI：headless `tools/perf_probe.gd` 跑 60 秒采样 → 报表
- 触底警戒触发时：CI 标黄但不阻塞 PR；连续 3 次或跌破 fail 线 → 必须修

F8/F9 首片的临时 runner 先放在 `client/tools/`：`l1_smoke.gd`、`replay_smoke.gd`、`replay_runner.gd`、`replay_input_smoke.gd`、`golden_replay_capture.gd`、`rng_audit.gd`、`perf_probe.gd`、`f9_demo_smoke.gd`、`debug_tools_smoke.gd`，由 `tools/godot_bridge.py` 暴露 `l1-smoke`、`replay-smoke`、`replay-runner`、`replay-input-smoke`、`capture-golden-replay`、`rng-audit`、`perf-probe`、`f9-demo-smoke`、`debug-tools-smoke`、`debug-tools-release-smoke`。`l1-smoke` 当前覆盖 `SkillSystem` 伤害 primitive、Combat 路由、ability tag gating、`StatusEffectComponent` 自我沉默、测试用 poison DoT、快照恢复、过期移除和恢复释放；`settings-smoke` 覆盖设置持久化、运行时语言刷新、输入绑定和当前可见按钮类 UI 在英文 `en` 下的文本宽度适配，防止中文短文本掩盖英文截断 / 溢出风险；`runtime-smoke` 覆盖有限地图菱形边界、敌人移动边界、机关续局、按住 `fire` 后主武器出弹、默认技能 `skill_overdrive_rounds` 的临时武器属性强化，以及 `show_stats_panel` 按住显示 / 松开隐藏且不暂停；`f9-demo-smoke` 覆盖 FEA-12 机关存在、造成玩家伤害和 run 保存 roundtrip。当前 golden 已放入 `client/tests/replays/golden_basic_run.replay`、`client/tests/replays/golden_pause_resume.replay`、`client/tests/replays/golden_full_death.replay` 和 `client/tests/replays/golden_level_up_choice.replay`，并覆盖扩展稳定帧样本 diff、死亡结算 runtime event 和升级选择 decision 语义；`rng_audit.gd` 用 10,000 个 run seed 检查默认 6 个 RNG 子流前 4 次 `randf()` 的 Pearson 相关性，最大绝对相关阈值为 0.06；`perf_probe.gd` 输出 schema v2 可比较性能 / 平衡基线，记录 warmup 后帧时间分布、实体峰值、池峰值和预算状态；`debug_tools_smoke.gd` 覆盖 GM 命令可用性和 release 禁用 guard；后续接入 GUT / 完整 L3 后再扩长期单测和更多黄金回放。

**命名约定**：
- 单测文件 `test_<被测模块>.gd`
- 测试函数 `test_<行为>_<期望>`（如 `test_apply_damage_with_armor_reduces_amount`）
- 用 GUT 的 `before_each` / `after_each` 重置 `RNG` 与假 `GameClock`，避免测试间污染

| 改动类型 | 必跑 | 必新增 / 更新 |
