## 多模式资源复用原则

- 输入：用户提出后续可能会有多种游戏模式，角色、遗物等资源不要做得强耦合，应易于复用。
- 评估：该需求与既有“扩展优先 / 数据驱动 / 禁止按 id 写特殊分支”一致，价值高；主要风险是过早做重模式系统，因此本次只落设计约束和数据边界，不实现模式系统。
- 产出：新增 ADR #65；同步 GDD、AI 导航、数据手册、三平台规则入口、项目记忆与 `current_state.json`。
- 结论：资源本体默认模式无关；后续 `game_modes.json` 等模式配置只负责组合资源池、权重、条件、禁用列表和轻量覆盖，禁止为某个模式复制资源或写 `if mode_id == ...` 内容分支。

## 未来多人 PvE / PvP 预留边界

- 输入：用户确认当前不做多人，但未来可能做多人 PvE 甚至 PvP，要求现在预留空间。
- 评估：多人会影响输入、阵营、伤害、模式、回放、存档和埋点边界；现在实现网络、同步协议、服务器权威或反作弊性价比低，因此只预留轻量架构边界。
- 产出：新增 ADR #66；同步 GDD §6.7、数据手册、AI 导航、三平台规则、AI 知识库索引、项目记忆与 `current_state.json`。
- 结论：当前仍单人离线优先；业务逻辑避免写死唯一玩家 / 唯一队伍 / 玩家只打敌人的假设，统一保留 actor、participant、team、intent 与 Combat source/target 边界。
