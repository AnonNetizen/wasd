## 0. 使用方式

本文档是小服务器条件下的玩法参考清单。它回答的是：在买不起大型服务器、不能承载实时联机或服务器权威战斗的情况下，服务器仍然能怎样让游戏更好玩。

核心原则：

- 战斗默认仍在客户端本地运行，保持离线可玩。
- 服务器只处理小数据：seed、挑战配置、排行榜摘要、死亡记录、路线压缩点、全服累计值、轻量构筑摘要。
- 不做实时 PvP、实时合作、服务器权威同步、大型大厅、可交易经济或强竞技反作弊。
- 所有下发内容必须能降级：服务器不可用时，客户端仍可使用本地每日挑战、历史缓存或普通模式。
- 真正落地前必须新增正式后端 / 在线服务边界，不能把业务脚本直接写死到具体 HTTP API。

## 1. 基础异步玩法

这一组方案更稳，工程成本低，适合先做第一批在线感。

| 方案 | 玩法价值 | 小服务器负载 | 适配方式 |
|------|----------|--------------|----------|
| 每日巢穴 / 每周巢穴 | 所有玩家打同一个 seed，比成绩，形成回访理由 | 极低：下发 seed 和规则 | 复用 `RNG`、`Replay`、`game_modes.json` 与固定数据指纹 |
| 异步排行榜 | 给高手和好友比较目标 | 低：上传结算摘要 | 上传角色、seed、存活时间、击杀、等级、遗物 / 技能摘要、版本号 |
| 死亡残响 | 玩家死亡会在别人局里留下信号，增强世界感 | 低：死亡坐标和死因摘要 | 同一挑战 seed 内显示阵亡信标，可给提示 / 小补给 / 小事件 |
| 幽灵路线 / 残影回放 | 看高手路线、好友残影或自己的历史残影 | 低到中：压缩位置关键帧 | 每 1 秒记录一个点位，不上传完整战斗 |
| 全服战役进度 | 让玩家感觉一起推进战线 | 极低：累计计数器 | 击杀、通关、摧毁巢核等推进全服节点 |
| 异步援军 / 影子队友 | 其他玩家的构筑变成可召唤支援 | 低：构筑摘要 | 本地 AI 读取上传摘要生成短时支援，不联网控制 |
| 限时巢穴词缀 | 每天 / 每周有变化，不用频繁发版 | 极低：配置下发 | 下发规则包：敌人权重、机关密度、禁用池、奖励加成 |
| 预设情报留言 | 玩家互相提示地图风险 | 低：预设短语 id | 只允许白名单短语，避免自由文本审核压力 |

推荐第一片：

1. 本地每日巢穴。
2. 结算摘要与本地最好成绩。
3. 服务器排行榜。
4. 死亡残响。

这样即使服务器暂时不上线，前两步也能先以本地功能验证。

## 2. 更新颖的异步玩法

这一组更有辨识度。它们的共同目标是：让服务器像“敌巢的活体大脑”，记住玩家群体行为，并用轻量数据反过来改变单机局。

### 2.1 敌巢进化系统

全服玩家常用什么打法，敌巢就逐渐适应什么。

例子：

- burn 构筑胜率过高，下周部分巢穴长出炭化甲壳，火伤抗性提高。
- 玩家普遍高速风筝，敌巢生成更多冲锋型 / 领地型敌人。
- 某个 Boss 被大量速杀，下次出现时带一个护卫词缀。

服务器只需要统计构筑标签、伤害类型、胜负、存活时间和版本号。它不需要跑战斗。

适合《破巢者》的原因：敌方不是固定关卡列表，而像一个会学习的跨宇宙生态。

### 2.2 玩家失败变成敌人内容

玩家死亡后，他的构筑被敌巢吞噬，变成其他玩家会遇到的小型精英、事件或残影。

例子：

- 玩家用 `burn + 近战 + 高攻速` 构筑死在第 12 分钟。
- 服务器保存轻量摘要。
- 其他玩家之后遇到“被巢同化的破巢者残影”，它使用类似风格攻击。
- 击败残影可获得与该构筑相关的临时奖励。

这个方案比普通死亡标记更有趣：失败不是墓碑，而是变成敌巢内容。

### 2.3 星域污染图

服务器维护一张星域战线图，每个节点对应一个挑战 seed 或规则包。玩家胜利会降低污染，失败会提高污染。

污染高的节点：

- 敌人更密。
- 机关更多。
- 奖励更稀有。
- 更容易出现被同化的玩家残影。

污染低的节点：

- 解锁补给。
- 适合新人刷资源。
- 开启剧情日志或战报。

服务器维护节点状态和计数即可，玩家会感觉全服在推进一场战争。

### 2.4 敌巢悬赏系统

服务器根据全服数据自动生成“今天最需要处理的问题”。

例子：

- 清除指定数量的猎物型感染体。
- 用非火焰伤害击败炭化单位。
- 在 FEA-12 机关区域存活 8 分钟。
- 禁用某个高胜率技能后进入悬赏榜。

它比固定每日挑战更活，因为它能根据玩家行为变化出题。

### 2.5 构筑孢子

玩家通关或死亡后，可以上传一个构筑孢子。其他玩家局内遇到时，不直接获得完整遗物，而是出现一个变异选择。

例子：

- 高射速构筑孢子：射速提高，但子弹变薄或伤害降低。
- 过载弹流构筑孢子：短时射速强化更猛，但敌人有概率获得弹幕抗性或部署压制节点。
- 生存构筑孢子：获得护盾，但降低移动速度。

这样能把玩家构筑变成异步内容，同时通过本地数据表控制奖励边界。

### 2.6 玩家投稿巢穴配方

不让玩家上传脚本或完整 mod，只允许上传白名单挑战配方：

- 角色限制。
- 敌人池。
- 机关密度。
- 初始遗物。
- 目标条件。
- 禁用列表。

服务器审核字段合法性后生成“玩家投稿巢穴”。这像轻量创意工坊，但安全和服务器成本都更低。

### 2.7 巢核人格

每周服务器给敌巢一个人格策略，本质是下发一组权重和词缀。

例子：

- 贪食型：吞噬掉落，玩家必须抢资源。
- 复仇型：优先生成杀死过大量玩家的敌人。
- 猎手型：针对高胜率角色追加追踪敌人。
- 筑巢型：机关密度提高，地图兴趣点更重要。

这个包装能把数据词缀变成有角色感的敌方意志。

### 2.8 间接阵营投票

玩家结算后把资源投给不同战线策略：

- 前线推进：降低某节点污染。
- 科研解析：解锁敌人弱点。
- 救援行动：增加别人局内补给信标。
- 高危突入：提高奖励，但提高污染波动。

玩家不是直接 PvP，但集体选择会塑造后续环境。

## 3. 我最推荐的组合

最有辨识度的组合是：

1. 敌巢进化系统。
2. 玩家失败变成敌人内容。
3. 星域污染图。

这三者放在一起，可以让服务器发挥出远超硬件成本的设计价值：它不承载实时战斗，却让游戏每天变化、记住玩家、反过来回应玩家。

第一片不要一口气做完整系统，建议拆成：

1. **本地每日巢穴**：固定 seed + 固定规则，先验证确定性入口。
2. **结算摘要 schema**：记录构筑标签、伤害类型、技能 / 遗物摘要、死因、存活时间。
3. **死亡残响本地模拟**：从本地历史结算生成残响，不依赖服务器。
4. **服务器排行榜 / 摘要上传**：只上传结算摘要。
5. **敌巢进化词缀下发**：服务器基于摘要统计下发下周词缀。
6. **同化残影首片**：服务器从失败摘要生成精英残影配置。
7. **星域污染图 UI**：把多个挑战节点包装成战线地图。

## 4. 与现有架构的关系

适配点：

- `RNG`：每日 / 每周巢穴使用固定 seed，回放和挑战可复现。
- `Replay`：上传摘要而不是完整战斗；必要时保留本地 replay 供调试。
- `SaveManager`：本地缓存挑战、最好成绩、已见残响和离线队列。
- `DataLoader`：服务器下发的规则包应映射为受控配置，不直接执行脚本。
- `GameState` / `UIManager`：星域图、排行榜、战报都走正式 UI 流程。
- `Combat` / `StatusEffect` / `SkillSystem`：敌巢进化词缀最终仍要落到正式伤害、状态和技能管线。
- `ModLoader`：玩家投稿巢穴可以借鉴白名单数据 patch 思路，但不能执行玩家脚本。
- `PlatformServices`：Steam 成就 / 富状态 / overlay 仍走平台门面；它不等于游戏自己的在线服务。

需要新增或明确的边界：

- 一个独立的在线服务门面，例如后续可设计为 `OnlineServices` 或 `BackendServices` autoload。
- 上传 payload schema：结算摘要、构筑摘要、死亡残响、路线关键帧、玩家投稿配方。
- 离线缓存和失败重试策略。
- 隐私开关与数据收集说明。
- 服务器不可用时的降级规则。

## 5. 不建议短期做

| 方向 | 为什么暂缓 |
|------|------------|
| 实时多人合作 | 会牵动网络同步、延迟、输入预测、相机、UI、断线恢复 |
| PvP | 需要服务器权威、反作弊、匹配和平衡，远超小服务器范围 |
| 强竞技排行榜 | 没有反作弊时容易被改档和伪造成绩污染 |
| 可交易装备经济 | 会引入账号、物品所有权、风控和客服成本 |
| 大型在线大厅 | 服务器压力和运维成本高，且不直接提升核心局内玩法 |
| 自由文本留言 | 审核、滥用和本地化成本高，先用预设短语 |

## 6. 点名推进时的推荐说法

以后可以直接用这些短句：

- “推进小服务器每日巢穴第一片，只做本地 seed 和本地最好成绩。”
- “把敌巢进化系统拆成工作包，不实现。”
- “先做死亡残响本地模拟，不接服务器。”
- “比较星域污染图和敌巢悬赏，告诉我哪个更适合 Demo。”
- “设计 OnlineServices 边界，但先别写代码。”
- “从小服务器玩法备忘里选一个最有卖点的，写成 GDD / ADR 草案。”

如果没有明确点名，AI 不应从本文档自行挑选功能推进。
