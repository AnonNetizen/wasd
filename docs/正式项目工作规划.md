# 正式项目工作规划

> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。
> 本文档的权威范围：MVP 验证和 AI 工作框架完成后，完整项目 `client/` 的阶段路线、任务顺序、交付物与验证门槛。
> 常见联动：`docs/TODO.md`、`docs/AI导航.md`、`docs/AI知识库索引.md`、`docs/_kb_index.json`、`docs/AI记忆/项目记忆.md`、`docs/AI记忆/current_state.json`。

---

## 1. 定位

本规划是正式项目从“文档 / MVP 验证”转入“完整项目实现”的工作路线图。后续 AI agent 接到任务时，先完成 `AGENTS.md` 开工 5 步，再读本文判断当前应该推进哪个阶段、需要读哪些权威文档、交付哪些文件、跑哪些验证。

本文不替代 GDD、ADR、词表或测试策略。发生冲突时按 `docs/AI知识库索引.md` 的权威层级处理：先看 `AGENTS.md` / 规则 / ADR / GDD / 词表 / 测试策略，再更新本文。

## 2. 工作原则

- **先骨架，后内容**：完整项目先立 `client/` Godot 工程、autoload、数据加载、存档、UI 栈和验证闭环，再扩展大量内容。
- **先可验证，后做大**：每个阶段都必须有可运行 / 可校验的门槛；不要把多个阶段的风险混在一次大改里。
- **MVP 只迁移经验，不搬临时代码**：MVP 结论可进入设计、数值和工具流程；临时代码默认不直接混入正式 `client/`。
- **数据驱动优先**：角色、武器、敌人、遗物、升级、局外成长、文案和设置尽量通过 `client/data/`、`client/locale/`、词表与生成常量表达。
- **文档与测试同阶段落地**：长期模块必须有 `docs/代码/<module_id>.md`；行为变化按 `docs/测试策略.md` §7 补测试或说明验证缺口。

## 3. 后续 AI 使用方式

1. 读 `docs/AI记忆/current_state.json` 确认当前阶段和候选下一步。
2. 读本文的阶段表，选择最靠前且未完成的阶段任务。
3. 按该阶段的“必读文档”和 `docs/AI协作/上下文预算.md` 控制读取范围。
4. 完成阶段任务后，更新 `docs/TODO.md`、`current_state.json` 和当日会话日志；若产生新决策，追加 ADR。
5. 大更改按 AI Git 提交策略自动 commit；只 stage 本次明确修改的文件。

## 4. 阶段路线

| 阶段 | 目标 | 主要交付物 | 验证门槛 |
|------|------|------------|----------|
| F0：转场收口 | 把 MVP 验证完成和 AI 框架完成转化为正式项目入口 | 本规划、TODO/current_state 更新、ADR | `docs_health_check`、JSON 校验 |
| F1：正式工程骨架 | 创建完整项目 `client/` Godot 工程和基础目录，headless 可启动 | `client/project.godot`、基础场景、目录骨架、运行说明 | `godot_bridge.py headless-boot` 或等价 Godot 启动验证 |
| F2：横向 autoload 骨架 | 落地基础设施与协调中枢空实现，建立 API 边界 | `DataLoader`、`Localization`、`Settings`、`Analytics`、`RNG`、`GameClock`、`Replay`、`GameState`、`UIManager`、`PoolManager`、`SaveManager`、`AudioManager` 骨架与模块文档 | headless 启动；核心 autoload smoke test |
| F3：数据 / 契约闭环 | 让正式项目数据、locale、词表常量可加载、可校验 | `player.json`、`meta_progression.json`、`growth.csv` / `growth_pools.json` 初版，`DataLoader` 同时支持 CSV / JSON 校验，生成常量引用 | `sync_contracts.py --check`、`validate_data.py` |
| F4：最小可玩闭环 | 完整项目内实现玩家、输入、武器、子弹、敌人、刷怪、失败重开 | 基础场景、对象池接入、Combat 接入、基础 HUD | headless boot；手动跑通 1 分钟基础战斗 |
| F5：暂停 / 存档 / 续局 | 强玩家存档闭环落地 | `SaveManager` meta/run、暂停保存退出、主菜单继续游戏、迁移 / 损坏隔离测试 | 存档 roundtrip、坏档隔离、run 恢复验证 |
| F6：成长与局外循环 | 经验升级和深局外成长进入正式项目 | `GrowthSystem`、升级选择 UI、`MetaProgressionSystem`、结算奖励、永久升级 | 固定 seed 升级验证；meta 存档 roundtrip |
| F7：设置 / 本地化 / UI 栈 | 玩家可见 UI 与可调偏好稳定 | 主菜单、HUD、暂停、设置、升级、结算、局外成长界面；语言切换 | 中英文切换、输入重绑定、UIManager 栈行为验证 |
| F8：回放 / 测试 / 平衡基线 | 建立回归安全网 | L1 单测、L2 headless、首批黄金回放、基础 perf / 平衡采样 | GUT/headless/replay 对应门槛通过 |
| F9：内容扩展与 Demo 打磨 | 开始规模化内容和手感打磨 | 首批角色、武器、遗物、敌人、机关、音频 / 美术占位规范 | 完整手动回归 checklist；性能预算检查 |

## 5. 分阶段执行细则

### F1：正式工程骨架

必读：`docs/AI导航.md`、`docs/代码文档规范.md`、`docs/测试策略.md`、`docs/游戏设计文档.md` 中项目结构 / autoload / 输入相关段落。

建议拆分：
- 创建 `client/project.godot`、`client/scenes/`、`client/scripts/`、`client/data/`、`client/locale/`、`client/assets/`、`client/templates/`。
- 添加最小启动场景和 README / 运行说明。
- 确认 `tools/godot_bridge.py` 能指向正式 `client/`，或记录暂用命令。

### F2：横向 autoload 骨架

必读：GDD §9.3~§9.19、`docs/AI导航.md` 5.2、`docs/代码文档规范.md`、`docs/测试策略.md` §7。

建议拆分：
- 先做 `DataLoader`、`RNG`、`GameClock`、`GameState`，再做 `Settings`、`Localization`、`UIManager`。
- `SaveManager`、`PoolManager`、`Combat` / `StatusEffect`、`AudioManager` 可独立提交，避免一次改太大。
- 每个长期模块同步 `docs/代码/<module_id>.md`，脚本头加 `# Doc:`。

### F3：数据 / 契约闭环

必读：`client/data/README.md`、`client/locale/README.md`、`docs/词表与契约.md`、`tools/sync_contracts.py`、`tools/validate_data.py`。

建议拆分：
- 先保证已存在的 `player.json`、`meta_progression.json`、`strings.csv` 可被正式项目加载。
- 新增 `growth.csv` / `growth_pools.json` 时同步数值手册、GDD 经验升级章节和词表中必要 id。
- 所有裸 id 都通过生成常量引用。

### F4：最小可玩闭环

必读：GDD 玩家 / 输入 / 武器 / 敌人 / 刷怪 / 对象池 / 伤害章节，`docs/测试策略.md`。

建议拆分：
- 先玩家移动和相机居中，再武器 / 子弹，再敌人 / 刷怪。
- 高频实体第一版就接 `PoolManager`，不要临时 `instantiate` / `queue_free`。
- 伤害第一版就走 `Combat.apply_damage`。

### F5：暂停 / 存档 / 续局

必读：GDD §9.12、§9.16、`docs/词表与契约.md` §14、`docs/测试策略.md` 存档义务。

建议拆分：
- `meta` 与 `run` kind 分开实现，但共享标准头字段、迁移、原子写入和损坏隔离。
- 暂停菜单“保存并退出”和主菜单“继续游戏”要通过 `GameState` / `UIManager` 协调。
- 恢复点至少覆盖玩家、敌人、子弹、掉落、经验、遗物、RNG 和 GameClock。

### F6：成长与局外循环

必读：GDD §7.1 / §7.2、`client/data/meta_progression.json`、`client/data/README.md`、`client/locale/strings.csv`。

建议拆分：
- 先 `GrowthSystem` 的经验阈值、升级暂停和候选抽取。
- 再 `MetaProgressionSystem` 的结算奖励、账号等级、升级购买和解锁。
- 待决策 E 未完成前，升级选项池只实现保守数据结构，不写死内容边界。

### F7：设置 / 本地化 / UI 栈

必读：GDD §9.4 / §9.5 / §9.14、`client/locale/README.md`、`docs/词表与契约.md` 第 5~7 节。

建议拆分：
- 主菜单、HUD、暂停菜单、设置菜单优先。
- 升级选择、结算、局外成长界面跟随 F6。
- 所有玩家文本走 locale key；输入全部走 InputMap action 和 Settings 重绑定。

### F8：回放 / 测试 / 平衡基线

必读：`docs/测试策略.md`、GDD §9.9 / §9.10 / §9.18、`docs/CICD规划.md`。

建议拆分：
- 先 L1 单测覆盖 `RNG`、`GameClock`、`GameState`、`SaveManager`、`ModifierEngine`、`Combat`。
- 再录制 `golden_basic_run`、`golden_pause_resume`、`golden_full_death`、`golden_relic_synergy`。
- 平衡 sim 先预留 `AIPlayer` 接口，避免过早做完整自动玩家。

### F9：内容扩展与 Demo 打磨

必读：内容对应任务模板、`docs/词表与契约.md`、`client/data/README.md`、`client/locale/README.md`。

建议拆分：
- 先建立内容黄金样例，再批量增加角色 / 武器 / 敌人 / 遗物。
- 每次新增内容优先走数据和 locale；只有缺 primitive 时才改逻辑层。
- Demo 前跑完整 L5 手动回归：输入设备、语言、存档迁移、回放重现、性能预算。

## 6. 阶段门禁

每个阶段结束至少满足：
- 代码或文档对应的权威来源已同步。
- `docs/AI记忆/current_state.json` 的 `next_actions` 指向下一阶段。
- 新增长期文档登记进 `docs/_kb_index.json`。
- 修改 ADR 后，`docs/AI知识库索引.md` 的 ADR 追踪矩阵覆盖最新编号。
- 验证命令写入当日会话日志；失败项要说明原因和后续处理。

## 7. 任务选择建议

后续 AI 如果没有用户指定任务，按下列优先级选择：

1. 当前阶段未完成且阻塞后续阶段的骨架任务。
2. 能形成独立 commit、独立验证的最小纵切。
3. 修复当前验证失败或文档漂移。
4. 扩展内容、调数值和打磨体验。

不要在 F1 / F2 期间提前批量加内容；不要在 SaveManager 和 GameState 未稳定前做复杂局外成长；不要在词表 / 数据校验未闭环前引入大量裸 id。
