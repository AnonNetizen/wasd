# 正式项目工作规划

> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。
> 本文档的权威范围：MVP 验证和 AI 工作框架完成后，完整项目 `client/` 的阶段路线、任务顺序、交付物与验证门槛。
> 常见联动：`docs/TODO.md`、`docs/AI导航.md`、`docs/AI知识库索引.md`、`docs/_kb_index.json`、`docs/AI记忆/项目记忆.md`、`docs/AI记忆/current_state.json`。

## 1. 定位

本规划是正式项目从“文档 / MVP 验证”转入“完整项目实现”的工作路线图。后续 AI agent 接到任务时，先完成 `AGENTS.md` 开工 5 步，再读本文判断当前应该推进哪个阶段、需要读哪些权威文档、交付哪些文件、跑哪些验证。

本文不替代 GDD、ADR、词表或测试策略。发生冲突时按 `docs/AI知识库索引.md` 的权威层级处理：先看 `AGENTS.md` / 规则 / ADR / GDD / 词表 / 测试策略，再更新本文。

## 2. 工作原则

- **先骨架，后内容**：完整项目先立 `client/` Godot 工程、autoload、数据加载、存档、UI 栈和验证闭环，再扩展大量内容。
- **先可验证，后做大**：每个阶段都必须有可运行 / 可校验的门槛；不要把多个阶段的风险混在一次大改里。
- **历史 MVP 只迁移经验，不搬临时代码**：MVP 结论可进入设计、数值和工具流程；隔离实验目录已移除，临时代码不得复活到正式 `client/`。
- **数据驱动优先**：角色、武器、敌人、遗物、升级、装备 Mod / 局外成长、文案和设置尽量通过 `client/data/`、`client/locale/`、词表与生成常量表达。
- **文档与测试同阶段落地**：长期模块必须有 `docs/代码/<module_id>.md`；行为变化按 `docs/测试策略.md` §7 补测试或说明验证缺口。

## 3. 后续 AI 使用方式

1. 读 `docs/AI记忆/current_state.json` 确认当前阶段和候选下一步。
2. 读本文的阶段表，选择最靠前且未完成的阶段任务。
3. 按该阶段的“必读文档”和 `docs/AI协作/上下文预算.md` 控制读取范围。
4. 完成阶段任务后，更新 `docs/TODO.md`、`current_state.json` 和当日会话日志；若产生新决策，追加 ADR。
5. 大更改按 AI Git 提交策略自动 commit；只 stage 本次明确修改的文件。

## 5. 分阶段执行细则

### F1：正式工程骨架

必读：`docs/AI导航.md`、`docs/代码文档规范.md`、`docs/测试策略.md`、`docs/游戏设计文档.md` 中项目结构 / autoload / 输入相关段落。

建议拆分：
- 创建 `client/project.godot`、`client/scenes/`、`client/scripts/`、`client/data/`、`client/locale/`、`client/assets/`、`client/templates/`。
- 添加最小启动场景和 README / 运行说明。
- 确认 `tools/godot_bridge.py` 能指向正式 `client/`，或记录暂用命令。

### F2：横向 autoload 骨架

必读：GDD §9.3~§9.19、`docs/AI导航.md` 5.2、`docs/代码文档规范.md`、`docs/测试策略.md` §7。

建议拆分：
- 先做 `DataLoader`、`RNG`、`GameClock`、`GameState`，再做 `Settings`、`Localization`、`UIManager`。
- `SaveManager`、`PoolManager`、`Combat` / `StatusEffect`、`AudioManager` 可独立提交，避免一次改太大。
- 每个长期模块同步 `docs/代码/<module_id>.md`，脚本头加 `# Doc:`。

### F3：数据 / 契约闭环

必读：`client/data/README.md`、`client/locale/README.md`、`docs/词表与契约.md`、`tools/sync_contracts.py`、`tools/validate_data.py`。

建议拆分：
- 先保证已存在的 `player.json`、`strings.csv` 可被正式项目加载。
- 新增 `growth.csv` / `growth_pools.json` 或给某个模式挂接 `growth_pools` 时同步数值手册、GDD 经验升级章节和词表中必要 id；默认标准模式按 ADR #120 保持不挂成长池。
- F3 阶段新增 `characters.json` / `weapons.json` / `enemies.csv` / `hazards.csv` / `relics.json` / `game_modes.json` 时只落数据边界、跨文件引用和 schema 诊断；选择 UI、武器运行时、敌人 / 刷怪运行时、机关运行时、遗物运行时或模式运行时留到后续阶段按用户点名推进。
- 所有裸 id 都通过生成常量引用。

### F4：最小可玩闭环

必读：GDD 玩家 / 输入 / 武器 / 敌人 / 刷怪 / 对象池 / 伤害章节，`docs/测试策略.md`。

建议拆分：
- 先玩家移动和相机居中，再武器 / 子弹，再敌人 / 刷怪。
- 高频实体第一版就接 `PoolManager`，不要临时 `instantiate` / `queue_free`。
- 伤害第一版就走 `Combat.apply_damage`。

### F5：暂停 / 存档 / 续局

必读：GDD §9.12、§9.16、`docs/词表与契约.md` §14、`docs/测试策略.md` 存档义务。

建议拆分：
- `meta` 与 `run` kind 分开实现，但共享标准头字段、迁移、原子写入和损坏隔离。
- 暂停菜单“保存并退出”和主菜单“继续游戏”要通过 `GameState` / `UIManager` 协调。
- 恢复点至少覆盖玩家、敌人、子弹、掉落、RNG、GameClock 和关键 UI 状态；非默认成长模式还需覆盖经验与升级 UI 状态。当前 F5 已覆盖普通游玩、暂停菜单、升级选择面板和升级面板上方暂停菜单叠层，并已通过用户最终手动存档 checklist；ADR #120 后默认标准模式不触发升级 UI。

### F6：成长与旧局外循环

必读：GDD §7.1 / §7.2、`client/data/README.md`、`client/locale/strings.csv`。旧 `MetaProgressionSystem` 只作历史记录，不作为当前实现入口。

建议拆分：
- `GrowthSystem` 的经验阈值、升级暂停和候选抽取已在 F4/F5 阶段形成最小闭环；ADR #120 后默认标准模式不启用它，后续只在明确的非默认模式或实验模式里重新挂接。
- 历史 `MetaProgressionSystem` 的结算奖励、账号等级、升级购买和解锁已按 ADR #117 退役，不再作为当前实现入口。
- 待决策 E 未完成前，升级选项池只实现保守数据结构，不写死内容边界。
- ADR #117 后，`MetaProgressionSystem` 运行时 / UI / smoke 已删除；ADR #118 后旧 `meta_progression.json` 与测试档迁移 / 补偿路径也已删除。

### F7：设置 / 本地化 / UI 栈

必读：GDD §9.4 / §9.5 / §9.14、`client/locale/README.md`、`docs/词表与契约.md` 第 5~7 节。

建议拆分：
- 主菜单、HUD、暂停菜单、设置菜单优先。
- 升级选择界面跟随未来非默认成长模式；结算、装备 Mod 和设置相关 UI 跟随当前默认短刷图循环。
- 所有玩家文本走 locale key；新增文案覆盖 `zh_CN` / `en`，AI 自动补齐另一语言首版译文后交人工复核。
- 输入全部走 InputMap action 和 Settings 重绑定。

### F8：回放 / 测试 / 平衡基线

必读：`docs/测试策略.md`、GDD §9.9 / §9.10 / §9.18、`docs/CICD规划.md`。

建议拆分：
- 先 L1 单测覆盖 `RNG`、`GameClock`、`GameState`、`SaveManager`、`ModifierEngine`、`Combat`；当前已用临时 `l1-smoke` 覆盖基础设施首片。
- 再推进 Replay runner 和黄金回放；当前已入库 `client/tests/replays/golden_basic_run.replay`、`golden_pause_resume.replay`、`golden_full_death.replay` 与 `golden_level_up_choice.replay` 的运行时摘要 + 扩展稳定帧样本 / 场景语义字段版，并已接入 gameplay 输入录制、runner 输入播放、full-death runtime event 播放和 level-up choice decision 覆盖；其中 `golden_level_up_choice` 由 harness 显式启用成长池，默认标准模式仍不启用升级。后续补更多 replay 场景，`golden_relic_synergy` 等到遗物运行时存在后再补。
- 平衡 sim 先预留 `AIPlayer` 接口，避免过早做完整自动玩家；当前 `perf-probe` 保留为用户明确要求时才运行的轻量 schema v2 可比较指标，覆盖 warmup 后帧时间分布、实体峰值、池峰值和预算状态。

### F9：内容扩展与 Demo 打磨

必读：内容对应任务模板、`docs/词表与契约.md`、`client/data/README.md`、`client/locale/README.md`。

建议拆分：
- 先建立内容黄金样例，再批量增加角色 / 武器 / 敌人 / 遗物。
- 每次新增内容优先走数据和 locale；只有缺 primitive 时才改逻辑层。
- Demo 前跑完整 L5 手动回归：输入设备、语言、存档迁移、回放重现、性能预算。

### F10：战区导演

必读：`docs/AI协作/工作包/F10-WarzoneDirector.md`、`docs/代码/warzone_director.md`、`docs/代码/gameplay_runtime.md`、`docs/代码/data_loader.md`、`client/data/README.md`。

建议拆分：
- 先建立 `warzone_directors.json` schema、DataLoader / `validate_data.py` 跨文件校验和坏样例回归。
- 再接入 `WarzoneDirector` 到 `GameplayRunLoop` 的刷怪门控，只按固定阶段和 `GameClock` 时间启用 wave。
- 巢变异、遭遇组合和兴趣点先作为数据声明与 debug summary 输出，后续真实效果按用户点名逐片接入。
- 设计红线：不读玩家血量、DPS、位置、失误、输入等玩家状态；不做隐藏 DDA；不在运行时接 LLM。

### F11：装备 Mod / 局外装配

- 掉落 HUD 暂存提示和标题菜单 Gear Mod 最小 UI 已完成；后续补更多 Mod 内容。

### F12：短刷图开放战区首片

必读：`docs/AI协作/工作包/F12-ShortLootRuns.md`、GDD §2 / §5 / §7、`docs/代码/gameplay_runtime.md`、`docs/代码/warzone_director.md`、`docs/代码/map_manager.md`、`docs/代码/gear_mod_system.md`。

建议拆分：

### F13：九宫模块大地图

必读：`docs/AI协作/工作包/F13-ModularGridWorld.md`、GDD §5.1、`docs/测试策略.md`、`docs/代码/module_world_manager.md`、`client/data/README.md`。旧 `F13-HandcraftedRooms.md` 只作 superseded 历史记录。

已锁定分层：
- 世界固定 9×9 模块，每模块 11×11 格，中心全局格 `(49,49)` 是世界原点；同一世界格尺寸统一。
- 起点 / 目标 / 撤离为固定键槽，普通槽位按 run seed 从 `approved` 池组合，失败用已校验安全布局。
- AI 只在编辑期产出 JSON candidate；不接运行时 LLM，不建首版可视化编辑器。
- 玩家周围最多 3×3 模块活跃，模块内容按世界槽位保存 / 恢复，不按模板 id 共享。
- run v4 保存 81 槽 assignment / 旋转、map hash、迷雾、目标 / 撤离和槽位状态；旧 v3 run 明确重置，`meta` 不受影响。

必读：`docs/AI协作/工作包/F14-EnemyNavigationAndPerception.md`、GDD §5.3、`docs/代码/enemy_ai.md`、`docs/代码/module_world_manager.md`、`client/data/README.md` 与 `docs/测试策略.md`。

- 玩家跨全局格时只在按最大视觉范围推导的局部窗口内重算一次确定性八方向共享流场；当前半径 8 格、最多 289 格，只清理上次触达索引。守家和记忆目标仍在完整 99×99 mask 上按决策 tick 查询 AStar waypoint。
- 感知按地形视线、路径距离和最后已知位置记忆排序；普通敌人记忆结束停止，守家敌人绕路回出生点。
- 冲锋与远程攻击受走廊 / 地形视线约束；玩家仍是唯一战斗目标，敌方友伤拒绝与中心分离不变。
- 不引入 `NavigationAgent2D`、运行时 navmesh、逐格 Node、视野锥、声音事件、怀疑度或巡逻。

## 6. 阶段门禁

每个阶段结束至少满足：
- 代码或文档对应的权威来源已同步。
- `docs/AI记忆/current_state.json` 的 `next_actions` 指向下一阶段。
- 新增长期文档登记进 `docs/_kb_index.json`。
- 修改 ADR 后，`docs/AI知识库索引.md` 的 ADR 追踪矩阵覆盖最新编号。
- 验证命令写入当日会话日志；失败项要说明原因和后续处理。

## 7. 任务选择建议

后续 AI 如果没有用户指定任务，按下列优先级选择：

1. 当前阶段未完成且阻塞后续阶段的骨架任务。
2. 能形成独立 commit、独立验证的最小纵切。
3. 修复当前验证失败或文档漂移。
4. 扩展内容、调数值和打磨体验。

不要在 F1 / F2 期间提前批量加内容；不要在 SaveManager 和 GameState 未稳定前做复杂局外成长；不要在词表 / 数据校验未闭环前引入大量裸 id。
