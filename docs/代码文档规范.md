# 代码文档规范

> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md`。
> 本文档是“代码变更需要对应哪些文档”的权威规范；改代码目录、模块边界、公共 API、数据 schema、扩展点或文档门禁时，必须同步当前平台编码规则入口、`docs/AI导航.md`、`docs/AI协作/文档维护指南.md`、`docs/决策记录.md`、`docs/AI记忆/项目记忆.md` 与当日会话日志。

---

## 1. 核心原则

- **不是每个文件一篇文档**：按“长期维护模块 / 公共契约 / 数据 schema”建立文档，避免文档爆炸。
- **完整项目必须是详细模块文档**：`client/` 长期模块文档不能只是几行自动摘要，必须能让人类维护者理解场景/节点结构、运行流程、API、数据、依赖、扩展点、常见改动、故障排查与测试义务。
- **代码与文档同 PR / 同任务完成**：新增或修改代码时，必须同步更新对应文档；若判断无需更新，需要在最终回复或 PR 备注写明原因。
- **文档写契约，不复述实现**：说明职责边界、公共 API、依赖、数据、扩展点、测试义务和已知限制，不逐行翻译代码。
- **自动化只做脚手架和检查**：AI 可以按模板自动创建/维护模块文档，未来工具可以生成目录、路径表或覆盖率报告；但不得用“从代码抽取的简陋摘要”替代人工可读的模块说明。
- **权威来源单一**：设计原因进 GDD / ADR，字符串 id 进词表，测试要求进测试策略，模块使用方式进 `docs/代码/`。
- **示例代码也守风格**：模块文档中的 GDScript 片段必须符合 [Godot 4.7 官方 GDScript style guide](https://docs.godotengine.org/en/4.7/tutorials/scripting/gdscript/gdscript_styleguide.html) 与项目更严格的类型化 / 数据驱动规则。
- **AI 可检索**：路径真实、标题稳定、表格表达映射关系；不要用“见上文”“同前”这类弱引用。

## 1.1 详细模块文档质量标准

完整项目 `docs/代码/<module_id>.md` 必须满足以下最低质量线：

| 类别 | 必须写清 |
|------|----------|
| 模块边界 | 负责什么、不负责什么、哪些行为属于正式契约 |
| 代码地图 | 涉及的脚本、场景、资源、数据文件和生成文件 |
| 场景 / 节点结构 | 有 `.tscn` 或节点路径依赖时，写场景树和关键节点路径 |
| 运行流程 | 初始化、每帧 / 物理帧、事件流、失败 / 清理流程 |
| 公共契约 | API、signal、输入输出、生命周期、错误处理 |
| 数据与词表 | JSON schema、字段单位 / 范围、词表 id、生成常量、校验规则 |
| 依赖关系 | 上游依赖、下游调用方、禁止依赖、与 autoload 的关系 |
| 扩展点 | 常见新增内容该改哪里，不允许的扩展方式 |
| 常见改动入口 | 调数值、改 UI、加行为、换资源、改输入等任务的入口文件 |
| 故障排查 | 常见现象与优先检查位置 |
| 测试义务 | 必跑命令、手动 checklist、何时补单测 / 回放 / 性能采样 |
| 迁移 / 兼容 | 需要迁移存档、数据、回放或旧行为时的约束 |

短小模块可以合并小节，但不能省略上述信息对应的内容；确实不适用时写“无”。

## 2. 代码变更到文档的映射

| 代码变更类型 | 必须同步的文档 | 说明 |
|--------------|----------------|------|
| 新增长期维护模块 / autoload / 核心系统 | `docs/代码/<module_id>.md`、`docs/AI导航.md` 第 5 节、必要时 GDD / ADR | 例如 `RNG`、`GameState`、`WeaponSystem`、`DataLoader` |
| 修改模块公共 API / signal / 生命周期 | 对应 `docs/代码/<module_id>.md`，必要时 `docs/AI导航.md` 依赖图 | 影响调用方或扩展点时必须记录 |
| 新增或修改数据 schema / 字段 | `client/data/README.md`、对应模块文档、必要时 `docs/词表与契约.md` | 字段含义、单位、默认值、范围、校验规则都要写清 |
| 新增或修改玩家可见文案 / 语言列 / 占位符规则 | `client/locale/README.md`、`client/locale/strings.csv`、必要时 `docs/词表与契约.md` | key 命名、必填语言、占位符集合和人工校对流程都要写清 |
| 新增 stat / effect / event / setting / action / pool / damage / status / audio / rng / character / capability / tag id | `docs/词表与契约.md` + 生成常量流程 | 不在模块文档里单独维护 id 白名单，避免双写漂移 |
| 修改玩法行为 / 系统职责 / 架构边界 | GDD、ADR、`docs/AI导航.md`、对应模块文档 | 行为变化不只写代码注释 |
| 修改测试义务 / 验证方式 | `docs/测试策略.md`、对应模块文档的测试段落 | 新增 primitive、回放、性能敏感逻辑时尤其重要 |
| 内部重构且不改行为 / API / 依赖 | 通常无需改文档 | 但如果模块边界、依赖方向或文件归属变化，仍需更新模块文档 |
| 修 bug | 若修复后改变对外行为，更新对应模块文档或 GDD；否则可只在最终回复 / PR 备注说明无文档变更 | 不要求每个小 bug 都追加长期文档 |

## 3. 文档位置约定

| 文档类型 | 路径 | 何时创建 / 更新 |
|----------|------|------------------|
| 代码文档索引 | `docs/代码/README.md` | 新增 / 删除模块文档时更新 |
| 详细模块文档 | `docs/代码/<module_id>.md` | 新增长期维护模块，或模块公共契约变化 |
| 数据字段说明 | `client/data/README.md` 或 `client/data/<name>.schema.md` | 数据文件落地后创建，字段变化时更新 |
| 多语言文案说明 | `client/locale/README.md` | 文案 key、语言列、占位符或翻译流程变化时更新 |
| 设计说明 | `docs/游戏设计文档.md` | 玩家可感知规则、系统设计、长期玩法约束变化 |
| 决策记录 | `docs/决策记录.md` | 新规则、新架构、新不可逆决策 |
| 约定字符串 | `docs/词表与契约.md` | 新增 / 改名 / 删除 id |
| 测试要求 | `docs/测试策略.md` | 测试层级、覆盖率、回放、性能预算变化 |

`module_id` 用 lower_snake_case，尽量与 `class_name` / autoload 名对应：`rng.md`、`game_state.md`、`weapon_system.md`、`data_loader.md`。

## 4. 代码文件头引用

长期维护脚本应在文件头放短引用，指向模块文档和上游权威：

```gdscript
# Doc: docs/代码/game_state.md
# Authority: docs/游戏设计文档.md §9.12, docs/决策记录.md ADR #21
```

例外：自动生成文件、测试文件、一次性调试脚本、极小私有 helper 可以不写文件头引用，但必须被上级模块文档覆盖。

## 5. 详细模块文档模板

新增 `docs/代码/<module_id>.md` 时使用这个结构。短模块可以合并不适用的小节，但不能降级成只有路径和 API 的简短摘要。

```markdown
# <ModuleName> 模块文档

> **AI 修改说明**：修改本文档前先读 `docs/AI协作/文档维护指南.md` 与 `docs/代码文档规范.md`。
> 本文档是 `<ModuleName>` 的代码契约权威；改公共 API、signal、数据 schema、依赖或测试义务时必须同步本文档。

## 职责
- 本模块负责什么
- 本模块不负责什么

## 阅读方式
| 你想做什么 | 先看哪里 |
|------------|----------|

## 代码位置
| 路径 | 作用 |
|------|------|
| `client/scripts/...` | ... |

## 场景 / 节点结构
- 有 `.tscn` 或节点路径依赖时写场景树。
- 没有场景结构时写“无”。

## 运行流程
| 阶段 | 发生什么 | 关键 API / signal |
|------|----------|-------------------|

## 公共 API
| 名称 | 输入 | 输出 | 约束 |
|------|------|------|------|

## Signal / Event
| 名称 | 参数 | 触发时机 |
|------|------|----------|

## 数据与契约
- 读取哪些 `client/data/` CSV / JSON
- 使用哪些词表 id / 生成常量
- fail-fast 校验规则

## 依赖
- 上游依赖
- 下游调用方
- 禁止依赖

## 扩展点
- 新内容应在哪里接入
- 不允许的扩展方式

## 常见改动入口
| 你想改什么 | 主要文件 | 同步文档 | 验证方式 |
|------------|----------|----------|----------|

## 故障排查
| 现象 | 优先检查 |
|------|----------|

## 测试义务
- 必跑测试
- 需要黄金回放 / 性能采样的条件

## 迁移 / 兼容
- 是否影响存档、数据 schema、回放、旧配置或外部工具。
- 不涉及时写“无”。

## 相关文档
- GDD / ADR / 词表 / 测试策略链接
```

## 6. 提交前自检

- [ ] 新代码是否属于某个已有模块文档？如果不是，是否创建了 `docs/代码/<module_id>.md`？
- [ ] 新增 / 修改完整项目长期模块时，模块文档是否达到“详细模块文档质量标准”，而不是简短自动摘要？
- [ ] 公共 API、signal、数据字段、依赖方向、扩展点或测试义务有变化时，模块文档是否同步？
- [ ] 设计 / 决策 / 词表 / 测试要求是否写回各自权威文档，而不是只写在代码注释里？
- [ ] 长期维护脚本是否有 `# Doc:` 文件头引用，或被上级模块文档覆盖？
- [ ] 文档中的 GDScript 示例是否符合 Godot 4.7 官方 GDScript style guide 和项目类型化规则？
- [ ] 若本次代码无需文档更新，最终回复或 PR 备注是否说明了原因？
