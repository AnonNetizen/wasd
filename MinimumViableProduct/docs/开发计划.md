# MVP 开发计划

> **AI 修改说明**：修改本文档前先读 `MinimumViableProduct/README.md` 与 `MinimumViableProduct/docs/MVP设计说明.md`。
> 本文档是 MVP 阶段计划；改阶段目标、验收标准或工作流时，必须同步 `MVP设计说明.md`、`经验记录.md` 与必要的 MVP 决策记录。

---

## 阶段划分

| 阶段 | 目标 | 产出 | 完成标准 |
|------|------|------|----------|
| M0 文档准备 | 明确 MVP 范围和边界 | README、设计说明、计划、经验记录模板 | 后续实现者不用读完整 GDD 也能开工 |
| M1 客户端骨架 | 建立独立 Godot 项目 | `MinimumViableProduct/client/project.godot` 与主场景 | 能打开并运行空场景 |
| M2 玩家与射击 | 固定玩家 + 四方向瞄准 + 自动射击 | Player、AimInput、Weapon、Bullet | 方向键可切换射击方向 |
| M3 敌人与刷怪 | 四方向刷怪并逼近玩家 | Enemy、Spawner | 敌人能从四方向出现并向中心移动 |
| M4 失败与 HUD | 加最小反馈闭环 | GameSession、HUD、重开 | 显示时间 / 击杀 / 失败状态 |
| M4.5 手柄输入试验 | 验证 MVP 能用手柄完成瞄准和重开 | InputMap runtime bindings、HUD 提示 | D-pad / 摇杆可四方向瞄准，手柄 A 可重开 |
| M4.6 数值配置化 | 把 MVP 核心调参值移出脚本 | `data/mvp_config.json`、配置接入、字段说明 | 调玩家 / 武器 / 敌人 / 刷怪 / 输入 / 背景数值不改脚本 |
| M5 复盘 | 总结工作流与玩法判断 | 经验记录、局部决策 | 至少 3 条可行动经验 |

## 阶段状态

| 阶段 | 状态 | 备注 |
|------|------|------|
| M0 文档准备 | 完成 | MVP 范围、计划、决策与经验模板已建立 |
| M1 客户端骨架 | 完成 | 已创建独立 `project.godot`、主场景与最小启动脚本 |
| M2 玩家与射击 | 完成 | 固定玩家、方向键四方向瞄准、自动射击已实现 |
| M3 敌人与刷怪 | 完成 | 四方向循环刷怪、敌人向中心推进、子弹可击杀敌人 |
| M4 失败与 HUD | 完成 | 已接入 HP、时间、击杀数、失败面板和 Enter / Space 重开 |
| M4.5 手柄输入试验 | 完成 | 方向键、D-pad、左右摇杆共用四方向 action；手柄 A 可重开 |
| M4.6 数值配置化 | 完成 | 核心调参值已集中到 `MinimumViableProduct/client/data/mvp_config.json` |
| M5 复盘 | 下一步 | 总结工作流、玩法判断和是否迁移到完整项目 |

## 当前优先级

1. M5 试玩并复盘哪些经验值得迁移到完整项目，重点检查键盘 / 手柄手感与 `mvp_config.json` 调参效率。
2. 若继续迭代 MVP，先基于复盘结果决定是否新增 M6。

## 工作流约定

- 每个阶段结束后，在 `经验记录.md` 记录“做了什么 / 遇到什么 / 下次怎么改”。
- MVP 的临时折中可以记录在 `MVP决策记录.md`，但不要自动升级为完整项目 ADR。
- 若某条 MVP 经验要影响完整项目，先在根目录 `docs/决策记录.md` 追加 ADR，再同步 GDD / AI导航 / 项目记忆。
- 默认不碰根目录 `client/`，也不读取或引用 `draft/`。
