## 记录原则

- 记录“可行动结论”，不要写流水账。
- 区分 MVP 临时折中和可以迁移到完整项目的经验。
- 每条经验尽量包含触发场景、观察、结论、下一步。

## 经验表

| 日期 | 阶段 | 观察 / 问题 | 结论 | 下一步 | 是否迁移到完整项目 |
|------|------|-------------|------|--------|----------------------|
| 2026-06-13 | M0 文档准备 | 先把 MVP 与完整项目隔离 | MVP 可以更激进地试错，但经验需要留痕后再迁移 | 搭建独立客户端骨架 | 待定 |
| 2026-06-13 | M1 客户端骨架 | 独立 Godot 项目只需要 `project.godot`、主场景和最小脚本即可形成可启动边界 | 先建立可运行空场景能降低后续功能堆叠风险 | M2 实现固定玩家与四方向自动射击 | 可迁移为“先立启动骨架再加玩法”的工作流经验 |
| 2026-06-13 | M2 玩家与射击 | 新脚本首次 headless 编译时跨脚本 `class_name` 强类型引用不稳定 | MVP 中跨节点引用先用基础类型 + `call/connect`，降低首次导入阻塞；等结构稳定再收紧类型 | M3 增加敌人与刷怪时继续先保启动稳定 | 可迁移为“原型阶段优先保持项目可启动”的经验 |
| 2026-06-13 | M3 敌人与刷怪 | 固定顺序四方向刷怪让敌人压力可复现，子弹命中反馈能在没有 HUD 时验证闭环 | 原型先用确定性刷怪和最小碰撞反馈，等 M4 再加统计与失败状态 | M4 接入玩家受伤、计时、击杀数和重开 | 可迁移为“先复现再随机”的原型经验 |
| 2026-06-13 | M3 视觉打磨 | 几何占位如果没有背景层级和 UI 质感，会显得像调试 demo | 先用 `_draw()` 做深色背景、网格、准星、描边和拖尾，可以显著提升原型可读性且不引入资源成本 | M4 HUD 继续沿用同一视觉语言，补计时 / 击杀 / 失败面板 | 可迁移为“无资源阶段也要定义临时视觉语言”的经验 |
| 2026-06-13 | M4 闭环 | HP、计时、击杀和失败重开能让 MVP 从“能动”变成“可试玩” | 原型做到首个闭环后应先复盘，不急着继续加系统 | M5 复盘玩法、工作流和可迁移经验 | 可迁移为“先闭环再扩展”的工作流经验 |
| 2026-06-13 | M4.5 手柄输入 | 手柄支持不需要改射击逻辑，只要把 D-pad / 摇杆绑定到同一组 InputMap action | 输入设备差异应收在绑定层，玩法代码只消费 action 和方向向量 | M5 人工试玩确认 D-pad、左摇杆、右摇杆、A 键重开手感 | 可迁移为“手柄优先走 InputMap，不在玩法层分设备”的经验 |
| 2026-06-13 | M4.6 数值配置化 | MVP 进入可试玩后，脚本里分散的 HP / 射速 / 敌人 / 刷怪数值会阻碍人工调参 | 即使是隔离原型，也应尽早把核心调参入口集中到一个轻量配置文件 | M5 试玩时直接改 `mvp_config.json` 验证节奏与手感 | 可迁移为“原型调参也要数据集中”的工作流经验 |
