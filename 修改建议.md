## A. 关于「射击不可控制是否开火」（原则 #2）
- **潜在问题**：全自动开火在某些场景（如想躲弹幕不想暴露位置、或想省充能）会让玩家失去策略空间。
- **建议**：保留自动射击作为基础设定，但**允许「松开方向键即停火」**作为可选项，给玩家「不开火走位」的微操空间。是否停火可作为一个设置或天生差异。

## B. 关于瞄准方式（原则 #2）
- 键盘 4 方向（上下左右）瞄准在开放大地图上可能略显粗糙（敌人来自四面八方，斜向敌人难以命中）。
- **建议（可选）**：提供「自动瞄准最近敌人」模式，或支持鼠标瞄准作为替代方案，照顾不同玩家。4 方向瞄准要做手感优化（朝向插值/记忆上次方向）。

## C. 大地图带来的节奏问题
- 《吸血鬼幸存者》靠「敌群包围」制造压力；纯开放地图若无引导易迷失。
- **建议**：加入**软边界或兴趣点**（精英刷新点、宝箱、商店、机关密集区），给玩家移动目标，避免无目的乱逛。

## D. 子弹有飞行速度 + 自动射击的配合（原则 #4）
- 移动 + 自动开火 + 4 方向瞄准时，子弹易打空（尤其斜向的敌人）。
- **建议**：初期把 `bulletSpeed` 调高、可加轻微「弹道宽容/小范围散射」，命中体验更好；后续用遗物（追踪弹）补足。

# 框架结构性增强建议（J~R，2026-06 评审提出）

> 以下为对现有框架的结构性补强建议，与「本地化/设置/埋点」三条横向基础设施同级别。
> 标注优先级：⭐⭐⭐=建议现在就写进框架（后期补成本最高）；⭐⭐=值得定但可稍后；⭐=工程隐患需先定规则。
> 采纳后再按规则 19/20 同步进 `游戏设计文档.md` / `词表与契约.md` / `决策记录.md` / `AI导航.md`。

## J. RNG / 随机种子管理 ⭐⭐⭐
- **缺口**：掉落、升级三选一、刷怪、机关撒布全依赖随机，但目前无统一随机源，也无种子化。
- **建议**：新增 `RNG`（autoload）。
  - 全局**种子化**：一局一个 seed，支持「输入种子复现同一局」——调试 bug、平衡测试、未来「每日挑战」的刚需。
  - **按用途分流**（`RNG.drop` / `RNG.spawn` / `RNG.ui_choice`），避免一处调用偏移影响全局随机序列。
  - 配合 `luck` 属性提供统一的加权抽样接口。
- **理由**：后期再塞种子化几乎要改动所有随机调用，必须框架阶段定。

## K. 游戏状态机 / 流程管理（GameState）⭐⭐⭐
- **缺口**：已有 `PauseMenu`，但「主菜单 → 开局 → 进行中 → 暂停 → 结算 → 死亡 → 回主菜单」整条流程无统一管理，散落各处易乱。
- **建议**：新增 `GameState`（autoload）：枚举状态 + `state_changed` 信号；暂停/结算/升级三选一都挂其下；`run_start` / `run_end` 埋点由它统一触发。

## L. 通用对象池（PoolManager）⭐⭐⭐
- **缺口**：规则只钉了 `BulletPool`，但敌人、掉落物、**伤害飘字、命中特效**同样高频。
- **建议**：抽象通用 `PoolManager`（按场景类型分池）；池类型在 `词表与契约.md` 登记，避免每类实体各写一套池逻辑。

## M. UI 界面栈管理器（UIManager）⭐⭐⭐
- **缺口**：暂停、设置、升级三选一、结算都是叠加弹窗，且都要联动 `get_tree().paused`，无界面栈则层级/开关/暂停联动散落。
- **建议**：新增 `UIManager` 统一 `push/pop` 界面，自动处理暂停联动与输入焦点；9.8 暂停逻辑归入其下。

## N. 伤害结算管线 + 状态效果系统 ⭐⭐
- **缺口**：`modifiers` 只解决「玩家属性聚合」，但「一次伤害怎么算」（暴击判定、点燃/中毒/冰冻等 DoT/debuff 的持续与叠加规则）目前糊在 `effect` 里。
- **建议**：
  - 统一 `DamageInfo` 结构 + 单一伤害结算入口；
  - 把 `ignite` / `chain` 等持续效果抽成 `StatusEffect`（带 duration / stack 规则），敌人与玩家共用。

## O. SaveManager 抽象（对齐 Settings 规格）⭐⭐
- **缺口**：元进度存档目前仅一句「写 `user://`」，未抽象成统一管理器，且无版本概念。
- **建议**：像 `Settings` 一样立 `SaveManager`（autoload）；**强制加 schema `version` 字段 + 迁移逻辑**——存档格式变更无版本号即灾难。

## P. 统一时间源（GameClock）⭐⭐
- **缺口**：刷怪曲线、充能、buff 持续都依赖「游戏时间」，且暂停时必须停。
- **建议**：提供随 `paused` 冻结的统一时间源；**禁止业务直接读 `Time.get_ticks_msec()`**，否则暂停/时间缩放会出 bug。

## Q. AudioManager ⭐⭐
- **缺口**：设置里有音量总线，但无统一播放接口。
- **建议**：统一 `AudioManager.play_sfx(id)`；音效 id 进词表，避免散落 `play()`。

## R. 词表（md）→ 代码常量的单一来源 ⭐
- **缺口**：`词表与契约.md` 是白名单，但落地后代码端必有 `Stats.gd` / `Effects.gd` / `Events.gd` 等常量枚举，**两边手动同步迟早漂移**。
- **建议**：明确约定二选一——
  - (a) md 为唯一来源，代码常量按它手写，并在 `DataLoader` 校验时反向核对；或
  - (b) 由脚本从一份 JSON **生成**常量。
- **理由**：现在定好规则，比以后对账成本低得多。

> 取舍参考：**J / L / M（及 K）建议现在写进框架**，后期补成本最高；其余作为 M1/M6 的明确扩展点登记即可。
